# AI.md

このファイルはすべてのAIエージェントが参照する**唯一のルールファイル**です。
Claude Code・Copilot・Codex・その他すべてのAIはセッション開始時にこのファイルを読んでください。

> **ルールはここにだけ書く。`CLAUDE.md` や `AGENTS.md` には書かない。**

---

## セッション開始時の手順

AIはセッション開始時に必ず以下の順序で確認してください：

1. この `AI.md` で、自分に課せられた行動ルールと禁止事項を確認する
2. `docs/project.md` を読み、Relicの内部向けプロジェクト概要を把握する
3. `docs/dev/phases.md` で、現在フェーズと読むべき文書を確認する
4. `phases.md` に書かれた現在フェーズの「最初に読む」文書だけ読む
5. 正本を読んでも判断できないことだけ、ユーザーに確認する

> 各段階では、その段階の文書が扱う範囲だけを認識すること。下位文書で扱うべきプロジェクト固有の内容を、上位文書から推測しないこと。
> `docs/journal/` は、経緯確認が必要な場合だけ読むこと。

### 最初の応答

セッション開始直後の応答では、現在フェーズを理解した状態で話すこと。

- 現在フェーズ名を明示する
- 現在フェーズの正本文書を読んだ前提で、次に進める具体的な作業範囲を提示する
- 汎用的な「今日は何を進めましょう？」だけで終わらない
- ユーザーに確認する場合も、現在フェーズを前提にした具体的な確認にする

例:

> 現在は `P9: 機能確認・不具合修正` です。正本は `docs/dev/phases/P9-feature-checklist.md` なので、まず「1. ワークスペース管理」からコード確認・自動テスト・ビルドで判断できる範囲を確認し、動かない項目は修正していきます。実アプリ操作が必要な項目は、必要になった時点で確認します。

## 作業区切り時の文書更新確認

AIは実装・検証・修正・調査の区切りごとに、完了報告の前に必ず以下を確認してください。

1. `docs/dev/phases.md` で現在フェーズを確認する
2. 現在フェーズの正本文書を確認する
3. 今回の作業内容に対して、更新すべき文書があるか判断する
4. 更新が必要な場合は、ユーザーが明示した編集範囲、または現在フェーズの正本文書に限って更新する
5. 関連ファイルの更新が必要だが、ユーザーの明示範囲・現在フェーズの正本文書を越える場合は、理由・対象ファイル・変更内容を説明してユーザーに確認する
6. `docs/dev/phases.md` と `docs/journal/YYYY-MM-DD.md` は、下記「セッション終了時の手順」と「編集範囲と許可のルール」に従い、勝手に更新しない
7. 完了報告には「更新した文書」または「更新不要と判断した文書」を含める

現在フェーズの正本文書へUI/UX判断や仕様判断を記録する場合は、方向性だけ・実施だけを分離して書いてはならない。
必ず、同じ項目の中に以下をセットで記録する。

- 方向性
- 実施
- 確認
- 残り

未実施・未確認の場合も省略せず、「未実施」「未確認」と明記する。
日誌に詳細を書く場合でも、現在フェーズの正本文書には次回作業時に判断できる粒度でこのセットを残す。

P9-runtime-verification では、コード確認・自動テスト・ビルド・実アプリ操作のいずれかで確認できた項目は `docs/dev/phases/P9-feature-checklist.md` を更新する。
確認方法は必要に応じて `docs/journal/YYYY-MM-DD.md` に記録する。

P9-runtime-verification では、GUI操作・実アプリ操作でしか分からない確認は、ユーザーが明示的に求めた場合を除き無理に実施しない。
トークン消費を抑えるため、コード確認・自動テスト・ビルドで判断できる範囲を優先し、実アプリ操作でしか判断できない項目だけ保留理由を完了報告に書く。

## セッション終了時の手順

AIはセッション終了時に必ず以下を行ってください：

1. ユーザーから開発の区切り・終了を明示された場合は、必要に応じて `docs/dev/phases.md` と `docs/journal/YYYY-MM-DD.md` を更新する
2. それ以外のタイミングでは、`docs/dev/phases.md` と `docs/journal/YYYY-MM-DD.md` はユーザーから明示的に指示された場合のみ更新する
3. 日誌を更新する場合は、今日やったこと・決定したこと・未決事項を記録する
4. 同じ日付の中で複数回更新する場合も、セッション別の見出しを作らず、やったことを時系列順に追記する

### フェーズ完了時の扱い

- あるフェーズが完了しても、次の開発フェーズへ自動的に進めてはならない
- ユーザーが次フェーズへの移行を明示した場合だけ、`docs/dev/phases.md` の `current` を次フェーズに変更する
- 次フェーズが未決定の場合は、完了内容・残り・次候補を整理し、ユーザーが次フェーズを明示するまで `current` を変更しない

## 仕様書の作成方針

仕様書（`docs/spec/`）を作成・更新する際は、以下の手順で進めてください：

1. AskUserQuestion ツールを使い、ユーザーに1問ずつヒアリングする
2. ユーザーの回答をもとに仕様を決定する
3. ユーザーの承認を得てからファイルに書き込む

ユーザーは非エンジニアのため、質問は具体的な例を交えてわかりやすく聞いてください。

---

## ユーザー要望の扱い

ユーザーが「〜したい」「〜をやりたい」「こういう方向にしたい」と言っただけで、AIは仕様書・設計書・フェーズ・チェックリスト・デザイン方針を勝手に作成してはならない。

この場合は、まずユーザーにヒアリングする。

- 何が嫌か、何を変えたいかを確認する
- 変更してよい範囲を確認する
- 文書化するか、実装するか、調査だけにするかを確認する
- AIが提案する場合も、提案として出し、ユーザーが採用したものだけを正本化・実装する

チェックリスト作成やフェーズ作成自体も作業であり、ユーザーの確認負担を増やす。
そのため、ユーザーが明示的に求めた場合、またはヒアリングの結果として合意した場合だけ作成する。

## 技術選定・意思決定のルール

- **決してユーザーへの説明なく独自判断によって選定しない**
- 技術的な選択肢が生じた場合は、各選択肢を非エンジニアにもわかる言葉で説明し、ユーザーに決定を委ねること
- AIが推薦を示すことはOK。ただし最終決定は必ずユーザーが行う
- 「技術的なことはわからない」と言われた場合も、わかりやすく言い換えて説明し直すこと。勝手に決めてはならない

## 未決定事項の管理ルール

- 仕様書・設計書・UI文書に「未決定事項」セクションを書いてはならない
- 未決定の問いはすべて `docs/dev/open-questions.md` に一元管理する
- 各問いには `関連:` フィールドで由来となる文書を明記する
- `open-questions.md` は未決定事項が発生したとき、または `phases.md` が読むよう指定したときだけ読む

---

## 編集範囲と許可のルール

- ユーザーが明示した対象ファイル・対象範囲以外を編集してはならない
- 関連ファイルの更新が必要だと判断した場合も、勝手に編集せず、理由・対象ファイル・変更内容を説明してからユーザーの許可を得る
- 変更の取り消し・巻き戻しも、ユーザーが明示的に指示するまで実行してはならない
- 「整合性のため」「慣例上必要」「セッション終了時の手順」などを理由に、ユーザーの直近指示を越えて作業してはならない
- UI変更では、ユーザーが指定した画面・場所・操作対象だけを変更する。別の場所に同じ機能の入口を追加する、表示要素・ボタン・説明・装飾を足す、操作方法を増やすなどの範囲拡張は、どれほど有用に見えても事前に確認する
- ユーザーが「Aの話をしている」と対象を訂正した場合、A以外へ加えた変更は撤回し、以後その会話ではA以外を触らない。提案がある場合も実装せず、提案として分けて伝える
- ただし、ユーザーが開発の区切り・終了を明示した場合は、必要に応じて `docs/dev/phases.md` と `docs/journal/YYYY-MM-DD.md` を更新する
- 上記以外では、`docs/dev/phases.md` と `docs/journal/YYYY-MM-DD.md` は、ユーザーが明示的に指示した場合のみ更新してよい
- 開発位置とフェーズ順は `docs/dev/phases.md` を正とする
- 実装時も、ユーザーが明示した対象・現在のフェーズ範囲を越える変更は行わない
- ユーザーから明示的に指示がない限り、ブランチは切らずにメインブランチで作業する

### Relic固有の検証禁止事項

- `app/out/` 配下のパッケージ版アプリは、ユーザーが明示的に配布ビルド確認を求めた場合以外、開発中のテスト・確認対象にしてはならない
- 実アプリ検証は、開発版であることを確認できる起動経路だけを対象にする
- テスト用ワークスペース・検証用リポジトリ・一時データをRelic開発リポジトリ内に作ってはならない
- 実アプリ検証でユーザーのワークスペースや外部サービスに触る必要がある場合は、事前に対象・操作内容・戻し方を説明し、ユーザーの明示許可を得る
- ユーザーが「触るな」「話しかけるな」と指示した場合は、必要な修正・検証が完了するまで途中報告を挟まない

### Git / GitHub UIの原則

- Gitの正式用語は必要な範囲でそのまま使い、幼稚な言い換えをしない
- コミットに「ローカル」などの冗長な開発者向け修飾を付けたUI文言を使わない
- GitHub接続済みなら、コミットや注釈タグに必要なユーザー情報はGitHub接続情報から内部で決定し、通常UIで作者名・メールを入力させない
- コミット作成UIは、原則としてコミットメッセージだけをユーザー入力にする

# docs/_rules.md

このファイルは `docs/` 以下のすべてのドキュメントに適用される作成・管理ルールです。
人間・AIを問わず、ドキュメントを作成・編集する際はこのルールに従ってください。

---

## 全般

- ドキュメントはすべて日本語で記述する
- 承認されていない内容は書かない
- 既存の内容は勝手に削除・変更しない
- リンクは標準Markdownの相対パスで記述する（例: `[仕様](spec/editor.md)`）
- 文書構成は、このプロジェクトだけでなく今後の開発でも雛形として再利用できるよう、役割・粒度・命名をできるだけ普遍的に保つ

---

## ファイル・フォルダ構成

- 新しいドキュメントを作る前に [dev/phases.md](dev/phases.md) と [INDEX.md](INDEX.md) を確認し、既存文書への追記で足りるか判断する
- 新規ファイルを作成したら [INDEX.md](INDEX.md) のドキュメントマップに追記する。ただし、既存文書と役割が重複する文書は作らない
- `_` で始まるファイル（`_rules.md`, `_template.md`）はメタドキュメント。内容ではなくルール・形式を定義する
- `AGENTS.md` と `CLAUDE.md` は薄い入口として扱い、独自ルールを書かない。AI向けルールは `AI.md` に一元管理する
- 現行仕様・設計・運用判断に使う正本と、履歴・アーカイブ・テンプレートを混同しない
- 現行実装と文書が食い違う場合は、現行正本を実装に合わせて更新する。履歴文書を現行仕様のように書き換えない

---

## 各フォルダの用途

| フォルダ | 書くもの |
|----------|----------|
| `product/` | プロジェクト固有の意図・計画・用語 |
| `spec/` | 機能の振る舞い・詳細仕様 |
| `architecture/` | 設計・データモデル・意思決定記録 |
| `tech/` | 技術調査・ライブラリ選定 |
| `ui/` | 画面構成・遷移フロー |
| `dev/` | フェーズ管理・開発規約・未決定事項・テスト方針 |
| `journal/` | 開発日誌 |
| `archive/` | 現行判断に使わない過去資料 |
| `assets/` | ロゴ案などの検討素材 |

アプリ本体の仕様だけでなく、AI運用、フェーズ管理、開発規約、検証方針、日誌、入口文書、補助ファイルの役割分担も文書整理の対象に含める。

---

## 未決定事項の管理

- **仕様書・設計書・UI文書に「未決定事項」セクションを書いてはならない**
- 未決定の問いは必ず [dev/open-questions.md](dev/open-questions.md) に一元管理する
- 各問いには `関連:` フィールドで由来となる文書を明記する（どのファイルを見ればその問いの背景がわかるかを示す）
- 決定した問いは、決定内容を該当する仕様書・設計書・ADRへ反映する
- 解決済みの大量履歴は open-questions.md に溜めず、ジャーナルと反映先文書を参照する

- [spec/_template.md](spec/_template.md) の形式に従って記述する

## 開発日誌（journal/）

- ファイル名は `YYYY-MM-DD.md` 形式（例: `2026-04-19.md`）
- 1日1ファイル。同じ日に複数回書く場合は同じファイルに追記する
- 同じ日付の中で複数回更新する場合も、セッション別の見出しを作らず、やったことを時系列順に追記する
- 開発日誌は履歴であり、現在の仕様・設計・運用の正本ではない
- 過去の日誌にある古い前提は、その時点の記録として残す。現行仕様を明確にする必要がある場合は、正本文書側を更新する
- セクション構成は [journal/_template.md](journal/_template.md) に従う
- ユーザーから `終了`、`ここまで`、`区切り` など開発の区切り・終了が明示された場合は、終了返答より先に日誌更新判断を行い、必要なら記録する
- それ以外では、日誌更新を明示的に指示された場合だけ記録する
- 未決定事項が生まれたら [dev/open-questions.md](dev/open-questions.md) にも追記する

### AIへの指示

- 作業開始時は `../AI.md`、[product/project.md](product/project.md)、[dev/phases.md](dev/phases.md) の順に必要な文脈を確認し、必要な場合だけジャーナルを読むこと
- 作業を行ったセッションは、ユーザーから開発の区切り・終了が明示された場合は終了返答より先に日誌更新判断を行い、必要なら当日のエントリーに記録を追記すること

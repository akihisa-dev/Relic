# docs/_rules.md

このファイルは `docs/` 以下のすべてのドキュメントに適用される作成・管理ルールです。
人間・AIを問わず、ドキュメントを作成・編集する際はこのルールに従ってください。

---

## 全般

- ドキュメントはすべて日本語で記述する
- 承認されていない内容は書かない
- 既存の内容は勝手に削除・変更しない
- リンクは標準Markdownの相対パスで記述する（例: `[仕様](spec/editor.md)`）
- 文書構成は、このプロジェクトだけでなく今後の開発でも雛形として再利用できるよう、役割・粒度・命名をできるだけ普遍的に保つ

---

## ファイル・フォルダ構成

- 新しいドキュメントを作る前に [dev/phases.md](dev/phases.md) と [INDEX.md](INDEX.md) を確認し、既存文書への追記で足りるか判断する
- 新規ファイルを作成したら [INDEX.md](INDEX.md) のドキュメントマップに追記する。ただし、既存文書と役割が重複する文書は作らない
- `_` で始まるファイル（`_rules.md`, `_template.md`）はメタドキュメント。内容ではなくルール・形式を定義する
- AI向けルールはルート直下の `AGENTS.md` に一元管理する。別のAI向け入口文書は作らない
- 現行仕様・設計・運用判断に使う正本と、履歴・テンプレートを混同しない
- 現行実装と文書が食い違う場合は、現行正本を実装に合わせて更新する。統合済み履歴を現行仕様のように書き換えない
- 履歴の扱いを説明する必要がある場合は、[INDEX.md](INDEX.md)、このルール文書、または現在フェーズ正本に追記する
- 旧フェーズ文書と旧日誌の履歴は、現在フェーズ正本の「過去作業記録（統合）」に集約する。現在の判断は [dev/phases.md](dev/phases.md) の `current` と現在フェーズ正本を優先する

---

## 各フォルダの用途

| フォルダ | 書くもの |
|----------|----------|
| `product/` | プロジェクト固有の意図・計画・用語 |
| `spec/` | 機能の振る舞い・詳細仕様 |
| `architecture/` | 設計・データモデル・意思決定記録 |
| `tech/` | 技術調査・ライブラリ選定 |
| `ui/` | 画面構成・遷移フロー |
| `dev/` | フェーズ管理・開発規約・未決定事項・テスト方針 |
| `assets/` | ロゴ案などの検討素材 |

アプリ本体の仕様だけでなく、AI運用、フェーズ管理、開発規約、検証方針、作業記録、入口文書、補助ファイルの役割分担も文書整理の対象に含める。

---

## 未決定事項の管理

- **仕様書・設計書・UI文書に「未決定事項」セクションを書いてはならない**
- 未決定の問いは必ず [dev/open-questions.md](dev/open-questions.md) に一元管理する
- 各問いには `関連:` フィールドで由来となる文書を明記する（どのファイルを見ればその問いの背景がわかるかを示す）
- 決定した問いは、決定内容を該当する仕様書・設計書・ADRへ反映する
- 解決済みの大量履歴は open-questions.md に溜めず、現在フェーズ正本の統合済み履歴と反映先文書を参照する

## 仕様書（spec/）

- [spec/_template.md](spec/_template.md) の形式に従って記述する

## 作業記録

- 作業記録は現在フェーズ正本へ書く
- 2026-05-24 13:29 JST以降の作業記録は `YYYY-MM-DD HH:mm JST` 形式の日時付きで書く
- 作業記録には、その作業単位で実施したことだけを簡潔に残す
- 現在フェーズ正本に作業記録を書いた場合は、その時点のすべての差分をまとめてコミットし、コミット後にpushする
- 未決定事項が生まれたら [dev/open-questions.md](dev/open-questions.md) にも追記する

## AIへの指示

- 作業開始時は `../AGENTS.md`、[product/project.md](product/project.md)、[dev/phases.md](dev/phases.md) の順に必要な文脈を確認すること
- 作業を行ったセッションは、ユーザーから開発の区切り・終了が明示された場合は終了返答より先に現在フェーズ正本の作業記録更新が必要か判断すること

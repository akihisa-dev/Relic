# P17

RelicのDiagram / 図解機能追加フェーズの正本。

このフェーズでは、Markdownファイルを正本として扱う前提を保ちながら、Diagram / 図解機能を小さな単位で実装・検証する。
仕様が未確定の部分は実装前にユーザーへ確認し、決定済みの範囲から進める。

---

## フェーズ固有ルール・制約

AIはこのフェーズを前提にユーザーへ接する。

### 読み込みルール

最初に読む:

- この文書

必要になったときだけ読む:

- プロダクト意図を確認する: `../../project/overview.md`
- 画面・見た目の正本を確認する: `../../design/DESIGN.md`
- 機能仕様に関わる場合: `../../features/`
- アーキテクチャやデータ構造に触れる場合: `../../engineering/architecture.md`, `../../engineering/data-model.md`, `../../engineering/decisions.md`
- 実装規約とテスト方針を確認する: `../coding-rules.md`, `../testing-rules.md`
- 未決定事項を扱う場合: `../questions.md`
- バージョンを切る場合: `../versioning-rules.md`

注意:

- 対象範囲は、ユーザーが明示したDiagram / 図解機能追加に限る。
- Markdownファイルを正本として扱う前提を変えない。
- Diagram / 図解機能が扱う情報の正本は、Markdown本文、リンク、タグ、フロントマター、コードブロックなど、Markdownに書ける情報を前提にする。
- Diagram / 図解機能の意味、対象データ、表示形式、編集可否、保存形式は、実装前にユーザーへ確認する。
- 仕様書、設計書、チェックリスト、詳細な作業項目は、ユーザーの明示指示または合意なしに作成しない。
- 新しい外部サービス、地図API、クラウド連携、位置情報利用、ネットワーク前提の技術は、事前に理由と選択肢を示してユーザー確認を取る。
- 大きな設計変更、新規大型機能追加、画面全体の方針変更は、事前に理由と選択肢を示してユーザー確認を取る。
- 未決定の問いは仕様書や設計書に書かず、必要になった場合だけ `../questions.md` に一元管理する。
- `../phases.md` の共通方針に従い、このフェーズでもDiagram / 図解機能追加を主テーマとしつつ、プロジェクト全体に関わる整理・修正・不具合修正をサブテーマとして扱う。

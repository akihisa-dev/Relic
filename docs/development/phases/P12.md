# P12

Relicのヴィジュアルデザイン・UIの調整・修正フェーズの正本。

このフェーズでは、既存のRelicの構造とローカルMarkdownアプリとしての実用性を保ちながら、画面の見た目、UI密度、余白、色、文字、操作部品、状態表示、レスポンシブ表示などの調整・修正を進める。

---

## フェーズ固有ルール・制約

AIはこのフェーズを前提にユーザーへ接する。

### 読み込みルール

最初に読む:

- この文書

必要になったときだけ読む:

- プロダクト意図を確認する: `../../project/overview.md`
- 画面・見た目の正本を確認する: `../../design/DESIGN.md`
- UI変更が機能仕様に関わる場合: `../../features/`
- アーキテクチャやデータ構造に触れる場合: `../../engineering/architecture.md`, `../../engineering/data-model.md`, `../../engineering/decisions.md`
- 実装規約とテスト方針を確認する: `../coding-rules.md`, `../testing-rules.md`
- 未決定事項を扱う場合: `../questions.md`
- バージョンを切る場合: `../versioning-rules.md`

注意:

- 対象範囲は、ユーザーが明示したヴィジュアルデザイン・UIの調整・修正に限る。
- UI変更では、ユーザーが指定した画面・場所・操作対象だけを変更する。
- 別の場所に同じ機能の入口を追加する、表示要素・説明・装飾を足す、操作方法を増やすなどの範囲拡張は事前に確認する。
- 見た目の調整でも、Markdown保存形式、ローカルファイル操作仕様、AI機能や外部サービス利用方針はユーザー確認なしに変更しない。
- 既存の設計正本 `../../design/DESIGN.md` と矛盾する調整が必要な場合は、理由と対象を説明してからユーザーに確認する。
- 仕様書、設計書、チェックリスト、詳細な作業項目は、ユーザーの明示指示または合意なしに作成しない。
- React Doctorなどの診断ツールは判断材料として使ってよいが、点数を上げるためだけに画面構造や状態管理を大きく作り替えない。
- 未決定の問いは仕様書や設計書に書かず、必要になった場合だけ `../questions.md` に一元管理する。

---

## 対象範囲

- 画面全体の見た目の調整
- 色、余白、境界線、文字サイズ、文字の強弱の調整
- レイアウト、サイドバー、レール、タブ、右パネル、ツールバー、ステータス表示のUI修正
- ボタン、入力欄、メニュー、トースト、モーダルなどの操作部品の見た目と状態表示の改善
- レスポンシブ表示、表示崩れ、重なり、はみ出しの修正
- アクセシビリティに関わるラベル、フォーカス表示、キーボード操作のUI面の改善
- 既存仕様に沿ったテスト追加・補強
- 変更に必要な最小限の関連文書更新

対象外:

- ユーザー確認なしの機能追加
- ユーザー確認なしの画面全体の設計方針変更
- ユーザー確認なしのMarkdown保存形式変更
- ユーザー確認なしのローカルファイル操作仕様変更
- ユーザー確認なしのAI機能や外部サービス利用方針変更
- 配布ビルド確認

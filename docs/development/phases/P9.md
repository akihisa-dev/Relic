# P9

Relicの全リファクタリングフェーズの正本。

このフェーズでは、Relicの外部仕様、Markdown保存形式、既存機能、画面上の操作導線を維持したまま、内部実装の見通し、責務分離、重複削減、型安全性、テストしやすさを整理する。

---

## フェーズ固有ルール・制約

AIはこのフェーズを前提にユーザーへ接する。

### 読み込みルール

最初に読む:

- この文書

必要になったときだけ読む:

- プロダクト意図を確認する: `../../project/overview.md`
- 既存仕様との整合を見る: `../../features/`
- 画面・遷移・見た目を確認する: `../../design/screens.md`, `../../design/navigation.md`, `../../design/system.md`
- アーキテクチャやデータ構造に触れる場合: `../../engineering/architecture.md`, `../../engineering/data-model.md`, `../../engineering/decisions.md`
- 実装規約とテスト方針を確認する: `../coding-rules.md`, `../testing-rules.md`
- 未決定事項を扱う場合: `../questions.md`
- バージョンを切る場合: `../versioning-rules.md`

注意:

- 対象範囲は、既存機能を壊さないための内部整理、分割、重複削減、型整理、テスト補強に限る。
- リファクタリングは小さい単位に分け、変更前後で同じ挙動を保っていることを検証する。
- 外部仕様、Markdown保存形式、IPCの公開的な意味、ファイル操作仕様、画面の操作導線、UI文言は、ユーザー確認なしに変更しない。
- 大規模な設計変更、状態管理の入れ替え、保存処理やファイル操作の再設計は、実装前に対象範囲と理由をユーザーへ確認する。
- React Doctorなどの診断ツールは判断材料として使ってよいが、点数を上げるためだけに構造変更しない。
- 未決定の問いは仕様書や設計書に書かず、必要になった場合だけ `../questions.md` に一元管理する。

---

## 対象範囲

- Reactコンポーネント、hooks、状態管理、CSS構成の整理
- main / preload / renderer / shared の責務境界の整理
- IPCハンドラー、ファイル処理、検索、AIワークスペース関連処理の見通し改善
- 重複コード、長すぎる関数、テストしにくい処理の分割
- 型定義、バリデーション、エラー処理の一貫性整理
- 既存テストの補強、関連テストの整理、`pnpm verify` が通る状態の維持

対象外:

- ユーザー確認なしの新機能追加
- ユーザー確認なしの画面全体リデザイン
- Markdownファイル保存形式の変更
- ローカルファイル操作仕様の変更
- AI機能や外部サービス利用方針の変更
- 配布ビルド確認

# P13

Relicの全体整理・リファクタリング・不具合修正フェーズの正本。

このフェーズでは、既存機能とユーザー体験を保ちながら、コード、構造、テスト、文書、既知の不具合を小さな単位で整理・修正する。

---

## フェーズ固有ルール・制約

AIはこのフェーズを前提にユーザーへ接する。

### 読み込みルール

最初に読む:

- この文書

必要になったときだけ読む:

- プロダクト意図を確認する: `../../project/overview.md`
- 画面・見た目の正本を確認する: `../../design/DESIGN.md`
- 機能仕様に関わる場合: `../../features/`
- アーキテクチャやデータ構造に触れる場合: `../../engineering/architecture.md`, `../../engineering/data-model.md`, `../../engineering/decisions.md`
- 実装規約とテスト方針を確認する: `../coding-rules.md`, `../testing-rules.md`
- 未決定事項を扱う場合: `../questions.md`
- バージョンを切る場合: `../versioning-rules.md`

注意:

- 対象範囲は、ユーザーが明示した全体整理、リファクタリング、不具合修正に限る。
- リファクタリングでは、既存の外部仕様、Markdown保存形式、ローカルファイル操作仕様、AI機能や外部サービス利用方針をユーザー確認なしに変更しない。
- 不具合修正では、原因を確認し、修正範囲を問題のある箇所と必要最小限の関連箇所に限定する。
- 大きな設計変更、機能追加、画面全体の方針変更は、事前に理由と選択肢を示してユーザー確認を取る。
- 仕様書、設計書、チェックリスト、詳細な作業項目は、ユーザーの明示指示または合意なしに作成しない。
- React Doctorなどの診断ツールは判断材料として使ってよいが、点数を上げるためだけに状態管理、保存、ファイル操作、エディタ本体を大きく作り替えない。
- 未決定の問いは仕様書や設計書に書かず、必要になった場合だけ `../questions.md` に一元管理する。

---

## 対象範囲

- コード構造の整理
- 重複や責務の整理
- 既存機能を保ったリファクタリング
- 不具合の調査と修正
- テストの追加・補強・整理
- 文書の最小限の整合性更新
- 依存関係や設定の小規模な整理

対象外:

- ユーザー確認なしの機能追加
- ユーザー確認なしの仕様変更
- ユーザー確認なしの画面全体のデザイン方針変更
- ユーザー確認なしのMarkdown保存形式変更
- ユーザー確認なしのローカルファイル操作仕様変更
- ユーザー確認なしのAI機能や外部サービス利用方針変更
- 配布ビルド確認

# P4

Relicの機能追加フェーズの正本。

このフェーズでは、Relicの目的に沿って、創作設定の作成、保管、閲覧を支える機能追加に必要な範囲だけを扱う。

---

## フェーズ固有ルール・制約

AIはこのフェーズを前提にユーザーへ接する。

### 読み込みルール

最初に読む:

- この文書

必要になったときだけ読む:

- プロダクト意図を確認する: `../../project/overview.md`
- 対象機能の仕様を確認する: `../../features/`
- 画面・遷移・見た目を確認する: `../../design/screens.md`, `../../design/navigation.md`, `../../design/system.md`
- アーキテクチャやデータ構造に触れる場合: `../../engineering/architecture.md`, `../../engineering/data-model.md`, `../../engineering/decisions.md`
- 実装規約とテスト方針を確認する: `../coding-rules.md`, `../testing-rules.md`
- バージョンを切る場合: `../versioning-rules.md`

注意:

- 対象範囲は機能追加に限る。
- 具体的な追加機能は、ユーザーに1件ずつ確認してから決める。
- ユーザーが「機能を追加したい」と言っただけで、仕様書、設計書、チェックリスト、実装を勝手に作成しない。
- 仕様書を作成・更新する場合は、ユーザーへ1問ずつヒアリングし、承認を得てからファイルに書く。
- 技術選定が必要な場合は、選択肢と推奨理由を分かりやすく提示し、ユーザーの決定を待つ。
- 既存機能のUI/UX改善だけを目的とする変更は、機能追加と混ぜずに扱う。

---

## 対象範囲

- 新しいユーザー向け機能
- 既存機能に対する明確な機能追加
- 機能追加に必要な仕様書、設計文書、実装、テスト
- 機能追加に伴って必要になる最小限のUI、データ構造、IPC、保存形式の変更

対象外:

- ユーザー確認なしの機能候補整理
- ユーザー確認なしの画面全体リデザイン
- 機能追加と関係しない見た目だけの調整
- 配布ビルド確認
- 既存Markdownファイルの一括変更

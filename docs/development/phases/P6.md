# P6

Relicのコンセプト再定義と安定化準備フェーズの正本。

このフェーズでは、Relicの根底を「Markdownに書ける情報をMarkdownファイルのまま保ち、その情報をもとに編集・閲覧・検索・可視化・出力を拡張するローカルアプリ」として整理し、創作設定を代表用途のひとつに位置づけながら、複数用途を前提にした安定化準備を扱う。

---

## フェーズ固有ルール・制約

AIはこのフェーズを前提にユーザーへ接する。

### 読み込みルール

最初に読む:

- この文書

必要になったときだけ読む:

- プロダクト意図を確認する: `../../project/overview.md`
- 既存仕様との整合を見る: `../../features/`
- 画面・遷移・見た目を確認する: `../../design/screens.md`, `../../design/navigation.md`, `../../design/system.md`
- アーキテクチャやデータ構造に触れる場合: `../../engineering/architecture.md`, `../../engineering/data-model.md`, `../../engineering/decisions.md`
- 実装規約とテスト方針を確認する: `../coding-rules.md`, `../testing-rules.md`
- バージョンを切る場合: `../versioning-rules.md`

注意:

- 対象範囲はコンセプト再定義、代表用途の整理、安定化準備に限る。
- Relicの根底は、創作設定専用アプリではなく、Markdownに書ける情報をMarkdownファイルのまま活かして拡張するアプリとして扱う。
- 創作設定は重要な代表用途だが、唯一の主要用途として固定しない。
- 用途例や説明文を決める場合は、ユーザーへ1問ずつ確認し、承認された内容だけを正本化する。
- コンセプト整理を理由に、仕様書、設計書、UI文書、機能一覧、チェックリストを勝手に作成・更新しない。
- 安定化準備は、コンセプト整理後に確認された範囲でのみ扱う。
- 新機能追加、画面全体の刷新、用途別テンプレート作成、配布ビルド確認は、ユーザーが明示した場合だけ扱う。

## 対象範囲

- Relicの根底コンセプトの言語化
- 創作設定、研究、学習、個人Wiki、プロジェクト資料など、複数の代表用途の整理
- 既存ドキュメントに書かれている「創作設定用」という表現の扱いの見直し
- コンセプト整理に基づく安定化準備の範囲決め
- 安定化準備に必要な最小限の仕様書、設計文書、実装、テスト

対象外:

- ユーザー確認なしのコンセプト確定
- ユーザー確認なしの用途一覧作成
- ユーザー確認なしの新機能追加
- ユーザー確認なしの画面全体リデザイン
- 配布ビルド確認
- 既存Markdownファイルの一括変更

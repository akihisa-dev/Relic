# development/development.md

Relicの開発作業全体に適用する開発ルール。
AI固有の対話・判断ルールはリポジトリルートの `AGENTS.md` を正本とし、この文書にはRelic開発そのものの運用を定義する。

---

## この文書の役割

この文書では、Relicの開発作業における作業範囲、文書更新、検証、コミット、Issue対応、バージョン更新の扱いを定義する。

この文書では、次の詳細は扱わない。

- コードの書き方: [coding-rules.md](coding-rules.md)
- テスト方針の詳細: [testing-rules.md](testing-rules.md)
- バージョン番号の詳細: [versioning-rules.md](versioning-rules.md)
- リリース手順: [release-rules.md](release-rules.md)
- ドキュメント作成・管理の詳細: [../document-rules.md](../document-rules.md)
- AIエージェントの対話、確認、委任、説明ルール: [../../AGENTS.md](../../AGENTS.md)

---

## 作業開始時の確認

- 作業前に [../project/overview.md](../project/overview.md) でRelicの目的と判断思想を確認する
- 作業内容に応じて、[../INDEX.md](../INDEX.md) から必要な現行正本文書を確認する
- 実装に入る場合は [coding-rules.md](coding-rules.md) を確認する
- 検証方針を決める場合は [testing-rules.md](testing-rules.md) を確認する
- 未決定事項を扱う場合は [questions.md](questions.md) を確認する

作業範囲は、ユーザーの直近指示、対象機能の現行正本文書、実装上の直接関係で判断する。

---

## 作業範囲

- ユーザーが明示した対象ファイル・対象範囲を越えて編集しない
- 関連ファイルの更新が必要な場合は、理由、対象ファイル、変更内容を明確にし、ユーザーの明示範囲を越える場合は事前に確認する
- 「整合性のため」「慣例上必要」「セッション終了時の手順」などを理由に、ユーザーの直近指示を越えて作業しない
- UI変更では、ユーザーが指定した画面、場所、操作対象だけを変更する
- 別の場所に同じ機能の入口を追加する、操作方法を増やす、説明や装飾を足すなどの範囲拡張は、事前に確認する
- ユーザーが対象を訂正した場合は、以後その対象以外を触らない
- ユーザーから明示的な指示がない限り、ブランチは切らずに現在のブランチで作業する
- 変更の取り消し、巻き戻し、履歴の書き換えは、ユーザーが明示した場合だけ実行する

---

## 仕様・設計・技術選定

- 仕様書（`docs/features/`）を作成・更新する場合は、ユーザーに1問ずつヒアリングする
- ユーザーの回答をもとに仕様を決定し、ユーザーの承認を得てからファイルに書き込む
- 技術的な選択肢が生じた場合は、各選択肢を一般の人にも分かる言葉で説明し、ユーザーに決定を委ねる
- 推薦を示す場合は、推薦理由と採用しない場合の影響を説明する
- ユーザーが「技術的なことはわからない」と言った場合も、説明を言い換えて判断材料を出す
- 説明なく独自判断で技術選定しない

---

## 未決定事項

- 仕様書、設計書、UI文書に「未決定事項」セクションを書かない
- 未決定の問いは [questions.md](questions.md) に一元管理する
- 各問いには `関連:` フィールドで由来となる文書を明記する
- [questions.md](questions.md) は未決定事項が発生したときだけ読む

---

## 文書更新

- 仕様変更を伴う実装をした場合は、該当する `docs/features/` または `docs/engineering/` の正本文書を更新する
- 画面構成、操作、見た目の判断を変えた場合は、必要に応じて `docs/design/` を更新する
- 技術選定を変更・追加した場合は、`docs/engineering/` と必要に応じて `docs/engineering/decisions.md` を更新する
- 未決定の問いが生まれた場合は [questions.md](questions.md) に追加する
- 文書作成・編集そのもののルールは [../document-rules.md](../document-rules.md) に従う
- 作業履歴を文書内に蓄積しない

作業内容に対して更新すべき文書がある場合は、文書更新を同じ作業単位に含める。
必要な文書更新を未反映のまま完了扱いにしない。

---

## 検証

- 検証範囲は、変更内容のリスクと影響範囲に合わせて決める
- 文書整理だけの作業では、対象文書と参照先文書の整合を確認し、`git diff --check` を最低確認にする
- コード、仕様、UI、保存形式、IPC、ファイル操作に触れた場合は、対象テスト、`pnpm typecheck`、`pnpm test`、実アプリ確認のうち必要な範囲を実施する
- 実施した検証は完了報告、Issue、Pull Request、またはコミット本文で確認できるようにする
- 実施しなかった検証がある場合は、その理由を明記する

`app/out/` 配下のパッケージ版アプリは、ユーザーが配布ビルド確認を明示した場合だけ確認対象にする。
テスト用ワークスペース、検証用リポジトリ、一時データをRelic開発リポジトリ内に作らない。
実アプリ検証でユーザーのワークスペースや外部サービスに触る必要がある場合は、対象、操作内容、戻し方を説明し、ユーザーの明示許可を得る。

## 診断ツール

React Doctorなどの診断ツールは、Relicの品質確認として定期的に利用してよい。
ただし、診断ツールの点数や指摘を盲信しない。

- React Doctorは、大きめの実装後、リリース前、依存更新後、または月1回程度の定期点検として実行する
- 指摘は「実害があるもの」「Relicの性質と合わないもの」「大規模な設計変更として別作業にすべきもの」に分けて判断する
- 実害がある指摘は、ユーザーの指示範囲に収まる場合に修正する
- Electron、CodeMirror、ローカルファイル操作、順序が重要な処理など、Relicの性質と合わない指摘は、理由を説明したうえで設定除外または保留にしてよい
- 設定除外を追加・変更する場合は、除外理由を説明し、必要に応じて関連する現行正本文書にも記録する
- 点数を上げるためだけに、保存、ファイル操作、エディタ本体、状態管理などの大規模な設計変更を行わない

## コミット

AIが実装、修正、検証、調査、文書整理を行った場合は、完了報告前に必要な確認を終え、必要な文書更新を反映したうえでコミットする。

- コミット時点でワークツリーに残っているすべての差分をまとめてステージングする
- 「自分が直接編集したファイルだけ」を選別してコミットしない
- コミット前に、必要な確認を終える
- ユーザーが「コミットしない」「まだコミットしない」と明示した場合は、その指示を優先する
- pushはユーザーが明示的に指示した場合だけ実行する

### コミットメッセージ

コミットメッセージは日本語で書く。
文書内に作業履歴を蓄積しない代わりに、コミット本文を後から作業内容が分かる粒度で書く。

件名は、変更の目的が分かる短い一文にする。
本文には原則として次の項目を書く。

```text
目的:
- なぜこの変更を行ったか

内容:
- 何を追加、変更、削除したか
- 重要な判断や置き換えがあれば何をどう変えたか

確認:
- 実行した確認
- 実行しなかった確認がある場合は、その理由

影響:
- ユーザーに見える変更
- 互換性、運用、文書導線への影響
```

単純な誤字修正などでも、件名または本文だけで目的と内容が分かるようにする。
複数ファイル変更、削除、仕様変更、運用変更、バージョン更新を含む場合は、本文を省略しない。

---

## GitHub Issue対応

GitHub Issueに対応した場合は、対応内容、確認したテスト・検証、関連コミットをIssueへコメントする。
コメント後、対応が完了しているIssueは `completed` として閉じる。
Issueを閉じない理由がある場合は、その理由をIssueコメントまたは完了報告に明記する。

---

## バージョン更新

バージョン番号は [versioning-rules.md](versioning-rules.md) に従う。
バージョンを上げる場合は、`app/package.json` の変更を同じコミットに含める。
MAJORとMINORは、ユーザーが明示的に指示した場合だけ変更する。

---

## 完了報告

完了報告では、次の内容を簡潔に明記する。

- 実施した変更
- 更新した文書、または更新不要と判断した文書
- 実施した検証
- 実施しなかった検証がある場合は理由
- コミットした場合はコミットID
- pushしていない場合は未pushであること

# spec/frontmatter.md

フロントマター機能の詳細仕様書。
Markdownファイルに付与する構造化メタデータの考え方、入力補助、表示、追加・削除の振る舞いを定義する。

---

## フロントマターとは

フロントマターは、Markdownファイル先頭の `---` で囲まれた YAML ブロック。

Relicでは、フロントマターをアプリがノートへ一律に押し付ける属性ではなく、ユーザーが任意のMarkdownファイルに後から付与できるメタ情報として扱う。

```yaml
---
tags: [小説, キャラクター]
状態: [下書き]
締切: [2026-05-20]
---
```

フロントマターの実体は、常に各 `.md` ファイル先頭にあるYAMLである。Relicはこれを隠れた内部データとして扱わず、Markdownファイルの一部として尊重する。

---

## 基本原則

- デフォルトのフロントマター項目は作らない
- 新規Markdownファイルに自動でフロントマターを付与しない
- フロントマターは任意のMarkdownファイルへ、いつでも追加できる
- 追加したフロントマターは、いつでもファイル単位で削除できる
- フロントマターの追加・削除で本文を壊さない
- 登録されていないYAMLキーも壊さず保持する
- 登録済みキーは検索・絞り込み・入力補助に使える
- YAMLを直接書かなくても扱えるように、Relicはフォーム表示と入力補助を提供する

本文は、あくまでユーザー自身が記述する場所である。Relicは本文領域に対して、ユーザーの明示操作なしに構造追加・自動挿入・自動整形を行わない。

---

## 表示と編集

フロントマターはMarkdownファイルの先頭に実在する特殊ブロックとして扱う。

- ライブプレビューでは、YAMLブロックを本文より控えめなフォームとして表示・編集できる
- ソースモードでは、フォーム化せずMarkdownファイル内のYAMLとして直接表示・編集する
- フォーム表示では、既存フィールドの値編集、フィールド追加、フィールド削除、配列値の追加ができる
- フロントマター表示は本文の見出しや本文テキストより強く見えないようにする

入力能力、候補、固定プロパティの説明など、ワークスペース横断の設定は、右パネルではなくメインエリアのフロントマター設定パネルタブで扱う。
右パネルはアウトラインとリンク一覧の補助表示に限定し、フロントマター構造の管理には使わない。

---

## ファイルタブで行うこと

- このMarkdownファイルにフィールドを追加する
- このMarkdownファイルからフィールドを削除する
- 既存フィールドの値を編集する
- 配列値に項目を追加する
- 必要ならソースモードでYAMLとして確認・編集する

フロントマターがないファイルにフィールドを追加する場合、Relicはファイル先頭にYAMLブロックを作成する。

```md
# 今日のメモ

本文...
```

`締切` フィールドを追加した場合：

```yaml
---
締切:
---
# 今日のメモ
```

既にフロントマターがある場合は、既存YAMLブロック内にキーを追加する。同じキーが既にある場合は、既存値を勝手に上書きしない。

---

## フロントマター設定パネルタブ

フロントマター設定パネルタブは、ワークスペース内で使うプロパティの入力能力を管理する場所として扱う。

フロントマター設定パネルタブで行うこと：

- 固定プロパティの意味と書き方を確認する
- カスタムプロパティを追加する
- カスタムプロパティの入力能力を選ぶ
- 選択肢・複数候補の候補値を管理する
- カスタムプロパティを削除する

ファイル単位の値編集、フィールド追加、フィールド削除は、対象Markdownファイルのファイルタブ内で行う。

---

## 能力付きフィールド

フロントマターの中心は、固定テンプレートではなく、能力付きフィールドである。

フィールドは任意の名前を持てる。その名前に対して、Relic側で入力能力を付与する。

例：

| フィールド名 | 入力能力 |
|--------------|----------|
| `締切` | 日付入力 |
| `状態` | 選択肢入力 |
| `登場人物` | 複数候補入力 |
| `参考URL` | URL入力 |

カスタムプロパティでは、ユーザーが任意の名前に任意の入力能力を割り当てられる。ただし、`aliases`、`tags`、`status`、`chronicle`、`date`、`plannedDate`、`actualDate` はRelicが意味を知っている予約名として扱い、カスタムプロパティには追加しない。

カスタムプロパティもMarkdown上では通常のYAMLキーとして書く。単体値でも `[]` を使い、固定プロパティと同じ1行配列に揃える。フロントマター設定パネルタブでは、入力能力ごとに実際の書き方例を表示する。

```yaml
状態: [下書き]
巻数: [3]
締切: [2026-05-20]
公開日時: [2026-05-20T09:30]
公開時刻: [09:30]
公開済み: [true]
登場人物: [Alice, Bob]
参考URL: [https://example.com]
```

---

## 入力能力

Relicは、登録されたフィールドの入力能力に応じてフォームUIと補助機能を提供する。

| 入力能力 | UI・補助 |
|----------|----------|
| テキスト | 通常のテキスト入力 |
| 数値 | 数値入力 |
| 日付 | カレンダー入力 |
| 日時 | 日時入力 |
| 時刻 | 時刻入力 |
| 真偽値 | トグル |
| 選択肢 | 候補から選ぶ入力 |
| 複数候補 | ピル入力、候補補完 |
| URL | URL入力 |

候補を持つ入力能力では、事前に登録した候補や、ワークスペース内で過去に使われた値をもとに、ユーザー入力に応じて候補を絞り込む。

---

## 固定プロパティ

`aliases`、`tags`、`status`、`chronicle`、`plannedDate`、`actualDate` はカスタム入力能力ではなく、Relicが意味を知っている固定プロパティとして扱う。`date` は過去互換の予定日フィールドとして読み書きできる予約名だが、フロントマター設定パネルタブの固定プロパティ一覧では `plannedDate` を主として扱う。

固定プロパティも、単体値でも複数値・期間でも1行の配列で書く。Relicのフォームで編集した場合も、この形で書き戻す。

```yaml
aliases: [帝都, 王都]
tags: [資料, 下書き]
status: [未着手]
chronicle: [1185, 1333]
plannedDate: [2026-05-12, 2026-05-20]
actualDate: [2026-05-12, 2026-05-20]
```

`tags` に書かれた値だけをRelicのタグとして扱う。本文中の `#タグ` 記法はタグ化しない。本文中で概念・人物・資料などを参照する場合は、インラインリンク `[[...]]` を使う。

タグはその場で追加できる。ただし、表記揺れを避けるため、Relicはワークスペース内の既存 `tags` から候補を出し、前後の空白や先頭の `#` はタグ値として正規化する。

`status` は、日付チャートの状態表示と絞り込みに使う固定プロパティである。候補は `未着手`、`進行中`、`完了`、`中断`、`中止` とし、フォーム編集では1つだけを選んで1要素の配列として書き戻す。

```yaml
status: [未着手]
```

`chronicle` は、そのMarkdownファイル自体を年表上の単年または期間に置くための固定プロパティである。値は常に1行の整数配列で書く。

- `[年]` は単年として扱う
- `[開始年, 終了年]` は期間として扱う
- 年は整数だけを有効とする
- `0` 年は無効とする
- 紀元前は負数、紀元後は正数で表す（例: `chronicle: [-300]`, `chronicle: [-300, 250]`）
- 期間は開始年が終了年以下の場合だけ有効とする
- チャートタブでは、ファイル名を左列に固定表示し、右側の横軸年表に単年または期間を塗りとして表示する
- 単年はその年、期間は開始年から終了年までの範囲を塗り、塗りの上には年または期間を表示する
- チャートはグリッドで表示し、表示スケールを変更できる
- チャートバーはクリックしてもファイルを開かない
- チャートバー本体をドラッグすると、対応Markdownファイルの `chronicle` を移動後の年または期間へ書き戻す
- 対応Markdownファイルに有効な `plannedDate`、`actualDate`、または互換用の `date` がある場合、`chronicle` バーの移動・長さ変更に合わせて日付側も同じ年差で書き戻す

`plannedDate` と `actualDate` は、そのMarkdownファイルの予定日・実績日または期間を表すための固定プロパティである。値は常に1行の日付配列で書く。既存ファイルに `plannedDate` がなく `date` がある場合、Relicは `date` を予定日として読む。

- `[YYYY-MM-DD]` は単日として扱う
- `[開始日, 終了日]` は期間として扱う
- 入力は年月日のみとし、時刻は扱わない
- 期間入力時は、開始日と終了日をフォームで入力できる
- フォーム編集後は `plannedDate: [2026-05-12]`、`plannedDate: [2026-05-12, 2026-05-20]`、`actualDate: [2026-05-12]`、`actualDate: [2026-05-12, 2026-05-20]` の形で書き戻す
- チャートタブの日付表示では、`plannedDate` と `actualDate` を同じファイル行の予定・実績として表示する
- 実績日のバーには `status` の値を状態バッジとして表示する
- チャートタブの日付表示では、検索に加えて `status` の固定候補で絞り込める
- 日付バー本体をドラッグすると、対応Markdownファイルの `plannedDate` または `actualDate` を移動後の日付または期間へ書き戻す
- 日付バー端をドラッグすると、対応Markdownファイルの `plannedDate` または `actualDate` の開始日または終了日を書き戻す
- `plannedDate` バーを更新すると、既存ファイルに互換用の `date` がある場合は `date` も同じ値へ書き戻す
- 日付バーの移動・長さ変更後の日付範囲から年を取り出し、対応Markdownファイルの `chronicle` も同じ年または期間へ書き戻す

## 削除方法

フロントマターの削除は、少なくとも以下を分けて扱う。

| 操作 | 内容 |
|------|------|
| このファイルから削除 | 現在のMarkdownファイルのYAMLからキーを削除する |
| 入力補助の登録を削除 | Relicがそのキーに付与していた入力能力や候補を削除する |
| 全ファイルから削除 | 現行UIには持たない。ワークスペース内のMarkdownファイルから該当キーを一括削除する場合は別作業として扱う |

通常の削除操作は「このファイルから削除」とする。入力補助の登録を削除しても、既存MarkdownファイルのYAML自体は勝手に消さない。

全ファイルから削除する操作は影響が大きいため、現行アプリの通常UIには置かない。

最後のフィールドを削除して空のフロントマターだけが残る場合、Relicは空のYAMLブロックを削除する。

## フロントマターテンプレート

フロントマターテンプレートは、複数の能力付きフィールド名をまとめるアプリ設定上の保存済みセットである。

現行UIでは、テンプレート管理画面やテンプレート適用操作は表示しない。テンプレート定義を扱う場合も、ワークスペース内の `templates/` フォルダや `.md` テンプレートファイルには依存しない。

例：

```yaml
---
type: [book]
title: []
author: []
状態: [読書中]
tags: []
---
```

テンプレート由来のキーも、Markdownに書き込まれた後は通常のYAMLとして扱う。ファイルごとに自由に追加・削除・編集でき、以後そのファイルがテンプレートに固定されるわけではない。

---

## 未認識フィールド

既存ファイルや外部エディタで作成されたYAMLには、Relicが入力能力を知らないキーが含まれる場合がある。

Relicは未認識フィールドを壊さず保持する。未認識フィールドはテキストとして表示でき、必要に応じて後から入力能力を割り当てられる。

---

## 検索・フィルターとの連動

登録済みフィールドは、構造的な検索・絞り込みの対象にできる。

未登録フィールドもファイル内のYAMLとして存在するため、全文検索の対象には含まれる。ただし、入力能力や型を前提にした構造的な絞り込みは、Relicが認識しているフィールドを対象とする。

---

## 一文での定義

Relicのフロントマターは、固定スキーマではなく、ユーザーが任意のMarkdownファイルへ後から自由に付与・削除できる、能力付きメタ情報フィールドである。

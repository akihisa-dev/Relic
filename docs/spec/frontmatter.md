# spec/frontmatter.md

フロントマター機能の詳細仕様書。Markdownファイルに付与する構造化メタデータの考え方、入力補助、表示、追加・削除の振る舞いを定義する。

---

## フロントマターとは

フロントマターは、Markdownファイル先頭の `---` で囲まれた YAML ブロック。

Relicでは、フロントマターをアプリがノートへ一律に押し付ける属性ではなく、ユーザーが任意のMarkdownファイルに後から付与できるメタ情報として扱う。

```yaml
---
tags: [小説, キャラクター]
状態: 下書き
締切: 2026-05-20
---
```

フロントマターの実体は、常に各 `.md` ファイル先頭にあるYAMLである。Relicはこれを隠れた内部データとして扱わず、Markdownファイルの一部として尊重する。

---

## 基本原則

- デフォルトのフロントマター項目は作らない
- 新規Markdownファイルに自動でフロントマターを付与しない
- フロントマターは任意のMarkdownファイルへ、いつでも追加できる
- 追加したフロントマターは、いつでもファイル単位で削除できる
- フロントマターの追加・削除で本文を壊さない
- 登録されていないYAMLキーも壊さず保持する
- 登録済みキーは検索・絞り込み・入力補助に使える
- YAMLを直接書かなくても扱えるように、Relicはフォーム表示と入力補助を提供する

本文は、あくまでユーザー自身が記述する場所である。Relicは本文領域に対して、ユーザーの明示操作なしに構造追加・自動挿入・自動整形を行わない。

---

## 表示と編集

フロントマターはMarkdownファイルの先頭に実在する特殊ブロックとして扱う。RelicはそのYAMLブロックをフォームとしてレンダリングし、ユーザーがYAML構文を手打ちしなくても値を編集できるようにする。

メインパネルは本文を書く場所として守る。メインパネルでは、フロントマターが存在することを確認でき、必要に応じて値の確認や軽い編集ができる。ただし、フィールド追加・削除・テンプレート適用・入力能力の設定など、構造を変える操作は右パネルで行う。

フロントマター表示は、本文より控えめなメタ情報領域として扱う。本文の見出しや本文テキストより強く見えないようにし、入力UIはコンパクトにまとめる。

必要な場合は、ユーザーがYAMLとして確認・編集できる表示に切り替えられる。

---

## 右パネルの役割

右パネルは、フロントマター操作の道具箱として扱う。

右パネルで行うこと：

- このMarkdownファイルにフィールドを追加する
- このMarkdownファイルからフィールドを削除する
- 能力付きフィールドを選ぶ
- 複数フィールドをまとめて追加する
- よく使う組み合わせをテンプレートとして適用する
- 候補値を管理する
- 入力能力を変更する
- 未認識のYAMLキーに入力能力を割り当てる
- 必要ならYAMLとして確認・編集する

---

## 能力付きフィールド

フロントマターの中心は、固定テンプレートではなく、能力付きフィールドである。

フィールドは任意の名前を持てる。その名前に対して、Relic側で入力能力を付与する。

例：

| フィールド名 | 入力能力 |
|--------------|----------|
| `締切` | 日付入力 |
| `状態` | 選択肢入力 |
| `登場人物` | 複数候補入力 |
| `tags` | タグ候補入力 |
| `参考URL` | URL入力 |

`status` だから選択式、`date` だから日付、という固定的な扱いにはしない。ユーザーが任意の名前に任意の入力能力を割り当てられる。

カスタムプロパティもMarkdown上では通常のYAMLキーとして書く。Relicの設定画面では、入力能力ごとに実際の書き方例を表示する。

```yaml
状態: 下書き
巻数: 3
締切: 2026-05-20
公開日時: 2026-05-20T09:30
公開時刻: 09:30
公開済み: true
登場人物: [Alice, Bob]
参考URL: https://example.com
```

---

## 入力能力

Relicは、登録されたフィールドの入力能力に応じてフォームUIと補助機能を提供する。

| 入力能力 | UI・補助 |
|----------|----------|
| テキスト | 通常のテキスト入力 |
| 数値 | 数値入力 |
| 日付 | カレンダー入力 |
| 真偽値 | トグル |
| 選択肢 | 候補から選ぶ入力 |
| 複数候補 | ピル入力、候補補完 |
| URL | URL入力 |

候補を持つ入力能力では、事前に登録した候補や、ワークスペース内で過去に使われた値をもとに、ユーザー入力に応じて候補を絞り込む。

`aliases`、`tags`、`chronicle` はカスタム入力能力ではなく、Relicが意味を知っている固定プロパティとして扱う。

`aliases`、`tags`、`chronicle` は、単体値でも複数値・期間でも1行の配列で書く。Relicのフォームで編集した場合も、この形で書き戻す。

```yaml
aliases: [帝都]
aliases: [帝都, 王都]
tags: [資料]
tags: [資料, 下書き]
chronicle: [1185]
chronicle: [1185, 1333]
```

`tags` に書かれた値だけをRelicのタグとして扱う。本文中の `#タグ` 記法はタグ化しない。本文中で概念・人物・資料などを参照する場合は、インラインリンク `[[...]]` を使う。

タグはその場で追加できる。ただし、表記揺れを避けるため、Relicはワークスペース内の既存 `tags` から候補を出し、前後の空白や先頭の `#` はタグ値として正規化する。

`chronicle` は、そのMarkdownファイル自体を年表上の単年または期間に置くための固定プロパティである。値は常に1行の整数配列で書く。

- `[年]` は単年として扱う
- `[開始年, 終了年]` は期間として扱う
- 年は整数だけを有効とする
- `0` 年は無効とする
- 紀元前は負数、紀元後は正数で表す（例: `chronicle: [-300]`, `chronicle: [-300, 250]`）
- 期間は開始年が終了年以下の場合だけ有効とする
- 年表モードでは、ファイル名を左列に固定表示し、右側の横軸年表に単年または期間を塗りとして表示する
- 単年はその年、期間は開始年から終了年までの範囲を塗り、塗りの上には年または期間を表示する
- 年表はグリッドで表示し、表示スケールを変更できる

## 追加方法

フロントマターは、以下の粒度で追加できる。

- 単体フィールドを追加する
- 複数フィールドを選んで追加する
- よく使う組み合わせをテンプレートとして追加する
- 既存YAMLキーに後から入力能力を割り当てる

テンプレートは便利機能だが、中心ではない。中心は、書いたあとからでも任意のMarkdownファイルに必要なフィールドを足せることである。

フロントマターがないファイルにフィールドを追加する場合、Relicはファイル先頭にYAMLブロックを作成する。

```md
# 今日のメモ

本文...
```

`締切` フィールドを追加した場合：

```yaml
---
締切:
---
# 今日のメモ

本文...
```

既にフロントマターがある場合は、既存YAMLブロック内にキーを追加する。同じキーが既にある場合は、既存値を勝手に上書きしない。

## 削除方法

フロントマターの削除は、少なくとも以下を分けて扱う。

| 操作 | 内容 |
|------|------|
| このファイルから削除 | 現在のMarkdownファイルのYAMLからキーを削除する |
| 入力補助の登録を削除 | Relicがそのキーに付与していた入力能力や候補を削除する |
| 全ファイルから削除 | ワークスペース内のMarkdownファイルから該当キーを一括削除する |

通常の削除操作は「このファイルから削除」とする。入力補助の登録を削除しても、既存MarkdownファイルのYAML自体は勝手に消さない。

全ファイルから削除する操作は影響が大きいため、通常UIには置かないか、強い確認を必要とする。

最後のフィールドを削除して空のフロントマターだけが残る場合、Relicは空のYAMLブロックを削除する。

---

## フロントマターテンプレート

フロントマターテンプレートは、複数の能力付きフィールドをまとめて現在のMarkdownファイルに付与するための保存済みセットである。

テンプレートは、単一目的のワークスペースを前提にした固定スキーマではない。広範な目的を含むワークスペースでも、ユーザーがそのとき必要なテンプレートを選び、任意のMarkdownファイルへ適用できるようにする。

例：

```yaml
---
type: book
title:
author:
状態: 読書中
tags: []
---
```

テンプレートを適用しても、以後そのファイルがテンプレートに固定されるわけではない。貼付後は通常のYAMLとして、ファイルごとに自由に追加・削除・編集できる。

---

## 未認識フィールド

既存ファイルや外部エディタで作成されたYAMLには、Relicが入力能力を知らないキーが含まれる場合がある。

Relicは未認識フィールドを壊さず保持する。未認識フィールドはテキストとして表示でき、必要に応じて後から入力能力を割り当てられる。

---

## 検索・フィルターとの連動

登録済みフィールドは、構造的な検索・絞り込みの対象にできる。

未登録フィールドもファイル内のYAMLとして存在するため、全文検索の対象には含まれる。ただし、入力能力や型を前提にした構造的な絞り込みは、Relicが認識しているフィールドを対象とする。

---

## 一文での定義

Relicのフロントマターは、固定スキーマではなく、ユーザーが任意のMarkdownファイルへ後から自由に付与・削除できる、能力付きメタ情報フィールドである。

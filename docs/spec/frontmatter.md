# spec/frontmatter.md

プロパティ機能の詳細仕様書。
カードに付与する構造化メタデータの考え方、入力補助、表示、追加・削除の振る舞いを定義する。

---

## プロパティとは

プロパティは、カード先頭の `---` で囲まれた YAML ブロック。

Relicでは、プロパティをアプリがノートへ一律に押し付ける属性ではなく、ユーザーが任意のカードに後から付与できるメタ情報として扱う。

```yaml
---
tags: [小説, キャラクター]
状態: [下書き]
締切: [2026-05-20]
---
```

プロパティの実体は、常に各 `.md` カード先頭にあるYAMLである。Relicはこれを隠れた内部データとして扱わず、カードの一部として尊重する。

---

## 基本原則

- デフォルトのプロパティ項目は作らない
- 新規カードに自動でプロパティを付与しない
- プロパティは任意のカードへ、いつでも追加できる
- 追加したプロパティは、いつでもカード単位で削除できる
- プロパティの追加・削除で本文を壊さない
- 登録されていないYAMLキーも壊さず保持する
- 登録済みキーは検索・絞り込み・入力補助に使える
- YAMLを直接書かなくても扱えるように、Relicはフォーム表示と入力補助を提供する

本文は、あくまでユーザー自身が記述する場所である。Relicは本文領域に対して、ユーザーの明示操作なしに構造追加・自動挿入・自動整形を行わない。

---

## 表示と編集

プロパティはカードの先頭に実在する特殊ブロックとして扱う。

- ライブプレビューでは、YAMLブロックを本文より控えめなフォームとして表示・編集できる
- ソースモードでは、フォーム化せずカード内のYAMLとして直接表示・編集する
- フォーム表示では、既存フィールドの値編集、フィールド追加、フィールド削除、配列値の追加ができる
- プロパティ表示は本文の見出しや本文テキストより強く見えないようにする

入力能力、候補、固定プロパティの説明など、カードブック横断の設定は、右パネルではなくメインエリアのプロパティ設定パネルタブで扱う。
右パネルはアウトラインとリンク一覧の補助表示に限定し、プロパティ構造の管理には使わない。

---

## カードタブで行うこと

- このカードにフィールドを追加する
- このカードからフィールドを削除する
- 既存フィールドの値を編集する
- 配列値に項目を追加する
- 必要ならソースモードでYAMLとして確認・編集する

プロパティがないカードにフィールドを追加する場合、Relicはカード先頭にYAMLブロックを作成する。

```md
# 今日のメモ

本文...
```

`締切` フィールドを追加した場合：

```yaml
---
締切:
---
# 今日のメモ
```

既にプロパティがある場合は、既存YAMLブロック内にキーを追加する。同じキーが既にある場合は、既存値を勝手に上書きしない。

---

## プロパティ設定パネルタブ

プロパティ設定パネルタブは、カードブック内で使うプロパティの入力能力を管理する場所として扱う。

プロパティ設定パネルタブで行うこと：

- 固定プロパティの意味と書き方を確認する
- カスタムプロパティを追加する
- カスタムプロパティの入力能力を選ぶ
- 選択肢・複数候補の候補値を管理する
- カスタムプロパティを削除する

カード単位の値編集、フィールド追加、フィールド削除は、対象カードのカードタブ内で行う。

---

## 能力付きフィールド

プロパティの中心は、固定テンプレートではなく、能力付きフィールドである。

フィールドは任意の名前を持てる。その名前に対して、Relic側で入力能力を付与する。

例：

| フィールド名 | 入力能力 |
|--------------|----------|
| `締切` | 日付入力 |
| `状態` | 選択肢入力 |
| `登場人物` | 複数候補入力 |
| `参考URL` | URL入力 |

カスタムプロパティでは、ユーザーが任意の名前に任意の入力能力を割り当てられる。ただし、`aliases`、`tags`、`status`、`chronicle` はRelicが意味を知っている予約名として扱い、カスタムプロパティには追加しない。

カスタムプロパティもMarkdown上では通常のYAMLキーとして書く。単体値でも `[]` を使い、固定プロパティと同じ1行配列に揃える。プロパティ設定パネルタブでは、入力能力ごとに実際の書き方例を表示する。

```yaml
状態: [下書き]
巻数: [3]
締切: [2026-05-20]
公開日時: [2026-05-20T09:30]
公開時刻: [09:30]
公開済み: [true]
登場人物: [Alice, Bob]
参考URL: [https://example.com]
```

---

## 入力能力

Relicは、登録されたフィールドの入力能力に応じてフォームUIと補助機能を提供する。

| 入力能力 | UI・補助 |
|----------|----------|
| テキスト | 通常のテキスト入力 |
| 数値 | 数値入力 |
| 日付 | カレンダー入力 |
| 日時 | 日時入力 |
| 時刻 | 時刻入力 |
| 真偽値 | トグル |
| 選択肢 | 候補から選ぶ入力 |
| 複数候補 | ピル入力、候補補完 |
| URL | URL入力 |

候補を持つ入力能力では、事前に登録した候補や、カードブック内で過去に使われた値をもとに、ユーザー入力に応じて候補を絞り込む。

---

## 固定プロパティ

`aliases`、`tags`、`status`、`chronicle` はカスタム入力能力ではなく、Relicが意味を知っている固定プロパティとして扱う。

固定プロパティも、単体値でも複数値・期間でも1行の配列で書く。Relicのフォームで編集した場合も、この形で書き戻す。

```yaml
aliases: [帝都, 王都]
tags: [資料, 下書き]
status: [未着手]
chronicle: [1185, 1333]
```

`tags` に書かれた値だけをRelicのタグとして扱う。本文中の `#タグ` 記法はタグ化しない。本文中で概念・人物・資料などを参照する場合は、インラインリンク `[[...]]` を使う。

タグはその場で追加できる。ただし、表記揺れを避けるため、Relicはカードブック内の既存 `tags` から候補を出し、前後の空白や先頭の `#` はタグ値として正規化する。

`status` は、カードの状態を表す固定プロパティである。候補は `未着手`、`進行中`、`完了`、`中断`、`中止` とし、フォーム編集では1つだけを選んで1要素の配列として書き戻す。

```yaml
status: [未着手]
```

`chronicle` は、そのカード自体を年表上の単年または期間に置くための固定プロパティである。値は常に1行の整数配列で書く。

- `[年]` は単年として扱う
- `[開始年, 終了年]` は期間として扱う
- 年は整数だけを有効とする
- `0` 年は無効とする
- 紀元前は負数、紀元後は正数で表す（例: `chronicle: [-300]`, `chronicle: [-300, 250]`）
- 期間は開始年が終了年以下の場合だけ有効とする
- Chronicleタブでは、カード名を左列に固定表示し、右側の横軸年表に単年または期間を塗りとして表示する
- 単年はその年、期間は開始年から終了年までの範囲を塗り、塗りの上には年または期間を表示する
- Chronicleはグリッドで表示し、`chronicle` は1年単位で固定表示する
- Chronicleバーはクリックしてもカードを開かない
- Chronicleバー本体をドラッグすると、対応カードの `chronicle` を移動後の年または期間へ書き戻す

## 削除方法

プロパティの削除は、少なくとも以下を分けて扱う。

| 操作 | 内容 |
|------|------|
| このカードから削除 | 現在のカードのYAMLからキーを削除する |
| 入力補助の登録を削除 | Relicがそのキーに付与していた入力能力や候補を削除する |
| 全カードから削除 | 現行UIには持たない。カードブック内のカードから該当キーを一括削除する場合は別作業として扱う |

通常の削除操作は「このカードから削除」とする。入力補助の登録を削除しても、既存カードのYAML自体は勝手に消さない。

全カードから削除する操作は影響が大きいため、現行アプリの通常UIには置かない。

最後のフィールドを削除して空のプロパティだけが残る場合、Relicは空のYAMLブロックを削除する。

## プロパティテンプレート

プロパティテンプレートは、複数の能力付きフィールド名をまとめるアプリ設定上の保存済みセットである。

現行UIでは、テンプレート管理画面やテンプレート適用操作は表示しない。テンプレート定義を扱う場合も、カードブック内の `templates/` カードフォルダや `.md` テンプレートカードには依存しない。

例：

```yaml
---
type: [book]
title: []
author: []
状態: [読書中]
tags: []
---
```

テンプレート由来のキーも、Markdownに書き込まれた後は通常のYAMLとして扱う。カードごとに自由に追加・削除・編集でき、以後そのカードがテンプレートに固定されるわけではない。

---

## 未認識フィールド

既存カードや外部エディタで作成されたYAMLには、Relicが入力能力を知らないキーが含まれる場合がある。

Relicは未認識フィールドを壊さず保持する。未認識フィールドはテキストとして表示でき、必要に応じて後から入力能力を割り当てられる。

---

## 検索・フィルターとの連動

登録済みフィールドは、構造的な検索・絞り込みの対象にできる。

未登録フィールドもカード内のYAMLとして存在するため、全文検索の対象には含まれる。ただし、入力能力や型を前提にした構造的な絞り込みは、Relicが認識しているフィールドを対象とする。

---

## 一文での定義

Relicのプロパティは、固定スキーマではなく、ユーザーが任意のカードへ後から自由に付与・削除できる、能力付きメタ情報フィールドである。

# engineering/architecture.md

Relicのアーキテクチャ全体像。主要コンポーネントの構成と責務を定義する。

---

## 全体構成

Electron アプリは「メインプロセス」「preload境界」「レンダラープロセス」に分かれる。

```
┌─────────────────────────────────────────────┐
│              Renderer Process               │
│  左レール・サイドバー・タブ・右パネル             │
│  （CodeMirror 6 / 画面描画）                 │
├─────────────────────────────────────────────┤
│              Preload / contextBridge         │
│  window.relic API と型付きIPC呼び出し          │
├─────────────────────────────────────────────┤
│                  IPC 通信                   │
│  （レンダラー ↔ メイン間のメッセージング）    │
├─────────────────────────────────────────────┤
│               Main Process                  │
│  ファイル読み書き・検索/リンク/タグ生成・設定保存 │
│  （Node.js）                                │
└─────────────────────────────────────────────┘
         ↕
  Local Workspace
  Folder
```

---

## メインプロセスの責務

ファイルシステム、設定保存、検索・バックリンク・タグ・別名・年表表示用データの生成はメインプロセスが担う。レンダラーはpreloadで公開された `window.relic` APIだけを通して操作する。

| コンポーネント | 責務 |
|--------------|------|
| File Operations | Markdownファイルとフォルダの読み書き・作成・削除・移動・リネーム・複製、移動・リネーム時の内部リンク更新 |
| File Recovery | 自動保存でMarkdownを書き換える直前の本文をアプリ管理領域へ保存し、現在ファイルの復元版として読み出す |
| Workspace Service | ワークスペース登録、切り替え、一覧削除、名前変更、前回ワークスペース復元 |
| Workspace Refresh | 手動リフレッシュ時に未保存内容の保存完了後、共有索引と検索要求を無効化し、現在のワークスペースをディスクから再走査する。監視通知との直列化とワークスペース世代確認はレンダラーとIPC境界で行う |
| Search / Backlinks / Tags / Aliases / Cards / Table / Graph | ワークスペース単位で共有するファイル索引と解析キャッシュから、検索結果、バックリンク、タグ一覧、別名索引、フロントマター値候補、カード一覧、テーブル行、グラフビュー用のノードとリンクを生成 |
| Chronicle | フロントマターから年表・年表表示用データを生成し、年の範囲とチャート書き戻し用の純粋処理を扱う |
| Tool Actions | タイトル一覧、目次生成、タグ別索引、フォルダ内マージを実行し、結果を新規Markdownファイルとして保存 |
| Settings Service | アプリ設定・ワークスペース設定を安全な原子的書き込みで直列化し、形式バージョンの移行と旧設定の互換読み込みを境界内で扱う |
| IPC Handlers | レンダラーへ公開する操作単位を検証し、active workspaceとメインプロセスの処理へ接続する。Markdown、添付、一般ワークスペース項目は別の登録単位で所有する |
| Window Security | ウィンドウ生成オプション、外部遷移制限、権限要求拒否、終了前確認を扱う |

アウトゴーイングリンクとアウトラインは、開いているファイルタブの内容からレンダラー側で導出する。ワークスペース全体を読む必要があるバックリンク、検索、チャートはメインプロセスに要求する。

## Electron境界と安全条件

- メインウィンドウは `contextIsolation: true`、`nodeIntegration: false`、`sandbox: true`、`webSecurity: true`、`allowRunningInsecureContent: false` で作成する
- レンダラーはNode.js APIへ直接アクセスせず、preloadの `contextBridge` で公開された `window.relic` APIだけを使う
- レンダラー内部は `window.relic` を直接参照せず、単一のclient境界から現在のpreload APIを取得する。テスト時の差し替えもこの境界に限定する
- `window.relic` は、ファイル操作、ワークスペース操作、検索、設定、チャート、出力、ツール、クリップボード、ワークスペース変更通知、ウィンドウ終了確認応答を公開する
- IPCの型、チャンネル、`RelicApi`の部分interfaceはworkspace、files、search、settings、editor、output、tools単位で所有し、`shared/ipc.ts` は既存import向けの公開re-export境界とする
- IPC契約テストは、flatな `window.relic`、preloadのメソッドとチャンネル対応、main handler登録、入力検証要否、契約バージョンと既存チャンネル文字列の一致を確認する
- グラフビューは `getWorkspaceGraph` IPCでワークスペース全体の派生データを取得し、Canvas 2Dの描画と表示状態の管理はレンダラーで行う
- カードビューは入力なしの `getWorkspaceCards` IPCで共有索引からファイル情報、`card` 由来の任意画像パス、本文の `flavortext` を取得し、選択中に有効な画像パスがある場合だけ既存の `readImageFile` IPCで安全に読み込む
- テーブルビューは入力なしの `getWorkspaceTable` IPCで共有索引から全Markdownファイルと型付きのトップレベルプロパティ値を取得する。列選択の保存は検証済みの `saveWorkspaceTableProperties` IPCへ分離する
- Markdownの通常プレビューとライブプレビューも `readImageFile` IPCを共用し、Rendererが保持する安全な相対パスをMain側で実体パス検証してからdata URLを返す。ワークスペース更新世代ごとに画像キャッシュを分け、古い非同期応答を現在のDOMへ反映しない
- IPC handlerは、レンダラーからの入力を処理前に検証し、不正な入力はメイン処理へ渡さず失敗として返す
- Markdownリンク、Wikiリンク、本文タグ、グラフ派生データは、共有のMarkdown走査処理でコード範囲とパス正規化を判定する。機能ごとにコードブロック除外や相対パス解釈を再実装しない
- ファイル・フォルダ・ツール・検索・チャート・グラフ系IPCでは、ワークスペース相対パス、ワークスペースID、設定値、選択肢配列、出力HTML/SVGなどを用途ごとに検証する
- 検索IPCだけは互換アダプタで旧呼び出し形式を現行の検索入力へ正規化してから処理する。正規化できない検索入力はメイン処理へ渡さない
- 設定の旧schemaから現行schemaへの変換は互換アダプタへ隔離し、現行の設定parse・serializeへ旧キーの分岐を持ち込まない
- 設定ファイルの読み込みでは既存の壊れた保存値を安全側に正規化または除外する。一方、設定保存IPCでは不正な入力を丸めて保存せず、検証失敗として返す
- アクティブワークスペースが必要なIPCは、アプリ設定から現在のワークスペースを確認し、未選択の場合は処理せずエラーを返す
- OSクリップボードの読み取りは、通常のpasteイベントやエディタの貼り付け操作など、ユーザー操作に紐づく経路に限定する。Rendererから任意のタイミングで読み取れる汎用IPC APIは公開しない
- メインプロセスはアクティブワークスペースだけを再帰監視し、`rename` と `change` イベントをワークスペース変更通知としてレンダラーへ送る。監視失敗時は間隔に上限を設けて再試行し、長時間継続した場合だけ利用者へ通知する。復旧時は変更の取りこぼしに備えてワークスペース全体を再確認する
- 手動リフレッシュIPCは対象ワークスペースIDを検証し、共有派生データと検索要求をワークスペース単位で無効化してから厳格な全再走査を行う。完了前にアクティブワークスペースを再確認し、切替後の古い結果を返さない
- ワークスペース変更通知は500ms程度の静かな時間を待ってまとめ、連続変更が続く場合でも最初のイベントから約2秒以内に通知する
- 通常のメインウィンドウでは、外部ページ遷移、新規ウィンドウ生成、webviewアタッチ、権限要求を許可しない
- 外部URLを開く場合は、`https://github.com` またはそのサブドメインだけをOS標準ブラウザへ渡し、アプリ内には読み込まない
- 開発時のVite dev server読み込みだけは、`localhost`、`127.0.0.1`、`::1` の候補へ短時間リトライする。本番ではビルド済みHTMLを読み込み、ナビゲーション先はそのHTML自身とハッシュ遷移だけに限定する
- 起動スモークは、専用の起動種別と2つの絶対パスがすべて揃った場合だけ有効にする。Mainがウインドウ生成、Renderer読込、Preload API、ワークスペース状態取得IPC、初期空状態を確認してJSON reportを出力し、通常起動へ診断用IPCやRenderer APIを追加しない
- 出力用の一時ウィンドウも、Node.js統合なし、sandboxあり、権限要求拒否、webview禁止、新規ウィンドウ禁止を維持する
- PDF出力用ウィンドウではインラインスクリプトを許可しない
- アプリ終了時はレンダラーへ終了確認を送り、未保存本文や外部変更衝突があれば保存確認を通してから閉じる。開発版の終了処理中だけは、この終了前確認を通さず終了できる

### 依存境界検査の保証範囲

`pnpm architecture:check` は `app/src` のproduction `.ts`、`.tsx`、`.mts`、`.mjs`、`.js`、`.jsx` を対象にし、`test`ディレクトリ、`*.test.*`、`*.spec.*`、型定義、生成物を除外する。TypeScript構文木から静的import、re-export、import equals、文字列固定のdynamic import、文字列固定の`require`を収集する。

- 相対importは上記コード拡張子と `.json`、`.css`、各拡張子の`index`形式を解決し、未解決の場合は検査を失敗させる
- Main、Preload、Renderer、Shared間の許可方向、Renderer／SharedからのNode.js・Electron API参照、コード間の循環依存を検査する
- 文字列を組み立てるdynamic import、外部package内部の依存、CSS内の`@import`は解析しない。TypeScript型検査は構文・型・一般module解決を別に確認する
- TypeScript `paths`／`baseUrl`、Vite `resolve.alias`、package imports、workspace packageは現在採用しない。設定へ追加するとarchitecture check自体が失敗し、aliasを実パスへ解決する検査を同時に設計するまで導入できない

この検査は許可された依存方向を保証するものであり、すべての実行時依存や外部packageの安全性を保証するものではない。

## レンダラープロセスの責務

画面の描画とユーザー操作の受け付けを担う。

| コンポーネント | 責務 |
|--------------|------|
| LeftRail | プライマリサイドバー、各種パネルタブ、チャートタブを開く入口 |
| FilesSidebar | プライマリサイドバー。ファイルツリー表示、ファイル/フォルダ操作、ピン留め、ファイル検索、ワークスペース作成/登録を扱う |
| FileTab | Markdownファイルの編集単位。CodeMirrorエディタ、ライブプレビュー、ソースモード、タイプライターモード、図表プレビューを含む |
| PanelTab | 設定などの補助画面を表示するタブ |
| ChartTab | カードビュー、テーブルビュー、グラフビュー、スフィアビュー、またはクロニクルビューを表示するタブ |
| CardView | `card` 固定プロパティを持つMarkdownファイルの一覧選択と、選択中の画像・表示名を一体化した単一カードから元ファイルを開く操作を扱うチャートタブ |
| TableView | 全Markdownファイルと選択したトップレベルプロパティを読み取り専用表として表示し、固定プロパティ列から説明と候補管理を開くチャートビュー |
| GraphView | ワークスペースのMarkdown、タグ、添付画像、未解決リンクをCanvas 2Dで表示し、検索、強調、パン、ズームを扱うチャートタブ |
| SphereView | GraphViewと同じ派生グラフを共有取得層から受け取り、3次元の描画、視点、強調を扱うチャートタブ |
| RightPanel | アウトライン、バックリンク、アウトゴーイングリンク、復元版を表示する補助パネル |
| Editor | Markdown編集本体。ライブプレビュー、ソースモード、タイプライターモード、フロントマター固定プロパティ追加メニュー、ライブプレビュー表操作、Mermaid / D2 図表表示を含む |
| Chronicle | 暦設定の入力正規化、設定パネル、Canvasのscene生成、描画とpointer操作を分離する。描画loop、camera、hover、dragは専用runtimeが所有する |
| Diagram Preview | `mermaid` / `d2` コードブロックをSVGへ変換し、DOMPurifyで安全化してpan / zoom可能な表示へ接続する |
| EditorContextMenu | エディタの右クリックメニューからMarkdown書式挿入操作を提供 |
| SplitView | タブ表示領域の左右2ペイン分割表示 |
| StatusBar | 文字数・単語数の表示 |

ワークスペース全体から非同期に取得する派生データは、レンダラーの取得hookがワークスペースIDと更新番号を要求キーとして所有する。切替直後はキーが一致しない以前の内容を表示せず、以前の要求が後から完了しても現在の状態へ適用しない。グラフビューとスフィアビューは同じ取得hookとワークスペース単位の取得キャッシュを共有し、カード一覧とテーブルも表示コンポーネントではなく専用hookがこの状態遷移を管理する。

`App` は機能別hookと画面構成を接続する。コマンドパレットなどのoverlay状態と、左右ペインのスクロール先、ソース表示、エディタ参照は用途別hookが所有し、画面全体の組み立て役へ個別の一時状態を集めない。

## データの流れ（基本）

1. ユーザーがファイルを開く → レンダラーがメインにIPCで要求
2. メインがファイルを読み取り → レンダラーに内容を返す
3. レンダラーがファイルタブを開き、CodeMirror 6 に内容をセット → 編集可能になる
4. ユーザーが編集 → CodeMirror内は即時更新し、通常の連続入力を短時間まとめてタブ状態へ反映 → 入力停止後1秒でメインに自動保存要求
5. メインがファイルに書き込み → 次の検索・リンク・タグ・カード・テーブル・年表読み取りでは更新後のMarkdownを正として生成する
   - アプリ内保存では、そのワークスペースの変更ファイルだけを共有索引と解析キャッシュで更新する
   - 同じ保存に由来するファイル監視イベントは重複無効化せず、外部変更、パス不明、リネームでは安全側に必要範囲を再生成する
6. アウトラインとアウトゴーイングリンクは、アクティブなファイルタブの内容からレンダラー側で再導出する
7. 年表はレンダラーでCanvas 2Dへ描画し、表示中のパン、ズーム、物理配置、一時ドラッグを永続化しない
8. グラフビューはメインプロセスがMarkdownから派生グラフを生成し、レンダラーがCanvas 2Dへイベント駆動で描画する。ノードの物理演算は `d3-force` のWeb Workerで行い、座標はTransferable `ArrayBuffer`で受け渡して描画を再開する
9. ワークスペースを切り替えた場合、レンダラーは以前のワークスペースに属するグラフ、カード、テーブル、チャート、別名、フロントマター候補を表示対象から外し、現在の要求が完了したものだけを反映する

## ストレージ

| 種別 | 場所 | 内容 |
|------|------|------|
| Markdownファイル | ユーザーが登録した任意のローカルフォルダ内 | ユーザーが作成した `.md` ファイル |
| アプリ設定 | Electronのユーザーデータフォルダ内の `app-settings.json` | エディタ設定・機能トグル・旧入力能力定義・フロントマターテンプレート・登録ワークスペース |
| ワークスペース設定 | Electronのユーザーデータフォルダ内の `workspaces/{workspaceId}.json` | ワークスペースパス・ピン留め・チャート設定・テーブル表示列 |
| ファイル復元版 | Electronのユーザーデータフォルダ内の `file-recovery/` | 自動保存前のMarkdown本文。ワークスペースIDと相対パス単位で保存し、ワークスペース内には作成しない |
| 検索・リンク・タグ・テーブル・年表表示用データ | メモリ上または要求時生成 | Markdownファイルとフロントマターから生成される派生データ。ワークスペース内には保存しない |
| グラフ表示用データ | メモリ上または要求時生成 | Markdownのリンク、タグ、添付画像参照、未解決リンクから生成されるノードとリンク。ワークスペース内には保存しない |
| グラフ表示設定 | レンダラーのブラウザー保存領域 | フィルター、力学設定、色グループ、設定パネルの開閉状態。Markdownやワークスペース設定には保存しない |

ローカルフォルダは、iCloud Drive / OneDrive / Dropbox などのクラウド同期フォルダでもよい。Relicは各クラウドサービスのAPI連携を持たず、OS上の通常のフォルダとして読み書きする。

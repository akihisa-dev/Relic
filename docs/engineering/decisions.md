# engineering/decisions.md

アーキテクチャ上の重要な意思決定を記録するファイル。
「なぜその選択をしたか」の理由と背景を残す。設計を変更する際は必ずここを確認すること。

---

## 001 · プラットフォームは macOS を対象にする

**決定**：Apple Silicon搭載MacのmacOSだけを対象にする。他OS向けのビルド、配布、動作保証、互換処理は持たない。iOSは、デスクトップ版を自分で継続利用し、外出先・モバイルで使いたい場面が明確になったら別アプリとして検討する。

**理由**：
- Markdown ノートアプリの主要ユーザーはデスクトップユーザー
- iOS と同時開発するとデスクトップ体験が中途半端になるリスクがある
- 開発、検証、配布を実際に利用するmacOSへ集中し、OS差分の保守を持ち込まない
- まずデスクトップで自分が継続利用できる完成度まで育て、その後にモバイルで必要な体験を判断する

---

## 002 · フレームワークに Electron を採用する

**決定**：デスクトップアプリのフレームワークとして Electron を採用する。

**理由**：
- TypeScript / JavaScript の資産をそのまま使える
- React、CodeMirror、図表、可視化のWeb技術と、Node.jsによるローカルファイル操作を一つのデスクトップ構成へまとめられる
- Main、Preload、Rendererの境界で、画面からOS処理へ直接アクセスさせない構成を維持できる
- エディタエンジン（CodeMirror 6）と相性が良い

---

## 003 · エディタエンジンに CodeMirror 6 を採用する

**決定**：Markdown エディタの中核ライブラリとして CodeMirror 6 を採用する。

**理由**：
- 既存の同種アプリでも採用実績が豊富
- ライブプレビュー中心の編集体験を拡張として実装しやすい
- TypeScript ネイティブで型安全な開発が可能
- `[[...]]` などのカスタム構文も拡張システムで実装しやすい

詳細は [editor-engine.md](editor-engine.md) を参照。

---

## 005 · ワークスペースは任意のローカルフォルダを登録する

**決定**：ユーザーがOS上の任意のローカルフォルダをワークスペースとして登録する。iCloud Drive / OneDrive / Dropbox などのクラウド同期フォルダも、通常のローカルフォルダと同じように登録できる。

**理由**：
- 既存の執筆フォルダをそのまま開ける
- 一般的なMarkdown編集体験と同じ方式で、ユーザーが自分のファイルの所在を把握しやすい
- クラウド同期を使いたい場合は、ユーザーがiCloud Drive / OneDrive / Dropbox などの同期フォルダを選べばよい
- 保存場所をアプリ側で固定しないことで、ローカル保存・クラウド同期フォルダのどちらにも対応しやすい
- Relicは各クラウドサービスのAPI連携を持たず、OS上の通常のフォルダとして読み書きする

---

## 006 · Markdown 内の画像記法だけを補助表示する

**決定**：Markdown本文に書かれた画像記法のみ補助表示する。ワークスペース内の相対パス画像だけライブプレビューで実画像として表示し、外部URLやワークスペース外参照は画像プレースホルダーとして表示する。画像ファイル単体の整理・タグ付け・検索・アルバム管理は行わない。

**理由**：
- 小説の挿絵や参考画像など、テキストを補助する画像が必要になる場面はある
- 完全非対応にすると、Markdownエディタとしての実用性が不自然に狭くなる
- 一方で画像管理アプリに広げると、Relicの中心であるMarkdownファイル管理がぼやける
- 画像は本文を補助する添付物として扱い、主役は `.md` ファイルのままにする
- 実画像表示をワークスペース内の相対パス画像に限定することで、外部URL取得や画像ファイル管理の責務をアプリに持ち込まない

---

## 007 · プラグイン機能を持たない

**決定**：ユーザーが機能を追加・拡張する仕組み（プラグイン）は実装しない。将来も原則として持たない。

**理由**：
- プラグインシステムの設計・維持コストが高い
- 機能をアプリ本体に内蔵することで品質と一貫性を保つ
- 必要な機能はアプリ自体に追加していく方針
- サポート範囲を広げすぎず、Relicとして統一された体験を保つ

---

## 008 · まず自分用ビルドとして開発する

**決定**：最初の配布対象は自分用ビルドとする。第三者に配る段階でコード署名と自動アップデートを検討し、一般ユーザー向けに継続提供する価値が見えたら配布方法を判断する。

**理由**：
- まず自分が使うために完成させることを優先する
- 開発初期から第三者配布の要件を背負うと、署名・更新導線・審査対応で開発が重くなる
- 自分で継続利用して価値と不満点が見えた後に、外部配布や収益化を判断する

---

## 009 · メインエリアはタブ式で扱い、ファイルタブを編集体験の中心にする

**決定**：メインエリアはタブ式メインエリアとして扱う。ファイルビューはMarkdownファイルをファイルタブとして開き、編集体験の中心にする。テーブルビューはフロントマター一覧とプロパティ設定を統合したチャートタブ、設定はパネルタブとして開き、グラフビュー、スフィアビュー、クロニクルビューもチャートタブとして開く。ファイル加工は対象を選んだファイルツリーの右クリックメニューから実行する。

**理由**：
- ファイル編集と補助機能を同じタブ管理に載せることで、作業対象を見失いにくい
- ファイルサイドバーはファイルを探して開く場所に絞り、補助機能はパネルタブとして扱うほうが役割が明確になる
- エディタはファイルタブとして常に戻れる対象にし、設定などの補助ビューで作業文脈を上書きしない

---

## 010 · UIフレームワークに React を採用する

**決定**：UIフレームワークとして React を採用する。

**理由**：
- AI駆動開発（Claude Code / Codex）との相性が良い
- 同様のユースケースでの採用実績がある
- TypeScriptとの相性が良く、大規模なUIでも管理しやすい

---

## 011 · パッケージマネージャに pnpm を採用する

**決定**：パッケージマネージャとして pnpm を採用する。

**理由**：
- インストールが高速でディスク使用量が少ない
- Electron × React 構成との相性が良く、現在のエコシステムで広く使われている
- npm と比べても運用上のデメリットがほぼない

---

## 012 · ビルド・配布ツールに Electron Forge を採用する

**決定**：Electronアプリのビルド・配布ツールとして Electron Forge を採用する。

**理由**：
- Electron 公式が推奨するツールであり、現在のスタンダード
- プロジェクト作成・ビルド・配布を一括で扱えるため設定が簡潔
- pnpm との組み合わせも公式にサポートされている

---

## 013 · テストフレームワークに Vitest を採用する

**決定**：テストフレームワークとして Vitest を採用する。

**理由**：
- 設定が簡潔で、TypeScript プロジェクトとの相性が非常に良い
- Jest に比べて実行が高速
- Vite ベースの現代的なツールチェーンと親和性が高い
- E2E テストが必要になった場合は Playwright を追加すればよい

## 015 · 料金モデルは現時点では設計対象外にする

**決定**：料金モデルは現時点では設計対象外にする。まず自分用に作って使い、良ければ収益化を検討する。

**理由**：
- 無料固定でも有料前提でもなく、まず自分用の道具として成立させる段階である
- 料金モデルを早く入れると、機能トグルや仕様判断が収益化前提に引っ張られる

## 016 · React状態管理にZustandを限定採用する

**決定**：React状態管理にはZustandを採用する。ただし、タブ・ペイン・サイドバー・右パネル・アクティブワークスペース・UI設定反映などの横断的UI状態に限定する。フォーム入力・メニュー開閉・モーダル内の一時状態はReactローカルstateで扱い、ファイル内容・検索結果・リンク/タグ生成結果・設定の永続化はZustandに入れない。

**理由**：
- Relicはタブ・分割表示・サイドバー・右パネル・ワークスペース復元など、横断的UI状態を扱う
- React標準だけで始めるより、予測できる複雑さに先に置き場所を用意できる
- 一方で、永続データやファイル内容までstoreに入れると責務が曖昧になるため、用途を限定する

---

## 017 · 設定は userData 配下のJSONに保存する

**決定**：アプリ設定とワークスペース設定はElectronの `userData` 配下にJSONで保存し、`app/src/main/settings/` の自前設定サービスで読み書きする。

**理由**：
- 設定構造はRelic固有であり、自前サービスのほうが仕様との対応関係を保ちやすい
- ユーザーのMarkdownワークスペース内にRelic専用設定ファイルを置かず、ノートフォルダをきれいに保てる

---

## 018 · Relic用の必須フォルダをワークスペース準備時に作らない

**決定**：ワークスペース作成時または既存フォルダ登録時に、`attachments/` や `templates/` などのRelic専用フォルダを自動作成しない。

**理由**：
- ユーザーの既存フォルダ構造をアプリ都合で変更しない
- Relic専用の隠れた前提を減らし、macOSの通常フォルダとして扱いやすくする
- 必要になった機能は、その操作時にユーザーが理解できる形で扱う

---

## 019 · グラフビューはCanvas 2Dで描画する

**決定**：グラフビューは、ワークスペースのMarkdown、タグ、添付画像、未解決リンクから生成した派生データのうち、ファイルと未解決リンクを固定条件でCanvas 2Dへ描画する。有効な単一の `category` を持つファイルは、カテゴリ名から決まるテーマ対応色の変形可能なバブルへまとめる。バブル位置は固定値にせず所属ノードの重心から再生成し、接触面は互いに押された輪郭として分離し、所属ノードはその輪郭内へ拘束する。バブルのドラッグは所属ノードの一時固定と一括移動として扱い、位置を保存しない。未設定のファイルはバブルへ所属させない。グラフのノード配置とバブルは表示中だけ使い、Markdown本文、ワークスペース設定、ブラウザー保存領域へ書き戻さない。

**理由**：
- グラフの表示結果をMarkdown本文やRelic固有の保存形式へ混ぜず、Markdownを正本として保てる
- Canvas 2Dでノード、リンク、ラベルをまとめて描画し、パン・ズーム・強調表示をレンダラー内で扱える
- `category` をMarkdown内の見える分類として利用し、グラフ専用の手動グループ設定を持たずに配置を再生成できる
- 表示状態とワークスペースデータを分離することで、ワークスペースを切り替えてもユーザーファイルの内容を変更しない
- グラフデータの生成と物理演算を分離し、表示層の変更範囲を限定できる

---

## 020 · 図表はMarkdownコードブロックを正本にして表示する

**決定**：Mermaid / D2 図表は、Markdown本文中の `mermaid` / `d2` コードブロックを正本として扱う。Relicは保存時に独自の図表モデルや座標メタデータを書き足さず、表示時だけSVGを生成する。

**理由**：
- MarkdownファイルをRelic外のエディタでも読める状態に保てる
- 図表の編集履歴や差分が、通常のテキスト差分として扱える
- MermaidとD2のどちらもコードブロック文化と相性がよく、Relic独自形式を作る必要がない
- SVG生成結果やpan / zoom状態を保存しないことで、Markdown本文と表示状態の責務を分けられる
- D2のブラウザ版はWebAssembly workerを使うため、現行CSPでは `worker-src blob:` と `script-src 'wasm-unsafe-eval'` を許可する。文字列から任意コード実行につながりやすい `'unsafe-eval'` は許可しない

## 023 · Coworkは現行アプリに含めない

**決定**：Coworkは現行アプリに含めない。将来あらためて必要性が明確になった場合も、現在の設計判断を前提にせず、その時点の目的と範囲から検討し直す。

**理由**：
- Relicの中核は、Markdownに書ける情報をMarkdownファイルのまま保ち、編集・閲覧・検索・可視化・出力を広げるローカルアプリである
- 現行アプリでは、ローカルMarkdownファイルを正本にした編集・閲覧・検索・可視化・出力に判断範囲を絞る
- 存在しない機能のUI、保存方式、接続方式をこの文書で決めると、現行仕様として誤読される

## 024 · グラフビューの物理演算をWeb Workerとd3-forceへ分離する

**決定**：グラフビューの物理演算はReactコンポーネント内で毎フレーム計算せず、Web Worker上のd3-forceシミュレーションへ分離する。描画は現行のCanvas 2Dを維持し、座標更新はTransferable `ArrayBuffer`で受け取る。`SharedArrayBuffer`はcross-origin isolation要件を満たす設計確認ができるまで使わない。描画エンジン導入は今回の決定に含めず、描画層を差し替えやすい構造に留める。

**理由**：
- ノード数が増えたときに、物理演算でUI応答性を落とさないようにする
- 成熟した力学レイアウトを使い、手書きの全ノード間反発計算を避ける
- Workerからrefへ座標を反映し、React state更新を毎フレーム発火させない
- Canvas 2D描画を維持して変更範囲を限定しつつ、将来の描画層差し替えに備える

---

## 025 · 3次元表示は独立した正式なスフィアビューとして提供する

**決定**：Canvas 2Dグラフビューと同じワークスペースグラフの固定表示対象をWebGLで3次元表示するスフィアビューを、独立した正式なチャートタブとして提供する。2次元だけが `category` バブルを描画し、スフィアビューは既定でオフの機能トグルに置き、描画用データ、視点、物理演算、描画ループ、GPU資源の所有と破棄をグラフビューから分離する。

**理由**：
- 2次元のグラフビューをそのまま維持し、3次元表示による性能や互換性の影響を隔離する
- 同じノードとリンクの意味を保ちながら、表示方式と操作方式だけを比較できる
- 3次元表示のコードと共有グラフは機能が有効な場合だけ待機時間に先読みし、通常起動の初期読込を増やさない
- グラフの接続構造、テーマ色、強調状態を分離して更新し、テーマ変更や再表示で不要な配置計算を繰り返さない
- 配置完了後は低頻度描画へ移り、短時間の再表示では停止中のruntimeを1個だけ再利用することで、待機中と連続操作時のCPU・GPU負荷を抑える
- WebGLを利用できない環境や描画状態の消失が起きても、2次元グラフを継続利用できる
- 独立したタブにすることで、同時表示、分割表示、個別の破棄を既存のタブ規則で扱える

# dev/testing.md

このファイルはテスト方針を定義するドキュメントです。
AIはテストコードを書く前にこのファイルを参照し、対象フェーズと仕様上のリスクに合わせてテスト範囲を決めます。

---

## 基本方針

- テストフレームワークは Vitest を使う
- React UIテストには React Testing Library を使う
- まずユニットテストとメインプロセス寄りの統合テストを厚くする
- ファイル破壊・リンク更新・Git操作など、失敗時の被害が大きい処理を優先してテストする
- React UIは、仕様上の分岐や状態遷移が複雑な箇所からテストする
- E2Eテストは初期必須にしない。必要になった段階で Playwright の追加を検討する
- テストは実ユーザーのワークスペースや実GitHubリポジトリを使わない

---

## テストの種類

| 種類 | 対象 | 方針 |
|------|------|------|
| ユニットテスト | 純粋関数・パーサー・パス処理・設定変換・Markdown解析 | 最優先で書く |
| 統合テスト | ファイル操作・インデックス更新・Git操作の薄い結合 | 一時ディレクトリを使って実ファイルに近い形で確認する |
| UIテスト | Reactコンポーネント・状態遷移 | 重要な画面状態と操作結果を確認する |
| E2Eテスト | アプリ全体 | 初期段階では対象外。必要になったら追加する |

## フェーズ別のテスト基準

### フェーズ0：プロジェクト基盤

- Vitestが実行できる
- sharedの型・純粋関数をテストできる
- IPCの最小APIについて、入力・出力の型が共有されている
- `app/` で `pnpm test` が通る

### フェーズ1：ワークスペースとファイル管理

- ワークスペース登録・切り替え・前回復元
- ワークスペース準備時に `attachments/` と `templates/` が作成されること
- `.md` 拡張子の自動付与
- 表示上の `.md` 拡張子非表示
- 禁止文字・重複名のエラー
- 作成・削除・リネーム・移動・複製
- 削除がゴミ箱移動として扱われること
- フォルダ削除前に確認が必要であること
- ワークスペース外パスの拒否
- 非Markdownファイルをサイドバーの主要対象にしないこと

### フェーズ2：エディタ基本体験

- 自動保存の1秒遅延
- 手動保存フローが不要であること
- 文字数・単語数
- 表示設定の反映
- ツールバーボタンのMarkdown挿入
- タブの表示履歴復帰
- タブ0枚時の新規ノート作成画面
- 分割表示中のペイン別タブ管理

### フェーズ3：Markdownライブプレビュー

- ライブプレビューのみの編集画面
- カーソルまたは選択範囲が触れている装飾範囲のみ記法表示
- 記法表示中も太字・ハイライト・見出しなどのレンダリングが維持されること
- CommonMark標準記法の表示
- Obsidian互換拡張記法の表示
- チェックボックスのクリック切り替え
- コードブロックのシンタックスハイライト
- KaTeX数式表示
- 添付画像表示
- ファイル埋め込みが一段階までで止まること
- 大きすぎる埋め込みファイルが全文表示されないこと
- 外部URL画像を初期対象外として扱うこと
- タイプライターモードのオン/オフ

### フェーズ4：リンク・タグ・検索

- `[[ファイル名]]` と `[[フォルダ/ファイル名]]` の解析
- エイリアスリンク・見出しリンク・ブロック参照の解析
- ブロック参照IDの自動生成・挿入
- 未作成リンクからの新規ファイル作成
- リネーム・移動時のリンク自動更新
- フォルダのリネーム・移動時に配下ファイルへのリンクが更新されること
- 本文とフロントマターの更新対象判定
- コードブロック内を更新しない挙動
- バックリンク・アウトゴーイングリンク生成
- フロントマター `tags:` のタグ集計
- 階層タグと日本語タグ
- 全文検索・ファイル名検索・タグ検索・正規表現検索
- 無効な正規表現のエラー
- 一括置換プレビューと実行結果

### フェーズ5：フロントマターとコマンド操作

- YAMLの読み書き
- フォームUIとソースYAMLの同期
- 任意のフィールド名に入力能力を割り当てられること
- 能力付きフィールドの型別入力
- フロントマターの後付け追加・ファイル単位削除
- 既存値を勝手に上書きしない挙動
- 未認識フィールドを壊さず保持する挙動
- フロントマター検索・フィルター
- コマンドパレットの候補絞り込み・実行・Escape終了
- クイックスイッチャーの候補絞り込み・タブアクティブ化・アクティブペインへの新規タブ追加

### フェーズ6：Git / GitHub

- ローカルGit初期化
- コミット作成
- Diff取得
- コミット履歴表示用データ生成
- ブランチ一覧・作成・切り替え
- 未コミット変更がある場合の3択分岐
- コミット作成時に作者名・メールをUIから渡さず、GitHub接続情報由来のauthorをMain側で補うこと
- GitHub未接続でもMarkdown編集とローカルGit状態確認が使えること
- 自動同期がデフォルトOFFであること
- 自動同期の間隔プリセット
- 自動プッシュ前に保存完了を待つこと
- 自動コミットメッセージ形式
- コンフリクト状態の検出と解決選択
- Gitタグの作成・一覧・プッシュ・削除
- GitHubエラーを原因別に返すこと

### フェーズ7：ファイル加工ツールと仕上げ

- 条件指定マージ
- 見出しによるファイル分割
- タイトル一覧生成
- 目次生成
- 既存ファイルを変更しないこと
- 同名出力時の別名保存
- ファイル名に使えない文字の置換
- 機能トグルによる表示・操作のオン/オフ
- テーマ切り替え
- macOS向けビルド前の回帰確認

## 優先してテストする領域

- ワークスペース外パスの拒否
- ユーザーファイルを書き換える処理
- リネーム・移動時の内部リンク自動更新
- コードブロック内を壊さない判定
- フロントマターの保持と同期
- 検索・置換・インデックス更新
- Git操作と同期前確認
- 自動保存と保存待ち処理
- 機能トグルで非表示・無効化される操作

---

## テストファイルの置き場所

- 原則として対象ファイルの近くに `*.test.ts` を置く
- 複数モジュールをまたぐ統合テストは `app/src/test/` 配下に置く
- テスト用のMarkdownサンプルは、読みやすい小さな文字列としてテスト内に書く
- 大きなfixtureが必要になった場合だけ `fixtures/` を作る
- ユーザーの実ワークスペースに依存するfixtureを作らない

---

## 命名規則

- テストファイル名は `対象ファイル名.test.ts`
- `describe` は対象モジュール名
- `it` は期待する振る舞いを自然文で書く
- 日本語仕様に密接なテストは、日本語の説明名を使ってよい

例：

```ts
describe("parseWikiLinks", () => {
  it("パス付き内部リンクを解析できる", () => {
    // ...
  });
});
```

---

## 一時ファイルと安全性

- テストは実ユーザーのワークスペースを使わない
- ファイル操作テストは一時ディレクトリを作って実行する
- テスト終了後に一時ディレクトリを削除する
- 破壊的操作のテストでは、対象パスが一時ディレクトリ内であることを確認する
- ゴミ箱移動のテストは、実OSのゴミ箱に依存しすぎないよう、境界を薄くして検証する
- GitHub連携テストでは実GitHubへ接続せず、認証・HTTP・リモート操作の境界を差し替える

## 実行

プロジェクト作成後、以下のコマンドを基本にする。

```sh
cd app
pnpm test
```

必要に応じて、監視実行やカバレッジ取得のコマンドを追加する。

---

## テスト追加の判断基準

- 仕様分岐がある処理を追加したらテストを書く
- ファイルを書き換える処理を追加したらテストを書く
- パーサー・検索・リンク更新の変更には回帰テストを書く
- IPCの入力検証を追加・変更したらテストを書く
- UI状態遷移を追加したら、重要な分岐をテストする
- UIだけの軽微な見た目調整は、既存テストが通る確認を基本にする
- バグを直したら、同じバグが再発しないテストを追加する

# dev/testing.md

このカードはテスト方針を定義するドキュメントです。
AIはテストコードを書く前にこのカードを参照し、対象フェーズと仕様上のリスクに合わせてテスト範囲を決めます。

---

## 基本方針

- テストフレームワークは Vitest を使う
- React UIテストには React Testing Library を使う
- まずユニットテストとメインプロセス寄りの統合テストを厚くする
- カード破壊・リンク更新など、失敗時の被害が大きい処理を優先してテストする
- React UIは、仕様上の分岐や状態遷移が複雑な箇所からテストする
- E2Eテストは通常の開発では必須にしない。必要になった段階で Playwright の追加を検討する
- テストは実ユーザーのカードブックや外部サービスを使わない

---

## テストの種類

| 種類 | 対象 | 方針 |
|------|------|------|
| ユニットテスト | 純粋関数・パーサー・パス処理・設定変換・Markdown解析 | 最優先で書く |
| 統合テスト | カード操作・インデックス更新の薄い結合 | 一時ディレクトリを使って実カードに近い形で確認する |
| UIテスト | Reactコンポーネント・状態遷移 | 重要な画面状態と操作結果を確認する |
| E2Eテスト | アプリ全体 | 通常は対象外。必要になったら追加する |

## フェーズ別のテスト基準

### P0：プロダクト意図・対象ユーザー整理

- テストコードの対象外
- `docs/product/principles.md` と `docs/product/PLAN.md` で、対象ユーザーとアプリの存在理由が説明されていること

### P1：機能収集・仕様化

- テストコードの対象外
- `docs/spec/` に主要機能の仕様が分かれて記録されていること

### P2：技術選定・設計判断

- テストコードの対象外
- `docs/tech/` と `docs/architecture/decisions.md` に、技術選定と主要な設計判断が記録されていること

### P3：実装前ドキュメント整備

- テストコードの対象外
- `docs/dev/conventions.md`、`docs/dev/testing.md`、`docs/dev/open-questions.md` が整備されていること

### P4：基礎実装

#### プロジェクト基盤

- Vitestが実行できる
- sharedの型・純粋関数をテストできる
- IPCの最小APIについて、入力・出力の型が共有されている
- `app/` で `pnpm test` が通る

#### カードブックとカード管理

- カードブック登録・切り替え・前回復元
- カードブック準備時にRelic専用の必須カードフォルダを自動作成しないこと
- `.md` 拡張子の自動付与
- 表示上の `.md` 拡張子非表示
- 禁止文字・重複名のエラー
- 作成・削除・リネーム・移動・複製
- 削除がゴミ箱移動として扱われること
- カードフォルダ削除前に確認が必要であること
- カードブック外パスの拒否
- 非カードをサイドバーの主要対象にしないこと

#### エディタ基本体験

- 自動保存の1秒遅延
- 手動保存フローが不要であること
- 文字数・単語数
- 表示設定の反映
- ツールバーボタンのMarkdown挿入
- タブの表示履歴復帰
- タブ0枚時の新規ノート作成画面
- 分割表示中のペイン別タブ管理

#### Markdownライブプレビュー

- ライブプレビューのみの編集画面
- カーソルまたは選択範囲が触れている装飾範囲のみ記法表示
- 記法表示中も太字・ハイライト・見出しなどのレンダリングが維持されること
- CommonMark標準記法の表示
- Obsidian互換拡張記法の表示
- チェックボックスのクリック切り替え
- コードブロックのシンタックスハイライト
- KaTeX数式表示
- Markdown画像記法が実画像ではなくプレースホルダーとして表示されること
- カード埋め込みが一段階までで止まること
- 大きすぎる埋め込みカードが全文表示されないこと
- 外部URL画像をフェッチせず、画像プレースホルダーとして扱うこと
- タイプライターモードのオン/オフ

#### リンク・タグ・検索

- `[[カード名]]` と `[[カードフォルダ/カード名]]` の解析
- エイリアスリンク・見出しリンク・ブロック参照の解析
- ブロック参照IDの自動生成・挿入
- 未作成リンクからの新規カード作成
- リネーム・移動時のリンク自動更新
- カードフォルダのリネーム・移動時に配下カードへのリンクが更新されること
- 本文とプロパティの更新対象判定
- コードブロック内を更新しない挙動
- バックリンク・アウトゴーイングリンク生成
- プロパティ `tags:` のタグ集計
- 階層タグと日本語タグ
- 全文検索・カード名検索・タグ検索・正規表現検索
- 無効な正規表現のエラー
- 一括置換プレビューと実行結果

### P5：プロパティとコマンド操作

- YAMLの読み書き
- フォームUIとソースYAMLの同期
- 任意のフィールド名に入力能力を割り当てられること
- 能力付きフィールドの型別入力
- プロパティの後付け追加・カード単位削除
- 既存値を勝手に上書きしない挙動
- 未認識フィールドを壊さず保持する挙動
- プロパティ検索・フィルター
- コマンドパレットの候補絞り込み・実行・Escape終了
- クイックスイッチャーの候補絞り込み・タブアクティブ化・アクティブペインへの新規タブ追加

### P7：カード加工ツールと仕上げ

- 条件指定マージ
- 見出しによるカード分割
- タイトル一覧生成
- 目次生成
- 既存カードを変更しないこと
- 同名出力時の別名保存
- カード名に使えない文字の置換
- 機能トグルによる表示・操作のオン/オフ
- テーマ切り替え
- macOS向けビルド前の回帰確認

### P8以降：確認・安定化・文書整理

- 機能確認や不具合修正では、対象機能の仕様リスクに応じて型チェック、既存テスト、対象テストを選ぶ
- 大型機能追加では、追加した機能の主要分岐と失敗時の影響が大きい処理を優先してテストする
- 文書整理の各単位では、コードテストではなく対象文書、現在フェーズ正本、参照先文書の整合を確認し、`git diff --check` を最低確認にする
- P23のような大規模な文書整合整理では、最後に `pnpm typecheck`、`pnpm test`、`git diff --check` を確認する
- 文書内の古い前提や参照切れは `rg` で確認し、必要な場合だけ対象文書の範囲で修正する

## 優先してテストする領域

- カードブック外パスの拒否
- ユーザーカードを書き換える処理
- リネーム・移動時の内部リンク自動更新
- コードブロック内を壊さない判定
- プロパティの保持と同期
- 検索・置換・インデックス更新
- 自動保存と保存待ち処理
- 機能トグルで非表示・無効化される操作

---

## テストファイルの置き場所

- 原則として対象カードの近くに `*.test.ts` を置く
- 複数モジュールをまたぐ統合テストは `app/src/test/` 配下に置く
- テスト用のMarkdownサンプルは、読みやすい小さな文字列としてテスト内に書く
- 大きなfixtureが必要になった場合だけ `fixtures/` を作る
- ユーザーの実カードブックに依存するfixtureを作らない

---

## 命名規則

- テストファイル名は `対象ファイル名.test.ts`
- `describe` は対象モジュール名
- `it` は期待する振る舞いを自然文で書く
- 日本語仕様に密接なテストは、日本語の説明名を使ってよい

例：

```ts
describe("parseWikiLinks", () => {
  it("パス付き内部リンクを解析できる", () => {
    // ...
  });
});
```

---

## 一時カードと安全性

- テストは実ユーザーのカードブックを使わない
- カード操作テストは一時ディレクトリを作って実行する
- テスト終了後に一時ディレクトリを削除する
- 破壊的操作のテストでは、対象パスが一時ディレクトリ内であることを確認する
- ゴミ箱移動のテストは、実OSのゴミ箱に依存しすぎないよう、境界を薄くして検証する

## 実行

アプリ本体の型チェックとテストは `app/` 配下で実行する。

```sh
cd app
pnpm typecheck
pnpm test
```

まとめて確認する場合は、`app/package.json` の `verify` を使う。

```sh
cd app
pnpm verify
```

Markdown差分の空白・改行確認はリポジトリルートで実行する。

```sh
git diff --check
```

必要に応じて、対象テスト、監視実行、カバレッジ取得のコマンドを追加する。

---

## テスト追加の判断基準

- 仕様分岐がある処理を追加したらテストを書く
- カードを書き換える処理を追加したらテストを書く
- パーサー・検索・リンク更新の変更には回帰テストを書く
- IPCの入力検証を追加・変更したらテストを書く
- UI状態遷移を追加したら、重要な分岐をテストする
- UIだけの軽微な見た目調整は、既存テストが通る確認を基本にする
- バグを直したら、同じバグが再発しないテストを追加する

# doc-code-alignment.md

コードと仕様書・設計書の間にある不一致を一覧化した調査レポート。
P11 フェーズで各項目をユーザーが判断し、コードまたは文書のどちらかに統一する。

> **このファイルの読み方**
> 各項目に「わかりやすく言うと」欄を設けた。エンジニア向けの詳細が不要なときはそこだけ読めばよい。
> 推奨は AI が示すもの。最終決定はユーザーが行う。

---

## 重要度の凡例

| 記号 | 意味 |
|------|------|
| 🔴 | 使っているときに気づく。ユーザー体験に直接影響する |
| 🟡 | 動作はするが、文書と現実がずれているため次の開発で混乱しやすい |
| 🟢 | 外から見えない内部の設計ズレ。今すぐ困ることは少ない |

## 🔴 ① エディタの「ライブプレビュー」モードが仕様と異なる

### わかりやすく言うと

仕様書には「Obsidian のように、編集画面の中で文字を打ちながら同時にMarkdownが装飾される（太字や見出しがリアルタイムで見た目に変わる）」モードをデフォルトにすると書いてある。

しかし実際のアプリは「**編集画面（地の文が見える）**」と「**プレビュー画面（HTMLで綺麗に表示）**」の2つのボタンで切り替える仕組みになっている。Obsidian 式のリアルタイム装飾は **コードとして書かれてはいるが、現在オンにされていない**。

### 技術的な詳細

- **仕様書（`spec/editor.md`）**：デフォルトは「ライブプレビュー」（CodeMirror 内でリアルタイムデコレーション）。「ソースモード」に切り替えると記法がそのまま見える。
- **実装（`app/src/renderer/store/editorStore.ts`・`PaneView.tsx`）**：`ViewMode = "source" | "preview"` という2択。`"preview"` は別コンポーネント `Preview`（静的HTMLレンダリング）を表示する。`"source"` は CodeMirror を表示するが、ライブプレビュー効果（`livePreview` prop）は `PaneView.tsx` から渡されていないため常に無効。
- **宙に浮いているコード**：`Editor.tsx` 内の `buildLivePreviewDecorations` 関数は完成しているが、どこからも有効化されていない。

### 推奨：コードを正とし、仕様書を書き直す

現行の「Preview / Source の2画面切り替え」を正式仕様にする。Obsidian 式のリアルタイム装飾は将来オプションとして残す（すでにコードがあるので ON にするのは数行の変更）。

**選択肢：**

| 選択 | 内容 |
|------|------|
| A（推奨） | 現行の Preview/Source 方式を正式仕様とし、spec を書き直す |
| B | Source モードで `livePreview={true}` を有効にして、Obsidian 式を復活させる |

## 🔴 ② フォーカスモードが未実装なのに「完了」と記録されている

### わかりやすく言うと

「フォーカスモード」とは、今書いている段落だけを明るく見せて、他の部分を薄暗くする集中支援モード。仕様書と確認チェックリストには「完了済み ✔」と書いてあるが、アプリの中に**その機能は存在しない**。

### 技術的な詳細

- **仕様書（`spec/editor.md`）と `feature-checklist.md`**：フォーカスモードが `[x]`（完了）として記録されている。
- **実装（`app/src/renderer/store/uiStore.ts`）**：`isTypewriterMode` のみ存在。`isFocusMode` に相当するステートがない。
- **`App.tsx`**：コマンドパレットにフォーカスモードのコマンドが存在しない。
- **`Editor.tsx`**：フォーカスモード用のデコレーションロジックがない。

### 推奨：実装するか、仕様から外すか、判断が必要

**選択肢：**

| 選択 | 内容 |
|------|------|
| A | P11 で実装する（段落ごとに opacity を変える CSS + ステート追加） |
| B | 仕様から外し、feature-checklist のチェックを外す |

## 🔴 ③ フロントマターの「ユーザー定義フィールド」設定画面が未実装

### わかりやすく言うと

フロントマターとは各ノートの先頭に書く整理用データ（日付・状態・著者など）のこと。仕様書には「設定画面でオリジナルのフィールドを自分で追加・削除できる（最大13個）」と書いてある。

しかし実際の設定画面にはその機能がなく、YAMLを直接書いた「自由フィールド」が表示されるだけ。

### 技術的な詳細

- **仕様書（`spec/frontmatter.md`）**：「ユーザーが設定画面でフィールドを追加・編集・削除できる。型（text/number/date/boolean/select 等）を設定できる。最大13個」
- **実装（`app/src/renderer/components/SettingsSidebar.tsx`）**：ユーザー定義フィールド管理 UI が存在しない。
- **実装（`app/src/renderer/components/FrontmatterForm.tsx`）**：YAML に手書きされた自由フィールドをテキスト入力として表示するだけ。

### 推奨：コードを正とし、仕様を縮小する

現行の「自由フィールドはYAML直接編集で対応」を正式仕様とする。設定画面からの管理は将来機能として格下げ。

**選択肢：**

| 選択 | 内容 |
|------|------|
| A（推奨） | 仕様から「設定画面での管理」を削除し、自由フィールド（YAMLを直接書く）として記述する |
| B | P11 以降に設定画面の実装を加える |

## 🟡 ④ 自動同期の設定がワークスペースごとではなくアプリ全体で共通

### わかりやすく言うと

「自動同期（Auto Push/Pull）」の設定を、仕様書では「ワークスペースごとに別々に持てる」と書いている。しかし実際は**アプリ全体で1つの設定**を共有している。

### 技術的な詳細

- **仕様書（`architecture/data-model.md`）**：「ワークスペース設定（`workspaces/{workspaceId}.json`）に autoSync: ワークスペース固有の自動同期設定」と記述。
- **実装（`app/src/main/settings/appSettings.ts`）**：`AppSettings` に `autoSync` が含まれ、`app-settings.json`（グローバル）に保存される。

### 推奨：コードを正とし、データモデル文書を更新

ワークスペースごとに異なる同期設定が必要かどうかを確認した上で、不要であれば文書を修正する。

**選択肢：**

| 選択 | 内容 |
|------|------|
| A（推奨） | 仕様書を「アプリ全体で共通」に修正する |
| B | ワークスペースごとに保存する仕組みに変更する（実装変更を伴う） |

## 🟡 ⑤ ワークスペース切り替えUIがドロップダウンではなくボタンリスト

### わかりやすく言うと

仕様書には「現在のワークスペース名をクリックすると下にリストが展開するドロップダウン形式」と書いてある。しかし実際は、登録されているワークスペースがサイドバーに常に並んで表示されるボタンリスト形式になっている。

### 技術的な詳細

- **仕様書（`spec/file-management.md`）**：「サイドバー上部のドロップダウン」「現在のワークスペース名 ▼ をクリックするとリストが展開する」
- **実装（`app/src/renderer/components/FilesSidebar.tsx`）**：ワークスペースが複数登録されている場合に、ボタンリストとして常時表示する。

### 推奨：コードを正とし、仕様書を更新

機能的に問題ないためコードを正とする。ドロップダウン化はUIの改善として将来フェーズで検討できる。

## 🟡 ⑥ ファイル作成・リネーム・複製の操作方法が仕様書と実装で食い違う

### わかりやすく言うと

仕様書には「ファイルを右クリックするとメニューが出て、そこから複製・リネームできる」と書いてある。しかし実際はファイルツリーに右クリックメニューがなく、コマンドパレット（`⌘⇧P`）または専用の入力バーから操作する。

### 技術的な詳細

- **仕様書（`spec/file-management.md`）**：複製は「右クリックメニューから」、リネームは「ダブルクリックまたは右クリックメニューから」、作成は「作成直後はインライン編集状態になる」。
- **実装**：
  - 作成：`FilesSidebar.tsx` に常時表示の入力フォーム（インライン編集ではない）
  - リネーム：`RenameBar`（`App.tsx`）というインライン入力バー。コマンドパレットからも起動
  - 複製：コマンドパレットから実行
  - ファイルツリーに右クリックメニューはない
- **`feature-checklist.md`** はコードと一致した記述になっている（こちらが正）

### 推奨：コードおよび `feature-checklist.md` を正とし、`spec/file-management.md` を修正

## 🟢 ⑦ TemplateManager コンポーネントが設計書に書かれているが存在しない

### わかりやすく言うと

設計書に「テンプレートを管理するモジュール」が書かれているが、実際にそういう名前のモジュールは存在しない。テンプレートフォルダを自動的に作る処理だけが存在する。また「新規ノートを作るときにテンプレートを選べる」機能も現時点では未実装。

### 技術的な詳細

- **設計書（`architecture/overview.md`）**：「TemplateManager：テンプレートファイルの管理と新規ファイル作成時の適用」
- **実装**：独立した TemplateManager モジュールは存在しない。`workspaceService.ts` の `prepareWorkspace()` が `templates/` フォルダを作るだけ。新規ファイル作成時のテンプレート選択 UI もない。

### 推奨：設計書を修正し、テンプレート選択機能は将来仕様に格下げ

---

### 推奨：仕様書から削除するか、将来仕様に格下げ

**選択肢：**

| 選択 | 内容 |
|------|------|
| A（推奨） | 仕様書から「自動生成UI」の記述を削除または「将来検討」に変更 |
| B | P11 以降で実装する |

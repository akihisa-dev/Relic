# doc-code-alignment.md

コードと仕様書・設計書の間にある不一致を一覧化した調査レポート。
P11 フェーズで各項目をユーザーが判断し、仕様としてよりよい形に統一する。

> **P11 の前提**
> 仕様書・設計書は先に合意した意図を表す正本として扱う。
> コードと文書が食い違う場合、まず「実装が仕様に追いついていない可能性」を疑う。
> そのうえで、現在のコードのほうが使いやすい、シンプル、または安全だと判断できる場合だけ、文書側を更新する。
> 差異の解消は「コードに合わせて文書を削る」ことではなく、「比較してよりよい方を残す」ことを目的とする。

> **このファイルの読み方**
> 各項目に「わかりやすく言うと」欄を設けた。エンジニア向けの詳細が不要なときはそこだけ読めばよい。
> 推奨は AI が示すもの。最終決定はユーザーが行う。

---

## 重要度の凡例

| 記号 | 意味 |
|------|------|
| 🔴 | 使っているときに気づく。ユーザー体験に直接影響する |
| 🟡 | 動作はするが、文書と現実がずれているため次の開発で混乱しやすい |
| 🟢 | 外から見えない内部の設計ズレ。今すぐ困ることは少ない |

## 🔴 ① エディタの「ライブプレビュー」モードが仕様と異なる

### わかりやすく言うと

仕様書には「Obsidian のように、編集画面の中で文字を打ちながら同時にMarkdownが装飾される（太字や見出しがリアルタイムで見た目に変わる）」モードをデフォルトにすると書いてある。

以前の実装は「**編集画面（地の文が見える）**」と「**プレビュー画面（HTMLで綺麗に表示）**」の2つのボタンで切り替える仕組みだった。判断の結果、2モード切り替えは煩わしいため廃止し、ライブプレビューだけを正式な編集画面にする。

### 技術的な詳細

- **仕様書（`spec/editor.md`）**：ライブプレビューを基本の編集画面とする。
- **実装方針**：`PaneView.tsx` から Preview / Source の切り替え UI を削除し、常に `Editor` を `livePreview` 有効で表示する。
- **タブ状態**：`editorStore.ts` の `viewMode` は不要になるため削除する。

### 決定：ライブプレビューのみ

2モード切り替えは廃止する。書く画面そのものをライブプレビューにし、ユーザーが Preview / Source を選ばなくてよい形に統一する。

## 🔴 ② フォーカスモードが未実装なのに「完了」と記録されている

### わかりやすく言うと

「フォーカスモード」とは、今書いている段落だけを明るく見せて、他の部分を薄暗くする集中支援モード。仕様書と確認チェックリストには「完了済み ✔」と書いてあるが、アプリの中に**その機能は存在しない**。
実際の使用感としても、周囲を薄くするより見づらさが勝つため、Relic には入れない。
代わりに、必要な改善は「カーソルがどこにあるか分かりやすくする」方向で扱う。

### 技術的な詳細

- **過去の仕様書（`spec/editor.md`）と `feature-checklist.md`**：フォーカスモードが `[x]`（完了）として記録されていた。
- **実装（`app/src/renderer/store/uiStore.ts`）**：`isTypewriterMode` のみ存在。`isFocusMode` に相当するステートがない。
- **`App.tsx`**：コマンドパレットにフォーカスモードのコマンドが存在しない。
- **`Editor.tsx`**：フォーカスモード用のデコレーションロジックがない。

### 決定：仕様から外す

フォーカスモードは実装しない。薄くする集中表示は見づらくなるため、仕様・コマンド・機能トグル対象から外す。
カーソル位置の分かりやすさは別のエディタ表示改善として扱う。

## 🔴 ③ フロントマターの「ユーザー定義フィールド」設定画面が未実装

### わかりやすく言うと

フロントマターとは各ノートの先頭に書く整理用データ（日付・状態・著者など）のこと。仕様書には「設定画面でオリジナルのフィールドを自分で追加・削除できる（最大13個）」と書いてある。

以前の実装では設定画面にその機能がなく、YAMLを直接書いた「自由フィールド」が表示されるだけだった。
判断の結果、コードを書かない人でも使えるように、設定画面からユーザー定義フィールドを管理できる仕様を残す。

### 技術的な詳細

- **仕様書（`spec/frontmatter.md`）**：「ユーザーが設定画面でフィールドを追加・編集・削除できる。型（text/number/date/boolean/select 等）を設定できる。最大13個」
- **実装方針（`SettingsSidebar.tsx`）**：設定画面でユーザー定義フィールドの追加・編集・削除、型、選択肢を管理する。
- **実装方針（`FrontmatterForm.tsx`）**：設定されたフィールドをフロントマターフォームに表示し、型に応じた入力UIで編集する。

### 決定：設定画面で管理できる仕様を実装する

ユーザー定義フィールドは仕様どおり実装する。設定画面で追加・編集・削除でき、フロントマターフォーム側にも反映する。

## 🟡 ④ 自動同期の設定がワークスペースごとではなくアプリ全体で共通

### わかりやすく言うと

「自動同期（Auto Push/Pull）」の設定を、仕様書では「ワークスペースごとに別々に持てる」と書いている。以前の実装は**アプリ全体で1つの設定**を共有していた。
判断の結果、仕事用フォルダ・日記フォルダなどで同期の事情が変わるため、ワークスペースごとに別々に保存する。

### 技術的な詳細

- **仕様書（`architecture/data-model.md`）**：「ワークスペース設定（`workspaces/{workspaceId}.json`）に autoSync: ワークスペース固有の自動同期設定」と記述。
- **実装方針**：`autoSync` は `workspaceSettings.ts` 側に移し、アクティブワークスペースの設定として読み書きする。
- **タイマー**：自動同期タイマーもアクティブワークスペースの `autoSync` を読む。

### 決定：ワークスペースごとに保存する

自動同期設定は仕様どおり、ワークスペースごとに保存する。

## 🟡 ⑤ ワークスペース切り替えUIがドロップダウンではなくボタンリスト

### わかりやすく言うと

仕様書には「現在のワークスペース名をクリックすると下にリストが展開するドロップダウン形式」と書いてある。以前の実装は、登録されているワークスペースがサイドバーに常に並んで表示されるボタンリスト形式だった。
判断の結果、サイドバーを静かに保つため、仕様どおりドロップダウン形式にする。

### 技術的な詳細

- **仕様書（`spec/file-management.md`）**：「サイドバー上部のドロップダウン」「現在のワークスペース名 ▼ をクリックするとリストが展開する」
- **実装方針（`app/src/renderer/components/FilesSidebar.tsx`）**：現在のワークスペース名をボタンとして表示し、クリック時だけ登録済みワークスペースのリストを開く。

### 決定：ドロップダウン式にする

サイドバーを静かに保つため、ワークスペース切り替え UI は仕様どおりドロップダウン式にする。

## 🟡 ⑥ ファイル作成・リネーム・複製の操作方法が仕様書と実装で食い違う

### わかりやすく言うと

仕様書には「ファイルを右クリックするとメニューが出て、そこから複製・リネームできる」と書いてある。以前の実装はファイルツリーに右クリックメニューがなく、コマンドパレット（`⌘⇧P`）または専用の入力バーから操作する形だった。
判断の結果、ファイルツリー上の対象を直接操作できるほうが分かりやすいため、右クリックメニューとインライン編集を実装する。

### 技術的な詳細

- **仕様書（`spec/file-management.md`）**：複製は「右クリックメニューから」、リネームは「ダブルクリックまたは右クリックメニューから」とする。
- **実装方針（`FileTree.tsx`）**：ファイル・フォルダ行に右クリックメニューを追加し、リネームはその場で入力欄に切り替える。
- **実装方針（`App.tsx`）**：既存の rename / duplicate / trash API を、ファイルツリーからも呼び出せるようにする。
- **作成操作**：サイドバーの入力フォームを維持する。作成直後のインライン編集までは今回の対象にしない。

### 決定：右クリック・インライン編集を実装する

ファイルツリーの対象を直接右クリックして、名前変更・複製・ゴミ箱移動を行えるようにする。
名前変更は、右クリックメニューまたはダブルクリックで、その場のインライン入力に切り替える。

## 🟢 ⑦ TemplateManager コンポーネントが設計書に書かれているが存在しない

### わかりやすく言うと

設計書に「テンプレートを管理するモジュール」が書かれているが、以前はテンプレートフォルダを自動的に作る処理だけが存在し、「新規ノートを作るときにテンプレートを選べる」機能は未実装だった。
判断の結果、`templates/` 内のMarkdownを新規ノート作成時に選べるようにする。

### 技術的な詳細

- **設計書（`architecture/overview.md`）**：「TemplateManager：テンプレートファイルの管理と新規ファイル作成時の適用」
- **実装方針（`templates.ts`）**：`templates/` 直下の `.md` ファイルをテンプレート一覧として返し、選択されたテンプレート本文を新規ファイルに適用する。
- **実装方針（`FilesSidebar.tsx`）**：テンプレートがある場合だけ、新規ノート作成フォームにテンプレート選択を表示する。

### 決定：テンプレート管理を実装する

`templates/` フォルダ内のMarkdownテンプレートを一覧化し、新規ノート作成時に選べるようにする。

## 🟢 ⑧ ブロック参照IDの自動生成UIが未実装

### わかりやすく言うと

ブロック参照とは、特定の段落に固有のIDをつけて、他のノートからそこだけを参照できる機能。仕様書には「ボタンを押すと自動でIDを生成して挿入できる」と書いてある。
判断の結果、ツールバーのブロックID挿入ボタンで、カーソル位置または選択範囲の段落末尾へIDを挿入できるようにする。

### 技術的な詳細

- **仕様書（`spec/links-and-tags.md`）**：「選択中の段落またはカーソル位置の段落に対してブロック参照IDを自動生成して挿入できるUIを持つ」
- **実装方針（`Toolbar.tsx`）**：ツールバーの `^ID` ボタンで段落末尾にIDを挿入する。選択範囲が複数段落にまたがる場合は各段落に挿入し、既存IDがある段落には重複挿入しない。
- **実装済み（`app/src/main/files/linkUpdater.ts`）**：ブロック参照（`^blockId`）の parse は実装済み。

### 決定：自動生成UIを実装する

ブロック参照IDは手入力だけでなく、ツールバーから自動生成して挿入できるようにする。

## 🟢 ⑨ マージツールの絞り込み条件に「フロントマターの値」がない

### わかりやすく言うと

「条件指定マージ」ツールの仕様書には「フロントマターのフィールド値でファイルを絞り込める」と書いてある。以前の実装では「すべて・フォルダ指定・タグ指定」の3択のみ対応していた。
判断の結果、フロントマターを整理に使う流れと相性がよいため、仕様どおり実装する。

### 技術的な詳細

- **仕様書（`spec/file-tools.md`）**：「絞り込み条件：タグ・フロントマターのフィールド値・フォルダ」
- **実装方針（`app/src/shared/ipc.ts`）**：`MergeFilterType` に `frontmatter` を追加し、`frontmatterField` と `filterValue` を渡す。
- **実装方針（`app/src/main/ipc/toolHandlers.ts`）**：Markdown本文を文字列検索するのではなく、YAMLフロントマターを構造的に解析して絞り込む。

### 決定：フロントマター値絞り込みを実装する

条件指定マージで、例 `status: draft` のようにフロントマターのフィールド名と値を指定して対象ファイルを絞り込めるようにする。

## 🟢 ⑩ featureToggles に「フォーカス系モード」が含まれるという記述が誤り

### わかりやすく言うと

アプリの設定データの説明に「機能オン/オフの設定にはフォーカスモードのトグルも含まれる」と書いてあるが、実際には含まれていない。
②でフォーカスモードは仕様から外すと決めたため、フォーカス系モードの機能トグルも不要。

### 技術的な詳細

- **設計書（`architecture/data-model.md`）**：「featureToggles: GitHub連携・ファイル加工ツール・フロントマター・右パネル・**フォーカス系モードのオン/オフ**」
- **実装（`app/src/shared/ipc.ts`）**：`FeatureToggles = { git, tools, frontmatter, rightPanel }` の4つのみ

### 決定：4トグルにする

機能トグルは GitHub連携・ファイル加工ツール・フロントマター・右パネルの4つにする。
フォーカスモードもタイプライターモードも、機能トグルの対象にはしない。

## 判断が必要な項目 まとめ

以下の表を使って、各項目を「仕様どおり実装」「現行コードを正式仕様化」「仕様から外す」のいずれかで決定する。

| # | タイトル | 重要度 | 主な選択肢 | 決定 |
|---|---------|--------|------|------|
| ① | ライブプレビューモードの定義 | 🔴 | ライブプレビュー実装 / 2画面切り替えを正式仕様化 | **ライブプレビューのみ**（2モード切り替えは廃止） |
| ② | フォーカスモード | 🔴 | 実装 / 仕様から外す | **仕様から外す**（薄くする集中表示よりカーソル視認性を優先） |
| ③ | フロントマターのユーザー定義フィールド | 🔴 | 設定画面で管理を実装 / YAML直接編集に縮小 | **設定画面で管理を実装** |
| ④ | 自動同期の保存場所 | 🟡 | ワークスペース別に実装 / アプリ全体共通を正式仕様化 | **ワークスペース別に実装** |
| ⑤ | ワークスペース切り替えUI | 🟡 | ドロップダウン実装 / ボタンリストを正式仕様化 | **ドロップダウン実装** |
| ⑥ | ファイル操作の入力方法 | 🟡 | 右クリック・インライン編集を実装 / コマンド中心を正式仕様化 | **右クリック・インライン編集を実装** |
| ⑦ | TemplateManager | 🟢 | テンプレート管理を実装 / 現行構成に設計書を合わせる | **テンプレート管理を実装** |
| ⑧ | ブロック参照ID自動生成 | 🟢 | 自動生成UIを実装 / 手入力のみを正式仕様化 | **自動生成UIを実装** |
| ⑨ | マージ絞り込み条件 | 🟢 | フロントマター値絞り込みを実装 / 3条件に縮小 | **フロントマター値絞り込みを実装** |
| ⑩ | featureToggles の記述誤り | 🟢 | フォーカス系トグルを実装 / 4トグルに設計書を合わせる | **4トグルにする**（フォーカス系トグルなし） |

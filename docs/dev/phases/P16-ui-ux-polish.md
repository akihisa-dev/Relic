# P16-ui-ux-polish.md

Relicの本当のUI/UX洗練フェーズの正本。

このフェーズでは、AIが先に方針・チェックリスト・実装内容を作らない。
ユーザーの「こうしたい」「こう見える」「ここが嫌だ」という感覚を出発点にし、必ずヒアリングしてから方針を決める。

---

## フェーズの目的

Relicを、書く道具として気持ちよく、分かりやすく、使い続けたくなるUIへ整える。

特に、以下を重視する。

- ユーザー本人の違和感を最優先する
- 見た目の派手さではなく、日常的に書くための落ち着きと清潔感を重視する
- AIが勝手に「つまらなさ」の原因や解決策を決めない
- デザイン原則は `../ui/DESIGN.md` を正とし、変更が必要な場合もユーザーの明示判断を待つ

---

## UI/UX判断と実施

このフェーズでは、方向性と実施内容を必ずペアで記録する。
方向性だけ、実施だけを離して書くと、次回以降の判断材料として価値が落ちるため。

### 色と印象

方向性:

- 現状はオレンジを使いすぎている
- 「オレンジは行動に対して使う」という定義は広すぎる。UI上の多くは行動なので、結果的にオレンジを使う理由が増えすぎる
- オレンジは通常操作ではなく、最終決定・不可逆操作・危険・強い注意など、ユーザーに一呼吸置かせたい場面へ絞る
- ブルーが十分に使えていない。ブルーは情報、案内、リンク、現在の対象、補助的な状態表示に使う
- モノクロは主軸だが、黒い印象にしたいわけではない
- 黒もモノクロではあるが、ライトテーマでは白をメインにしたクリーンな印象にしたい
- シンプルなデザインにしたい。色・装飾・説明・強調を増やす方向ではなく、白い余白、薄いグレーの区切り、必要最小限のアクセントで整える

実施:

- 未実施。次回以降、この方向性に沿って画面単位で調整する

確認:

- 未実施

残り:

- 具体的にどの画面から色・余白・アクセントを調整するかは、ユーザーの違和感を起点に決める

### フロントマターUI

方向性:

- フロントマターは右パネルの専用編集欄ではなく、MD本文先頭のYAMLをObsidian式のプロパティフォームとして見せる
- フロントマターの実体はMDファイル内の構造を保ち、UIは入力補助として日付・真偽値・候補入力などを支援する
- 右パネルは本文外の補助情報に寄せ、フロントマター編集の主導線から外す
- 本文そのものの構造は保護し、アプリ独自の隠しメタデータにはしない

実施:

- 右パネルのフロントマター専用編集UIを撤去した
- 右パネルはアウトラインとリンクの補助情報に戻した
- MDファイル先頭のYAMLフロントマターを、エディタ本文上ではプロパティフォームとして表示するようにした
- プロパティフォームは折りたたみ可能にした
- プロパティフォームからプロパティを追加・削除できるようにした
- 登録済みフィールドの入力能力をエディタ内フォームに反映した
- `date` は日付入力、`boolean` はチェックボックス、`select` / `multi-select` / `tags` は候補付き入力として扱うようにした
- ワークスペース内Markdownから既存フロントマター値候補を収集し、検索とプロパティ入力の候補に使えるようにした
- フロントマターテンプレートは作成後に編集できるようにした
- フィールド名変更時はテンプレート側も追従し、フィールド削除時はテンプレートからも除外するようにした
- フロントマターがないMarkdownファイルでも、本文先頭のプロパティ入力からYAMLブロックを新規作成できるようにした
- array / tags 系の値を、プロパティフォーム上で個別に編集・削除できるようにした
- プロパティ編集時に、トップレベルのYAMLコメント行・空行・フィールド順をできるだけ保持するようにした
- 単純なスカラー値の同一行にあるインラインコメントを、プロパティ編集後も保持するようにした
- 単純な文字列スカラーのクォートを、プロパティ編集後も保持するようにした
- 1行配列として書かれたYAML値は、array / tags 系の編集後も1行配列として保持するようにした
- ネストしたオブジェクトなどの複雑なYAML値は読み取り表示にし、フォーム編集で壊さないようにした
- 複雑なYAML値は、フォーム上のYAML入力で直接編集できるようにした
- 不正なYAML入力は本文へ反映せず、入力欄をエラー状態にするようにした

確認:

- `pnpm exec tsc --noEmit` 通過
- `pnpm test` 通過（31ファイル / 254テスト）
- 開発版Electronアプリ（`pnpm start`）で、既存フロントマターのプロパティフォーム・配列pill・追加行が本文先頭に表示され、右パネルへ戻っていないことを目視確認
- 開発版Electronアプリで、フロントマターなしファイルのプロパティ追加からYAMLブロックが新規作成されることを実操作確認
- 開発版Electronアプリで、複雑YAML値をフォーム上のYAML入力から編集でき、ファイルへ保存されることを実操作確認
- 開発版Electronアプリで、不正なYAML入力がファイルへ反映されないことを実操作確認

残り:

- なし

### 動作とリアクション

方向性:

- すべての動作は、結果だけがパッと現れるのではなく、操作に見合った短い過程を持つ
- モーションは装飾ではなく、押す・引く・閉じる・開く・選ぶ・移動する・なぞるなどの操作に対する手触りとして扱う
- ユーザーがしたアクションに対して、アプリはその動作から想像されるリアクションを返す
- ボタンは、ユーザーが押したなら、視覚上も押されなければならない
- 反応は大げさにせず、書く道具の落ち着きを壊さない範囲で、ユーザーが「いま何が起きたか」を身体感覚で追えるようにする
- 閉じる動作は、突然消すのではなく、少し押し込まれてから縮む・抜ける・吸い込まれるような過程を持たせる
- 選択・切り替えは、重心や現在地が移ったことが分かる反応にする
- 開閉・展開は、面や情報が押し出される・折りたたまれる感覚にする
- ドラッグ・移動は、掴んでいる対象と移動先の受け入れ状態が分かる反応にする

実施:

- 動作の洗い出しを開始した
- 最初の実装対象をタブを閉じる動作にした
- タブのバツボタンで閉じるとき、タブが即時に消えず、短く押し込まれて横幅と透明度が縮んでから閉じるようにした
- タブのコンテキストメニューから閉じる、他を閉じる、右側を閉じる、すべて閉じる場合も、対象タブが縮んでから閉じるようにした
- `⌘W` で閉じる場合も、同じタブ閉じモーションを通るようにした
- タブを開くとき、タブが列へ短く入ってくる動きを付けた
- タブを選ぶとき、現在タブが軽く受け止める動きを付けた
- ボタン全体に、押したとき少し沈む基本リアクションを付けた
- サイドバーを開閉するとき、内側の内容が左へ畳まれる / 左から戻る反応を付けた
- 右パネルを開閉するとき、内側の内容が右へ畳まれる / 右から戻る反応を付けた
- ファイルツリーのフォルダ開閉で、矢印が回転し、子項目が短く展開される反応を付けた
- ファイルツリーの選択で、対象行が軽く受け止める反応を付けた
- ファイルツリーのドラッグ移動先で、受け入れ先が少し膨らむ反応を付けた
- コマンドパレットとクイックスイッチャーが、呼び出し時に面として浮き上がる反応を付けた
- コマンドパレットとクイックスイッチャーの選択行が、選ばれた対象として軽く受け止める反応を付けた
- トーストが表示されるとき、下から短く出てくる反応を付けた
- 分割表示を開始するとき、右ペインが右から差し込まれる反応を付けた
- 右パネルのアウトライン / リンク切り替えで、内容が右側から短く入れ替わる反応を付けた
- 右パネル内のアウトライン項目とリンク項目に、選ぶ対象として軽く動く反応を付けた
- サイドバーのビュー切り替えで、内容が左側から短く入れ替わる反応を付けた
- ツールバーのドロップダウンとインライン入力ダイアログが、押したボタンから開く小さな面として出る反応を付けた
- ツールバーのドロップダウン項目に、選ぶ対象として軽く動く反応を付けた
- ファイル名変更、移動先入力、ファイルツリーのリネーム入力が、ラベルから入力欄へ変わる反応を付けた
- ツールバー操作が本文へ反映されたとき、対象エディタ面が短く受ける反応を付けた
- エディタ内のプロパティフォームで、フォーム全体と各行が短く入ってくる反応を付けた
- プロパティフォームの折りたたみで、矢印が回転し、本文が畳まれる反応を付けた
- プロパティ入力欄、追加欄、配列pill入力欄が、フォーカス時に軽く浮く反応を付けた
- プロパティ削除ボタンとpill削除ボタンが、押したときに沈む反応を付けた
- 複雑YAML入力の不正値で、入力欄が短く揺れてエラー色になる反応を付けた
- 検索結果・Git関連リスト・リンク一覧が、結果として短く入ってくる反応を付けた
- 検索結果・Git関連リスト・リンク一覧の項目が、選ぶ対象として軽く受け止める反応を付けた
- モーションを抑える環境設定では、タブ・ボタン・ファイルツリー・モーダル・トースト・パネル切り替え・ツールバー・入力変形・プロパティフォーム・結果リスト系の遷移時間を最小化するようにした

## 作業ルール

- このフェーズは大きな計画を先に固めるのではなく、ユーザーが気になった箇所を都度、少しずつ直していく。
- ユーザー判断で実装したUI/UX変更は、この文書の「UI/UX判断と実施」に方向性・実施・確認・残りをセットで記録する。日誌だけに閉じ込めない。
- 小さなUI変更ごとに、対象・意図・確認方法を短く扱う。
- ユーザーが「やりたい」「こうしたい」と言っただけで、仕様・フェーズ・チェックリスト・デザイン方針を作らない。
- AIが勝手に作った方針やチェックリストをユーザーが確認すること自体が無駄になるため、確認させる前に作らない。
- まずユーザーへヒアリングし、何が嫌か、何を変えたいか、どの程度変えてよいかを確認する。
- ヒアリングの結果、ユーザーが「作ってよい」と明示した場合だけ、方針・チェックリスト・文書・実装案を作る。
- UI/UX方針は、提案として出すことはできる。ただし正本化・実装はユーザーが明示的に許可した範囲だけ行う。
- デザイン原則の変更は `../ui/DESIGN.md` に関わるため、ユーザーが明示した場合だけ編集する。
- 実装済みの変更は戻さない。戻す場合はユーザーが明示的に指示したときだけ行う。

## 最初に確認すること

このフェーズでは、大きな変更に入る前だけ次を1問ずつ確認する。
小さな違和感修正は、ユーザーが具体的に対象と方向を示した場合、その範囲で都度直す。

- いま一番嫌な画面・状態はどこか
- どんな印象にしたいか
- 変えてよい範囲はどこまでか
- 先にデザイン原則を直すか、画面単位で直すか
- 実アプリ目視確認を行うか、コード上の確認に留めるか

# P15-ui-ux-polish.md

RelicのUI/UXを、書く道具として使いたくなる体験へ洗練するフェーズの正本。

P14でGitHub連携の安全導入はいったん凍結した。P15ではGitHub機能の追加実装へ進まず、現状のアプリが「なんかつまらないものになっている」問題を解消する。

---

## 優先度

1. 書く気持ちが起きること
2. 情報構造が分かりやすいこと
3. 操作が迷いなく軽いこと
4. Relicらしい個性があること
5. 既存機能を壊さないこと

---

## 最初に読む

- `../ui/screens-macos.md`
- `../ui/navigation.md`
- `../ui/DESIGN.md`

必要になったときだけ読む:

- 挙動の正解を確認する: `../spec/`

- 過去の経緯を確認する: `../journal/` の該当日

---

## 基本方針

- GitHub機能の追加実装は凍結する。
- 既存のGitHub関連UIは、必要なら見た目や導線の整理だけ行う。
- 新機能を増やす前に、現在ある画面の密度、階層、余白、言葉、操作の流れを整える。
- ランディングページ風の派手さではなく、書く・探す・整理する日常操作が気持ちよくなる方向へ寄せる。
- 「説明が多いUI」ではなく、見れば分かる構造にする。
- 既存の仕様を大きく変える場合は、実装前にユーザーへ確認する。

---

## 確認する観点

- [ ] 第一印象が「Markdownエディタ」ではなく「書く場所」として成立している
- [ ] サイドバーの情報量と優先順位が整理されている
- [ ] ファイル一覧、エディタ、プレビュー、右パネルの関係が分かりやすい
- [ ] Git / 検索 / 設定などの補助機能が主役を邪魔していない
- [ ] 空状態が無機質すぎず、次の行動が自然に分かる
- [ ] ボタン、入力欄、リスト、選択状態の見た目が揃っている
- [ ] 日本語UIとして文言が硬すぎず、冗長でもない
- [ ] ライト / ダークテーマの両方で読みやすい
- [ ] 画面幅が狭い状態でも破綻しない

---

## 導入順序

### 1. 現状診断

- [ ] 現在の主要画面をコード上で確認する
- [ ] UI上の「つまらなさ」の原因を、情報設計・視覚設計・文言・操作感に分けて整理する
- [ ] すぐ直せる小さな違和感と、設計判断が必要な変更を分ける

### 2. 体験の芯を整える

- [ ] 初期表示と空状態を整える
- [ ] 書く領域の集中感を上げる
- [ ] サイドバーの主従関係を整理する
- [ ] 主要操作の見た目と配置を揃える

### 3. 補助機能の整理

- [ ] Gitパネルを主張しすぎない見た目にする
- [ ] 検索、リンク、タグ、ツール、設定の表示密度を整える
- [ ] 操作後のフィードバックを分かりやすくする

### 4. 仕上げ確認

- [ ] `pnpm exec tsc --noEmit`
- [ ] `pnpm test`
- [ ] 必要に応じて開発版アプリで主要画面を確認する

- [ ] 主要画面のUI階層と視覚的な一貫性が改善されている
- [ ] 既存機能の型チェック・自動テストが通っている

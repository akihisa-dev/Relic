# P18-ui-ux-design-fixes.md

RelicのUX/UIデザイン修正フェーズの正本。

このフェーズでは、AIが先に全体方針・チェックリスト・修正案一覧を作らない。
ユーザーの「ここが気になる」「こう見える」「この画面を直したい」という具体的な違和感を出発点にし、対象を確認してから必要最小限の修正を行う。

---

## フェーズの目的

Relicを、書く道具として見やすく、迷わず、日常的に使いやすいUIへ整える。

特に、以下を重視する。

- ユーザー本人が気になっている画面・状態を最優先する
- 見た目の修正に機能追加・導線追加・仕様変更を混ぜない
- 変更範囲を、ユーザーが指定した画面・場所・操作対象に限定する
- デザイン原則は `../../ui/DESIGN.md` を正とし、変更が必要な場合もユーザーの明示判断を待つ

---

## 作業ルール

- このフェーズは大きな計画を先に固めず、ユーザーが気になった箇所を都度、少しずつ直す。
- 小さなUI変更ごとに、対象・意図・確認方法を短く扱う。
- ユーザーが「やりたい」「こうしたい」と言っただけで、仕様・チェックリスト・デザイン方針を作らない。
- まずユーザーへ、何が嫌か、何を変えたいか、どこまで変えてよいかを確認する。
- UI/UX方針は提案として出せる。ただし正本化・実装はユーザーが明示的に許可した範囲だけ行う。
- デザイン原則の変更は `../../ui/DESIGN.md` に関わるため、ユーザーが明示した場合だけ編集する。
- 実装済みの変更は戻さない。戻す場合はユーザーが明示的に指示したときだけ行う。

---

## 最初に確認すること

このフェーズでは、大きな変更に入る前だけ次を1問ずつ確認する。
小さな違和感修正は、ユーザーが具体的に対象と方向を示した場合、その範囲で都度直す。

- いま一番気になる画面・状態はどこか
- 何が見づらい、分かりにくい、気持ち悪いと感じるか
- 変えてよい範囲はどこまでか
- 実アプリ目視確認を行うか、コード上の確認に留めるか

---

## 完了条件

- ユーザーが指定したUX/UI修正対象に対して、変更範囲が合意されている
- 実装した場合は型チェック・自動テストが通っている
- 実アプリ目視確認が必要な項目は、確認済みまたは未確認理由が明記されている
- 実装したUX/UI判断が、この文書に実施したことだけのチェックリストとして記録されている

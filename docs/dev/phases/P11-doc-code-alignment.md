# P11-doc-code-alignment.md

Relicの文書・コード整合化フェーズの正本。

このフェーズでは、実装済みの挙動と仕様文書の差分を確認し、現行コードを正としてよい箇所は文書側を更新した。

---

## 読み込みルール

文書とコードの不一致を解消するフェーズ。

AIはこのフェーズを前提にユーザーへ接する。
最初にこの文書の「判断が必要な項目まとめ」を提示し、
1項目ずつユーザーに決定を求めること。決定後、合意した方向でコードまたは文書を修正する。

最初に読む:

- この文書

必要になったときだけ読む:

- 修正対象のコード: `../../../app/src/` 配下の該当ファイル
- 修正対象の仕様書: `../../spec/` 配下の該当ファイル
- 修正対象の設計書: `../../architecture/` 配下の該当ファイル
- 過去の判断経緯: `../../journal/` の該当日

作業の進め方:

1. この文書の「判断が必要な項目 まとめ」表を開く
2. 🔴 の項目から順に、ユーザーに「コード正・実装・省略」を1問ずつ確認する
3. 決定したらこの文書の「決定」欄を埋める
4. 全項目の決定が終わったら、合意した変更をコードまたは文書に反映する
5. 変更後、`P9-feature-checklist.md` と関連する spec ファイルを更新する

注意:

- 1セッションで全項目を解決しようとしない。決定→修正→確認のサイクルを短くする
- 「コード正」の場合は文書だけを変更する。「実装」の場合はコード変更も伴う
- 実装を伴う変更は、変更前に影響範囲をユーザーに説明してから着手する

---

## フェーズの目的

- コードと仕様文書の不一致を減らす
- 現行アプリのほうが使いやすい、シンプル、安全だと判断できる箇所は文書を合わせる
- 実装変更が必要な箇所と、文書更新だけでよい箇所を切り分ける

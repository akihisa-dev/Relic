# P19-refactoring-organization.md

Relicのリファクタリングおよび整理フェーズの正本。

このフェーズでは、既存挙動を保ったまま、コード・テスト・文書・構成を読みやすく、直しやすく、壊れにくい状態へ整える。

---

## フェーズの目的

- P18までに増えたUI/UX実装を、保守しやすい単位へ整理する
- 重複、肥大化、責務の混在、テストしにくい構造を減らす
- 仕様変更や新機能追加ではなく、既存機能の維持と安全な整理を優先する
- 次の開発フェーズに入る前に、変更しやすい足場を作る

---

## 作業方針

- 1回の変更範囲を小さく区切る
- 変更前に、対象・目的・影響範囲・確認方法を短く明示する
- UI文言、操作導線、保存形式、IPC API、既存データ形式を不用意に変えない
- 挙動変更が必要になった場合は、リファクタリングとして進めず、理由を説明してユーザー確認を取る
- 変更後は `pnpm exec tsc --noEmit` と、対象に応じたテストを確認する

---

## 実装チェックリスト

P19では、アプリの現状を正として、既存挙動の確認・Git機能削除・Mac/Windows両対応の安定化・仕様書整合をこの順序で進める。

### 1. 現状把握

目的:

- 現在のアプリ挙動を確認し、残す機能・削除するGit機能・仕様書との差分を大まかに分ける

確認:

- [ ] 現在の主要画面を洗い出す
- [ ] 現在の主要操作を洗い出す
- [ ] 残す機能を洗い出す
- [ ] 削除対象のGit / GitHub機能を洗い出す
- [ ] 仕様書とアプリ現状の大きな差分を洗い出す

### 2. Git機能削除

目的:

- RelicからGit / GitHub連携機能を削除し、書く道具としての安定性を優先する

確認:

- [ ] Git / GitHub関連の画面・ボタン・文言を洗い出す
- [ ] Git / GitHub関連のIPC APIを洗い出す
- [ ] Git / GitHub関連のメインプロセス実装を洗い出す
- [ ] Git / GitHub関連のフロントエンド実装を洗い出す
- [ ] Git / GitHub関連の設定・保存データを洗い出す
- [ ] Git / GitHub関連の依存関係を洗い出す
- [ ] 削除してよいもの、互換性のため残すものを分ける
- [ ] 削除後にアプリ起動、ファイル操作、保存が壊れていないことを確認する

### 3. Mac / Windows両対応の安定性確保

目的:

- OS差で壊れやすい箇所を確認し、Mac / Windowsの両方で安定して動く状態に近づける

確認:

- [ ] パス区切り文字の扱いを確認する
- [ ] 改行コードの扱いを確認する
- [ ] ファイル名・フォルダ名の扱いを確認する
- [ ] ファイル監視の挙動を確認する
- [ ] ElectronのOS別分岐を確認する
- [ ] macOS前提のUI文言・説明・処理を確認する
- [ ] Windowsで問題になりそうな保存・読み込み・起動処理を確認する

### 4. 全件検証

目的:

- Git機能削除とOS安定化後の状態で、現在のアプリに残す機能を全件確認する

確認:

- [ ] 自動テストで確認できる範囲を確認する
- [ ] 型チェックで確認する
- [ ] ビルドで確認する
- [ ] 実アプリ操作が必要な項目を分けて確認する
- [ ] 不具合、未確認、仕様差分を記録する

### 5. アプリ現状に仕様書を合わせる

目的:

- 仕様書を理想や過去方針ではなく、現在のアプリ挙動に合わせる

確認:

- [ ] 現在のアプリ挙動を確認する
- [ ] 仕様書とアプリの差分を洗い出す
- [ ] アプリに存在しない機能の記述を整理する
- [ ] Git機能削除後の状態に仕様書を合わせる
- [ ] UI文書と現在画面の差分を整理する
- [ ] 「今後やりたいこと」と「現在の仕様」を混ぜない

# dev/roadmap.md

このファイルは開発フェーズとロードマップを記録するドキュメントです。
実装再開時に、どの順序で、どの仕様を参照し、どの状態になれば完了とみなすかを定義します。

---

## 方針

- **実際のアプリ実装は、2026-05-04 にユーザー指示で凍結解除済み。** 以後はフェーズ0から順番に進める
- 実装再開後は、フェーズ0から順番に進める。後続フェーズのためだけの先回り実装はしない
- 各フェーズは、ユーザーが動作を確認できる単位で完了させる
- 仕様の詳細は `docs/spec/`、画面構成は `docs/ui/`、技術判断は `docs/tech/` と `docs/architecture/` を正とする
- 技術選定が必要になった場合は、選択肢・影響・推薦を整理し、ユーザー確認後に `docs/dev/open-questions.md` を解決済みにしてから実装する
- フェーズ完了時は、該当するテストを追加し、必要なドキュメントと `docs/journal/YYYY-MM-DD.md` を更新する

---

## 実装前に解決すること

フェーズ0へ入る前に、`docs/dev/open-questions.md` の未解決P0をすべて解消する。

特に、実装基盤・設定保存・認証情報保存・Markdownレンダリング・テスト補助ライブラリなど、プロジェクト雛形や依存関係に影響する問いは、実装開始前に確定する。

---

## 共通完了条件

すべてのフェーズで、以下を満たすこと。

- 対象仕様に書かれた挙動が、推定ではなく該当ドキュメントに基づいて実装されている
- ワークスペース外のファイルを書き換えない
- ユーザーのMarkdown本文にRelic専用の不可視メタデータを埋め込まない
- 失敗時にユーザーが次の行動を判断できるエラーを返す
- 追加・変更した処理に対応するテストがある
- `app/` で `pnpm test` が通る

---

## フェーズ0：プロジェクト基盤

**ゴール**：Electron + React + TypeScript + pnpm + Vitest の開発基盤を作り、macOS上で起動できる状態にする。

### 参照

- `docs/tech/stack.md`
- `docs/architecture/overview.md`
- `docs/dev/conventions.md`
- `docs/dev/testing.md`

### 対象

- Electron Forge によるアプリ雛形
- React レンダラー
- Zustand による横断的UI状態管理の土台
- TypeScript 設定
- pnpm 設定
- Vitest 設定
- React Testing Library 設定
- macOS 15 Sequoia 以降を対象にした開発設定
- メインプロセス / preload / レンダラープロセス / shared の基本構成
- IPCの型定義と最小API
- アプリ名 `Relic` の反映
- macOS向けの開発起動コマンド

### 完了条件

- `app/` で `pnpm install` 後に開発用アプリを起動できる
- レンダラーに2カラム構成の仮画面が表示される
- メインプロセスとレンダラーがpreload経由の型付きIPCで通信できる
- 最小のユニットテストが実行できる
- 実装基盤に関わるP0未決定事項が残っていない

## フェーズ1：ワークスペースとファイル管理

**ゴール**：任意のローカルフォルダをワークスペースとして開き、Markdownファイルを安全に管理できる。

### 参照

- `docs/spec/file-management.md`
- `docs/spec/navigation.md`
- `docs/ui/screens-macos.md`
- `docs/architecture/data-model.md`

### 対象

- 初回起動ビュー
- 新規ワークスペース作成
- 既存フォルダをワークスペースとして開く
- `attachments/` と `templates/` の自動作成
- 複数ワークスペース登録と切り替え
- 前回ワークスペースの復元
- `.md` ファイルのツリー表示
- 添付画像以外の非Markdownファイルを主要管理対象にしない表示制御
- ファイル作成・削除・リネーム・移動・複製
- フォルダ作成・削除・リネーム・移動
- ピン留め
- テンプレートフォルダの通常表示
- アプリ設定とワークスペース設定の保存

### 完了条件

- 任意のローカルフォルダをワークスペースとして登録できる
- ワークスペース準備時に `attachments/` と `templates/` が作成される
- iCloud Drive / OneDrive / Dropbox などの同期フォルダも通常のローカルフォルダとして扱える
- `.md` ファイルは拡張子を隠して表示される
- 新規ファイル作成時に `.md` が自動付与される
- `/` を含む名前と同一フォルダ内の重複名はエラーになる
- 削除はOSのゴミ箱へ移動し、削除前に確認ダイアログを表示する
- フォルダ削除時も確認ダイアログを表示する
- ファイル操作の結果が実ファイルシステムに反映される
- アプリ再起動後に前回のワークスペースを復元できる

## フェーズ5：フロントマターとコマンド操作

**ゴール**：Markdown本文以外のメタデータ管理と、キーボード操作を整える。

### 参照

- `docs/spec/frontmatter.md`
- `docs/spec/command-palette.md`
- `docs/spec/search.md`
- `docs/spec/navigation.md`
- `docs/ui/screens-macos.md`

### 対象

- フロントマターの読み書き
- ライブプレビュー中のフォームUI
- ソースモード中のYAML直接編集
- 組み込みフィールド（`tags`, `aliases`）
- 固定システムフィールド（`date`, `status`, `publish`, `url`, `author`）
- ユーザー定義フィールド
- 自由フィールド
- フロントマタースキーマ管理
- ワークスペース直下の `frontmatter.md` による候補定義
- フロントマター検索・フィルター
- コマンドパレット
- クイックスイッチャー

### 完了条件

- フロントマターをフォームとソースの両方から扱える
- フォームとYAMLが同期する
- 組み込みフィールドを削除・型変更できない
- ユーザー定義フィールドは最大13個、合計20フィールドの上限を守る
- `frontmatter.md` がある場合は候補入力に連動し、ない場合は自由入力として扱える
- 自由フィールドを壊さず保持できる
- 主要操作をコマンドパレットから実行できる
- クイックスイッチャーで現在のワークスペース内の `.md` ファイルだけを開ける
- 既に開いているファイルを選択した場合は、そのタブをアクティブにする

## フェーズ6：Git・GitHub連携

**ゴール**：GitHub接続なしでもローカル履歴管理でき、接続時は同期まで行える。

### 参照

- `docs/spec/github.md`
- `docs/tech/git-implementation.md`
- `docs/architecture/overview.md`
- `docs/architecture/data-model.md`
- `docs/principles.md`

### 対象

- isomorphic-git によるローカルGit操作
- ローカルGit初期化
- ローカル履歴管理
- コミット履歴
- Diff表示
- ブランチ作成・切り替え
- ブランチ切り替え時の未コミット変更確認
- ブラウザOAuthによるGitHub接続
- GitHubリポジトリのクローン
- ローカルGitリポジトリの後からGitHub接続
- 手動同期
- 手動同期前の差分確認
- 自動同期設定（デフォルトOFF、5分 / 15分 / 30分 / 60分）
- 自動プッシュ時の保存完了待ち
- Gitタグの作成・一覧・プッシュ・削除
- コンフリクト解決UI
- GitHubエラー表示と再試行

### 完了条件

- GitHub未接続でもローカルGit履歴を使える
- GitHub認証はブラウザOAuthで行い、Personal Access Token貼り付け方式を持たない
- 手動同期時に差分確認画面が必ず表示される
- 自動同期はユーザーがオンにした場合だけ動く
- 自動コミットメッセージが `Update notes: YYYY-MM-DD HH:mm` 形式になる
- ブランチ切り替え時、未コミット変更がある場合は3択を表示する
- コンフリクト時に二択と手動編集の両方を選べる
- GitHubエラーは原因別メッセージ・再試行・詳細表示を持つ

**ゴール**：初期リリース対象の追加ツールを揃え、自分用ビルドとして継続利用できる状態にする。

### 参照

- `docs/spec/file-tools.md`

- アクセントカラーのユーザー指定を持たない
- 自分用ビルドとして日常利用を開始できる

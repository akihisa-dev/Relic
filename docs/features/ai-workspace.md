# AI Workspace

AI Workspaceは、Relicで開いているローカルMarkdownワークスペースを、AIとの共同作業対象として扱う機能である。

ユーザーが外部のCodexやCoworkでワークスペースを開いて行うような、ファイルの読み取り、整理、作成、編集、削除を、Relic内の右パネルから対話的に行えるようにする。

中央のMarkdown編集体験を保ったまま、AIとの共同編集をRelic内で完結させる。

---

## 基本方針

- AI Workspaceは、Relic全体をAI中心のアプリへ置き換えるものではない。
- 中央はMarkdownを読む、書く、編集する場所のままにする。
- 左はワークスペースとファイルを扱う場所のままにする。
- 右パネルを、AIと共同作業するための操作場所として扱う。
- AIは、ワークスペース内の全Markdownを作業対象にする。
- AIは、Markdownファイルの読み取り、検索、新規作成、編集、削除を対話の中で行う。
- 作業結果は、Relic内のMarkdownファイルへ反映される。
- Markdownファイルが正本であり、AI系データはMarkdown正本ではない。
- Diff、履歴、参照ファイル表示、承認UIは中核体験ではなく、共同作業を安全または便利にするための補助要素として扱う。

## 体験の中心

AI Workspaceの中心は、ユーザーとAIが会話しながらローカルMarkdownワークスペースを一緒に編集していく体験である。

ユーザーは右パネルで、たとえば次のように自然文で指示する。

- この設定資料を整理して
- この内容を仕様書に反映して
- さっきの変更はやめて
- このファイルだけ直して
- 新しい説明ページを作って
- この古いメモは削除して

AIは会話の文脈とワークスペース内のMarkdownをもとに、必要なMarkdownを読み、探し、作成し、編集し、削除する。

「それ反映して」「やめて」「この部分だけ」などの自然文のやり取りも、AI Workspaceの操作として扱う。ボタンや差分表示だけを操作の中心にしない。

## 対象範囲

AI Workspaceは、ワークスペース内の全Markdownを対象にする。

- `README.md` や `docs/**` などの固定パスには限定しない。
- RelicはMarkdown操作アプリであるため、AIもMarkdownだけを作業対象にする。
- 初期仕様では、除外フォルダや除外ファイルの設定は持たない。
- 将来必要になった場合のみ、除外設定を検討する。

大きすぎるMarkdownは、AI参照とインデックス対象から除外する。除外されたファイルは、AIが読んでいないことをユーザーに分かる形で表示する。大きいファイルの先頭だけを読む仕様にはしない。

読めないMarkdownがあっても、AI Workspace全体は止めない。読めないMarkdownはAI参照とインデックス対象から除外し、AIは残りのMarkdownで動作する。AIは読めなかったファイルを読んだかのように扱わない。

## 右パネル

右パネルは、AI WorkspaceにおけるAI共同作業パネルである。

右パネル全体には、少なくとも次の表示モードを置く。

- AI
- アウトライン
- リンク / バックリンク

アウトライン、リンク、バックリンクは、Markdown作業を助ける情報としてAIと同じ右パネル内で共存させる。

AI表示モード内には、少なくとも次の表示を置く。

- 会話
- 提案または作業中の変更
- 履歴

会話では、AIとのやり取りと回答ごとの参照ファイルを表示する。提案または作業中の変更では、AIがこれから反映しようとしている作成、編集、削除の内容を確認できる。履歴では、現在ワークスペースの過去の会話と変更作業を見返せる。

AIの作業によって、中央エディタの表示を勝手に切り替えない。作成、編集、削除の結果は右パネルに表示し、ユーザーは右パネルの結果表示や履歴から対象ファイルを開ける。

## AIによるMarkdown操作

AIは、会話の中で次のMarkdown操作を行える。

| 操作 | 内容 |
|------|------|
| 読み取り | 既存Markdownを読む |
| 検索 | ワークスペース内のMarkdownを探す |
| 新規作成 | 新しいMarkdownファイルを作る |
| 編集 | 既存Markdownを変更する |
| 削除 | 既存Markdownを削除する |

初期仕様では、削除は完全削除ではなくOSのゴミ箱への移動として扱う。

将来的には、Markdownファイルのリネームや移動もAI操作の対象にできる。

AIは複数Markdownにまたがる作業を扱える。1つの会話の流れの中で、新規作成、編集、削除が混在してよい。

## 変更反映

変更反映は、会話の流れの中で扱う。

ユーザーが「それ反映して」「この変更を適用して」など、反映を意図する指示をした場合、AIは会話中の作業内容をMarkdownへ反映する。ユーザーが「やめて」「まだ反映しない」「その変更は不要」などと指示した場合は反映しない。

ボタンによる適用、差分表示、変更対象一覧は、自然な対話を補助するために使う。これらをAI Workspaceの中核体験にはしない。

未保存変更があるMarkdownには、AIの変更を自動適用しない。ユーザーに先に保存または破棄を求める。未保存変更へAI変更を自動マージせず、AI変更で未保存内容を上書きしない。

AIが作業内容を組み立てたあと、対象Markdownが外部アプリなどで変更され、作業時点の内容と現在の内容が違う場合は、古い内容に基づく変更を自動適用しない。この場合は、そのファイルへの反映を止め、再作業または再提案を求める。

## 削除

AIはMarkdown削除を会話の中で扱える。

- 削除対象のMarkdownパスを表示する。
- AIが削除しようとしている理由や概要を表示する。
- 右パネル内に削除対象本文全文を常時表示しない。
- 必要な場合、ユーザーは対象Markdownを開いて本文を確認できる。
- 実行時は完全削除ではなく、OSのゴミ箱へ移動する。

## 参照ファイルと送信確認

AIの回答には、参照したMarkdownファイルを回答ごとに表示する。

- 参照ファイルから該当Markdownを開けるようにする。
- AIが参照していないファイルを根拠のように表示しない。
- 大きすぎる、読めないなどの理由で除外されたファイルは、回答の根拠に含めない。

外部AIへ問い合わせる前に、参照予定のMarkdownファイル一覧を表示する。

- ユーザーは内容を確認して送信できる。
- ユーザーは送信をキャンセルできる。
- 初期仕様では、その場で個別ファイル除外はしない。
- 一覧には、大きすぎる、読めないなどの理由で除外されたMarkdownも分かる形で表示する。

ローカルに保存されるAI系データと、外部AIへ送信される可能性があるMarkdown内容は、別の問題として説明する。

## AI系データ

AI系データは、Relicのアプリデータとしてローカル保存する。

- 保存先はElectronの `userData` 配下を想定する。
- ワークスペース内にはAI用キャッシュや独自データベースを作らない。
- AI系データはMarkdown正本ではなく、再生成可能または削除可能な派生データとして扱う。
- AI系データはワークスペースごとに分けて保存する。

AI系データには、次のものを含める。

- AIインデックス
- チャンク
- embeddings
- メタ情報
- 必要な場合の要約
- AI会話履歴
- AI回答履歴
- AI変更作業履歴

AI履歴は現在ワークスペースの履歴として右パネルで見返せる。別ワークスペースのAI履歴やインデックスを混ぜない。

ユーザーは、現在ワークスペースに紐づくAIデータを削除できる。削除対象には、AIインデックス、チャンク、embeddings、メタ情報、要約、AI会話履歴、AI回答履歴、AI変更作業履歴を含める。AIデータ削除は、ワークスペース内のMarkdownファイルには影響しない。

## 履歴

AI履歴は、AIとワークスペースで行った共同作業を見返すための補助機能である。

AI変更作業履歴には、作業内容だけでなく状態も保存する。

状態には少なくとも次のものを含める。

- 未適用
- 適用済み
- 取りやめ
- 適用失敗
- 再作業が必要

複数ファイルにまたがる作業の場合、作業全体の状態とファイルごとの状態を分けて扱う。

## codexappserver

AI Workspaceでは、codexappserverをAI処理の実行基盤として扱う。

Relic本体は、Markdown操作、右パネルUI、作業内容の確認、ファイル反映、AI系ローカルデータ管理を担当する。

codexappserverは、AI会話、推論、Workspace Context処理、必要なAIツール実行の基盤として扱う。

ユーザーがcodexappserverの使い方を直接理解したり、直接操作したりする前提にはしない。Relicの画面上では、右パネルからAIと共同作業できる状態を目指す。

codexappserverの具体的な接続方法、API、起動方法は、実装前の調査で確認して設計へ反映する。

## 実装前に確認すること

実装前に、既存アーキテクチャを調査する。

- Directory Structure
- State Management
- Editor Implementation
- File Management
- IPC境界
- codexappserverとの接続方法
- AI系アプリデータの保存場所と削除方法

調査後、設計案を提示し、設計確認後に実装を開始する。

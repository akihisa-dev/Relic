# AI Workspace

AI Workspaceは、Relicで開いているローカルMarkdownワークスペースを、AIとの共同作業対象として扱う機能である。

ユーザーが外部のCodexやCoworkでワークスペースを開いて行うような、ファイルの読み取り、整理、作成、編集、削除を、Relic内の右パネルから対話的に行えるようにする。

中央のMarkdown編集体験を保ったまま、AIとの共同編集をRelic内で完結させる。

---

## 位置づけ

AI Workspaceは、「MarkdownアプリにAIチャットを追加する」機能ではない。

Relicで開いているローカルMarkdownワークスペースを、そのままAIと共同作業するローカルな作業場として扱うための機能である。

ユーザーは、必要な作業のために外部のCodexやCoworkを別途開き、Relicのワークスペースフォルダを指定して作業することもできる。しかし、Relic内で同じ体験が完結すれば、ワークスペースの閲覧、編集、AIとの対話、結果確認が分断されない。

AI Workspaceの価値は、Relicで見ているMarkdownワークスペースを離れずに、AIと対話しながらその場でMarkdownを育てられることにある。

## 基本方針

- AI Workspaceは、Relic全体をAI中心のアプリへ置き換えるものではない。
- 中央はMarkdownを読む、書く、編集する場所のままにする。
- 左レールはワークスペース、ファイル、AIなどの作業モードを切り替える場所として扱う。
- 左サイドバーは通常時はワークスペースとファイルを扱い、AIモード中はAIチャット履歴を扱う。
- 右パネルを、AIと共同作業するための操作場所として扱う。
- AIは、ワークスペース内の全Markdownを作業対象にする。
- AIは、Markdownファイルの読み取り、検索、新規作成、編集、削除を対話の中で行う。
- 作業結果は、Relic内のMarkdownファイルへ反映される。
- Markdownファイルが正本であり、AI系データはMarkdown正本ではない。
- 初期実装では、右パネルは会話だけに集中する。作業中の変更、変更履歴、送信前の参照ファイル確認は置かない。
- ユーザーがワークスペースを開いてAI Workspaceを使う時点で、そのワークスペース内のMarkdownをAIが読む前提として扱う。

## 体験の中心

AI Workspaceの中心は、ユーザーとAIが会話しながらローカルMarkdownワークスペースを一緒に編集していく体験である。

ユーザーは右パネルで、たとえば次のように自然文で指示する。

- この設定資料を整理して
- この内容を仕様書に反映して
- この内容をもう少し短くして
- このファイルを直して
- 新しい説明ページを作って
- この古いメモは削除して

AIは会話の文脈とワークスペース内のMarkdownをもとに、必要なMarkdownを読み、探し、作成し、編集し、削除する。

「それ反映して」「やめて」「この部分だけ」などの自然文のやり取りも、AI Workspaceの操作として扱う。ボタンや差分表示だけを操作の中心にしない。

## 作業モデル

AI Workspaceでは、ユーザーとAIの会話が作業の中心になる。

AIは、会話中の指示や文脈に応じてワークスペース内のMarkdownを読み、必要な作業を組み立てる。ユーザーは、作業内容を自然文で続けて調整できる。

たとえば、次のような流れを自然な作業として扱う。

1. ユーザーが「このあたりの設定資料を整理して」と依頼する
2. AIが関連するMarkdownを読み、整理方針や変更内容を説明する
3. AIが必要な場合はRelic内のファイル操作としてMarkdownへ反映する
4. ユーザーが「この部分を短くして」または「別ファイルに分けて」と続ける
5. AIが同じ会話の文脈で追加作業を行う

この体験では、AIが変更内容を「提案カード」として出し、ユーザーが常に専用ボタンだけで承認することを前提にしない。

AIとの作業はチャット単位で分けられる。ユーザーは新規チャットを作り、過去のチャットへ切り替えて、その会話文脈でMarkdownワークスペースとの共同作業を再開できる。

チャット名は最初のユーザー発言から自動で仮タイトルを作る。ユーザーが毎回名前を付けなくても、左サイドバーの履歴一覧で作業内容を見分けられるようにする。

## 対象範囲

AI Workspaceは、ワークスペース内の全Markdownを対象にする。

- `README.md` や `docs/**` などの固定パスには限定しない。
- RelicはMarkdown操作アプリであるため、AIもMarkdownだけを作業対象にする。
- 初期仕様では、除外フォルダや除外ファイルの設定は持たない。
- 将来必要になった場合のみ、除外設定を検討する。

大きすぎるMarkdownは、AI参照とインデックス対象から除外する。大きいファイルの先頭だけを読む仕様にはしない。

読めないMarkdownがあっても、AI Workspace全体は止めない。読めないMarkdownはAI参照とインデックス対象から除外し、AIは残りのMarkdownで動作する。AIは読めなかったファイルを読んだかのように扱わない。

## 右パネル

右パネルは、AI WorkspaceにおけるAI共同作業パネルである。

右パネル全体には、少なくとも次の表示モードを置く。

- AI
- アウトライン
- リンク / バックリンク

アウトライン、リンク、バックリンクは、Markdown作業を助ける情報としてAIと同じ右パネル内で共存させる。

AI表示モード内には、選択中チャットの会話を置く。

会話では、ユーザーの依頼、AIの回答、Markdownへ反映した内容、反映できなかった理由を文章で表示する。作業中の変更タブ、変更履歴タブ、参照ファイル確認ダイアログは初期実装では置かない。

AIの作業によって、中央エディタの表示を勝手に切り替えない。作成、編集、削除の結果は右パネルの会話に表示する。

右パネルはAIだけの場所ではない。アウトライン、リンク、バックリンクも、現在のMarkdown作業を助ける情報として右パネル内に残す。

## 左レールとAIチャット履歴

左レールには、AIモードを置く。AIモードのアイコンは `Sparkles` とし、AI Workspaceに関する入口であることを示す。

AIモードを選んだとき、左サイドバーには現在ワークスペースのAIチャット履歴を表示する。

- 「新規チャット」から空のチャットを作れる。
- チャット一覧から過去チャットへ切り替えられる。
- 選択中チャットの会話は右パネルのAI表示モードで表示する。
- AIモードへ切り替えた場合、右パネルはAI表示モードを開く。
- ファイルツリーはAIモード中だけ左サイドバーから隠れる。ファイル操作の場所としての左サイドバーは、ファイルモードへ戻すことで維持する。

既存の1本だけのAI会話履歴がある場合は、「以前のチャット」として1件のチャットへ移行する。既存履歴を破棄せず、新しいチャット履歴UIから見返せるようにする。

## AIによるMarkdown操作

AIは、会話の中で次のMarkdown操作を行える。

| 操作 | 内容 |
|------|------|
| 読み取り | 既存Markdownを読む |
| 検索 | ワークスペース内のMarkdownを探す |
| 新規作成 | 新しいMarkdownファイルを作る |
| 編集 | 既存Markdownを変更する |
| 削除 | 既存Markdownを削除する |

初期仕様では、削除は完全削除ではなくOSのゴミ箱への移動として扱う。

将来的には、Markdownファイルのリネームや移動もAI操作の対象にできる。

AIは複数Markdownにまたがる作業を扱える。1つの会話の流れの中で、新規作成、編集、削除が混在してよい。

## 変更反映

変更反映は、会話の流れの中で扱う。

AIがMarkdown変更を返した場合、Relicは安全検証を通った変更をMarkdownへ反映する。反映したファイル、反映しなかったファイル、反映しなかった理由はAIの回答内に表示する。

ボタンによる適用、差分表示、変更対象一覧は初期実装では置かない。ユーザーは、反映後のMarkdown本文を見ながら、右パネルの自然文で追加修正を依頼する。

未保存変更があるMarkdownには、AIの変更を自動適用しない。ユーザーに先に保存または破棄を求める。未保存変更へAI変更を自動マージせず、AI変更で未保存内容を上書きしない。

AIが作業内容を組み立てたあと、対象Markdownが外部アプリなどで変更され、作業時点の内容と現在の内容が違う場合は、古い内容に基づく変更を自動適用しない。この場合は、そのファイルへの反映を止め、再作業または再提案を求める。

## 削除

AIはMarkdown削除を会話の中で扱える。

- 削除対象のMarkdownパスを表示する。
- AIが削除しようとしている理由や概要を表示する。
- 右パネル内に削除対象本文全文を常時表示しない。
- 必要な場合、ユーザーは対象Markdownを開いて本文を確認できる。
- 実行時は完全削除ではなく、OSのゴミ箱へ移動する。

## 参照ファイルの扱い

AI Workspaceでは、ユーザーが毎回参照ファイルを選んだり、送信前に参照ファイル一覧を確認したりしない。

- Relicで開いているワークスペース内の通常サイズで読めるMarkdownを、AIが読む対象として扱う。
- 現在開いていて未保存のMarkdown本文は、必要に応じて保存済み本文より優先してAIへ渡す。
- 大きすぎる、読めないなどの理由で除外されたMarkdownは、AIが読んだ前提にしない。
- ユーザーに「どのファイルを渡すか」を毎回指定させない。

ワークスペースは有限であり、AI Workspaceはそのワークスペースを共同作業対象として指定して使う機能である。そのため、初期実装では送信前の参照ファイル確認を挟まない。

ローカルに保存されるAI系データ、OpenAI APIキー、外部AIへ送信される可能性があるMarkdown内容は、それぞれ別の問題として説明する。

## 外部送信とローカル保存

AI Workspaceは、ユーザーのPC上にあるMarkdownワークスペースを作業対象にする。

AI系データをローカルのアプリデータとして保存することと、AI処理のためにMarkdown内容が外部AIへ送信される可能性があることは、別の性質の問題として扱う。

- ローカル保存されるAI系データは、ユーザーのPC内に置かれる。
- OpenAI APIを使う場合、問い合わせに必要なMarkdown内容やユーザー入力がOpenAIへ送信される可能性がある。
- 外部へ送る可能性がある内容は、AI Workspaceの説明とAI接続方式の設定で明確にする。
- 初期実装では、送信ごとの参照ファイル確認や個別除外は行わない。

ユーザーがOpenAI API、Responses API、Structured Outputsの仕組みを理解していることを前提にしない。Relicの画面上では、右パネルからAIと共同作業できる状態を提供する。

## AI接続方式

AI Workspaceは、AI接続方式として次の2種類を持つ。

| 方式 | 内容 |
|------|------|
| Codex App Server | Codexアプリ側のApp Serverを使ってAI共同作業を行う |
| OpenAI API | ユーザーが登録したOpenAI APIキーを使ってAI共同作業を行う |

既定のAI接続方式はCodex App Serverとする。

Codex App Server方式では、RelicにOpenAI APIキーを入力しない。ChatGPT Pro / Codex利用者が、API利用料やAPIキー準備で最初の体験を止めずにAI Workspaceを試せることを優先する。

OpenAI API方式は、Codexを使わないユーザー、Codex連携が動かない環境、または明示的にOpenAI API課金で運用したいユーザーの代替経路として残す。

Codex App Serverが起動できない、未インストール、利用できない、通信に失敗するなどの場合でも、Relic本体は落とさない。右パネルには、Codex App ServerでAI処理を完了できなかったことと、必要に応じてAI設定からOpenAI API方式へ切り替えられることを表示する。

既存のアプリ設定にAI接続方式がない場合は、Codex App Server方式として読み込む。既にOpenAI APIキーを保存している場合でも、自動でOpenAI API方式へ切り替えない。AI接続方式はユーザーが明示的に選ぶ。

## AI設定

AI設定には、AI接続方式の選択を置く。

- Codex App Server
- OpenAI API

Codex App Server方式を選んでいる場合、OpenAI APIキー入力、保存、接続確認、削除、APIキー取得リンク、OpenAIモデル選択は表示しない。代わりに、Codexアプリを使うこと、OpenAI APIキー入力が不要であること、利用できない場合はOpenAI API方式へ切り替えられることを表示する。

OpenAI API方式を選んでいる場合、OpenAI APIキーの入力、保存、接続確認、削除、OpenAIモデル選択、OpenAIのAPIキー画面へのリンクを表示する。

## OpenAI APIキー

AI WorkspaceのOpenAI API方式では、ユーザーが自分のOpenAI APIキーをRelicに登録して使う。

APIキーはRelicのAI設定で入力、保存、削除、接続確認できるようにする。AI設定にはOpenAIのAPIキー画面へのリンクを表示する。AI設定は、右パネルの会話とは別の設定画面として扱う。右パネルには、APIキー未設定時にAI設定を促す案内を表示する。

APIキーはワークスペース内のMarkdownファイル、ワークスペース内の設定ファイル、AI系データ、AIインデックス、通常のJSON設定ファイルには保存しない。保存先はOSの安全な資格情報保存領域を優先する。OSの資格情報保存領域を使えない環境では、保存しない、または明示的に安全性が落ちることを説明した代替方式を検討する。

RelicはOpenAI APIキーそのものを表示し続けない。登録済みかどうか、接続確認の結果、削除操作だけを画面に表示する。

Relic独自のユーザーアカウント、Relic独自の課金システム、Relic側サービスを経由した代理課金は、初期仕様の対象外とする。

## OpenAI API利用

AI WorkspaceのOpenAI API方式では、AI処理基盤としてOpenAI APIを使う。

OpenAI API呼び出しでは、Responses APIを使う。AIがMarkdown変更案を返す場面では、Structured Outputsを使い、作成、編集、削除の変更案をRelicが検証できる構造化データとして受け取る。

OpenAIモデルはAI設定で選べるようにする。表示は「標準」「高品質」「軽量」のような抽象ラベルではなく、`gpt-5.5`、`gpt-5.4`、`gpt-5.4-mini`、`gpt-5.4-nano` のような実モデル名をそのまま使う。既定モデルは `gpt-5.4-mini` とする。Markdownワークスペースの整理、要約、作成、編集を日常的に扱う用途では、品質、速度、費用のバランスを優先する。

OpenAI APIの応答が構造化データとして読めない場合、RelicはMarkdown変更案を作らない。AIの文章回答を会話に残せる場合は残し、変更案として扱えなかったことを表示する。

OpenAI APIキーが未設定、無効、権限不足、残高不足、ネットワーク失敗、レート制限などで使えない場合は、Markdownを変更しない。右パネルには、ユーザーが次に何を確認すればよいか分かる表現でエラーを表示する。

OpenAI APIのquota、billing、rate limitなどのエラーは、英語のAPIエラー文をそのまま主表示にせず、日本語で原因候補と次の確認先が分かるように表示する。

## Codex App Server利用

AI WorkspaceのCodex App Server方式では、Codexアプリ側のApp Serverへ依頼を送り、AI回答とMarkdown変更案を受け取る。

Codex App Server方式でも、RelicがMarkdownファイルを直接AIに自由操作させるわけではない。Codex App Serverから返る作成、編集、削除の変更案を、Relic側で既存のAI変更案と同じ `message` と `operations` の形式へ変換し、Relic側で検証したうえで、通ったものだけMarkdownへ反映する。

Codex App Server経路では、少なくとも次の保護を行う。

- 起動失敗時にRelicを落とさない
- stdin送信失敗を長い待機にせず即時エラーにする
- 応答待ちとturn待ちにタイムアウトを設ける
- JSON構造化応答を抽出する
- 構造化JSONでない文章回答も会話には残し、Markdown変更案としては扱わない
- Markdown以外、ワークスペース外、存在しないMarkdownへの更新・削除、既存Markdownと同じパスへの新規作成案をRelic側で採用しない

## AI系データ

AI系データは、Relicのアプリデータとしてローカル保存する。

- 保存先はElectronの `userData` 配下を想定する。
- ワークスペース内にはAI用キャッシュや独自データベースを作らない。
- AI系データはMarkdown正本ではなく、再生成可能または削除可能な派生データとして扱う。
- AI系データはワークスペースごとに分けて保存する。

AI系データには、次のものを含める。

- AIインデックス
- チャンク
- embeddings
- メタ情報
- 必要な場合の要約
- ワークスペースごとのAIチャット
- チャットごとの会話、作業中変更、変更履歴

AIチャットは、ワークスペースごとに分けて扱う。別ワークスペースの会話、作業中変更、変更履歴、インデックスを混ぜない。

ユーザーは、現在ワークスペースに紐づくAIデータを削除できる。削除対象には、AIインデックス、チャンク、embeddings、メタ情報、要約、AIチャット、チャットごとの会話、作業中変更、変更履歴を含める。AIデータ削除は、ワークスペース内のMarkdownファイルには影響しない。

## チャット履歴と作業履歴

AI Workspaceでは、過去のAI作業をチャット単位で見返せるようにする。

チャット履歴は、ChatGPTのように新規チャット単位で分かれる。ただし、RelicのAI Workspaceは汎用チャットではなく、ローカルMarkdownワークスペースとの共同編集を扱うため、チャットごとの作業中変更と変更履歴も同じチャットに紐付ける。

作業中変更や変更履歴は、選択中チャットの文脈として扱う。別チャットの作業中変更を、現在のチャットへ混ぜない。

## 初期実装で守る境界

初期実装では、AI Workspaceを「Relic内でMarkdownワークスペースをAIと共同編集する体験」として成立させることを優先する。

次のことは、初期実装の中心には置かない。

- 汎用ファイル編集
- コード編集
- ワークスペース外ファイル操作
- Markdown以外のファイル作成、編集、削除
- 料金モデル
- 外部同期
- AI用データをワークスペース内に保存すること
- APIキーをワークスペース内や通常のJSON設定ファイルに保存すること
- Codex App Serverだけを唯一のAI処理基盤として必須にすること
- 差分レビューや履歴管理だけを主目的にした画面

Relicが扱うのはMarkdownであり、AI WorkspaceもMarkdownワークスペースの共同編集に集中する。

## 実装前に確認すること

実装前に、既存アーキテクチャを調査する。

- Directory Structure
- State Management
- Editor Implementation
- File Management
- IPC境界
- Codex App ServerとOpenAI APIの呼び出し方法
- APIキーの安全な保存、削除、接続確認方法
- AI系アプリデータの保存場所と削除方法

調査後、設計案を提示し、設計確認後に実装を開始する。

# features/frontmatter.md

フロントマター機能の詳細仕様書。
Markdownファイルに付与する構造化メタデータの考え方、入力補助、表示、追加・削除の振る舞いを定義する。

---

## フロントマターとは

フロントマターは、Markdownファイル先頭の `---` で囲まれた YAML ブロック。

Relicでは、フロントマターをアプリがノートへ一律に押し付ける属性ではなく、ユーザーが任意のMarkdownファイルに後から付与できるメタ情報として扱う。

```yaml
---
tags:
  - 小説
  - キャラクター
状態: [下書き]
締切: [2026-05-20]
---
```

フロントマターの実体は、常に各 `.md` ファイル先頭にあるYAMLである。Relicはこれを隠れた内部データとして扱わず、Markdownファイルの一部として尊重する。

---

## 基本原則

- デフォルトのフロントマター項目は作らない
- 新規Markdownファイルに自動でフロントマターを付与しない
- フロントマターは任意のMarkdownファイルへ、いつでも追加できる
- 追加したフロントマターは、いつでもファイル単位で削除できる
- フロントマターの追加・削除で本文を壊さない
- 登録されていないYAMLキーも壊さず保持する
- 固定プロパティは検索・絞り込み・入力補助に使える
- YAMLを直接書かなくても扱えるように、Relicはフォーム表示と入力補助を提供する

本文は、あくまでユーザー自身が記述する場所である。Relicは本文領域に対して、ユーザーの明示操作なしに構造追加・自動挿入・自動整形を行わない。

---

## 表示と編集

フロントマターはMarkdownファイルの先頭に実在する特殊ブロックとして扱う。

- ライブプレビューでは、YAMLブロックを本文より控えめなフォームとして表示・編集できる
- ライブプレビューでは、フロントマターを初期状態で折りたたみ、本文がすぐ見えるようにする
- ソースモードでは、フォーム化せずMarkdownファイル内のYAMLとして直接表示・編集する
- フォーム表示では、既存フィールドの値編集、フィールド削除、配列値の追加ができる
- フォーム入力欄へフォーカスしている間は、本文エディタ本体の直接編集を一時的に止め、本文側をクリックすると本文編集へ戻る
- ファイルタブのエディタ右上には、フロントマター固定プロパティを追加するための常設 `+` ボタンを表示する
- フロントマター表示は本文の見出しや本文テキストより強く見えないようにする

折りたたみ時は、フロントマターが存在することとプロパティ件数が分かるヘッダーだけを表示し、隠れたYAML行の空白や行番号は残さない。
ヘッダーをクリックすると展開し、既存フィールドの値編集、削除、配列値の追加を行える。
エディタ設定変更などでエディタ内部を再生成する場合も、同じファイルのフロントマター折りたたみ状態、選択範囲、スクロール位置、フォーカスをできるだけ維持する。
フロントマターを削除してから作り直した場合は、以前の展開状態を引き継がず、初期状態と同じく折りたたみ状態にする。

固定プロパティの説明や `category` 候補など、アプリ全体で使うフロントマター設定は、メインエリアのフロントマター設定パネルタブで扱う。

ファイルタブ内のフォームから値を編集した場合は、ファイル先頭のフロントマター範囲だけを書き換え、本文部分の文字列はそのまま保持する。
フロントマターがないファイルへプロパティを追加した場合は、ファイル先頭にYAMLブロックを作成する。
壊れたフロントマター、未完了のフロントマター、またはYAMLとして安全に読めない値は、フォームから書き戻さない。

フロントマターが壊れていてフォームとして読めない場合でも、エディタ本文の編集は続けられる。
この場合は、フォーム編集や固定プロパティ追加など、YAMLを安全に書き戻せない操作を行わない。

## ファイルタブで行うこと

- このMarkdownファイルに固定プロパティを追加する
- このMarkdownファイルからフィールドを削除する
- 既存フィールドの値を編集する
- 配列値に項目を追加する
- 必要ならソースモードでYAMLとして確認・編集する

固定プロパティの追加は、エディタ右上の常設 `+` ボタンから行う。
追加メニューには、まだそのファイルに存在しない固定プロパティだけを表示する。
すでに存在するプロパティは候補に出さず、既存値を勝手に上書きしない。
候補は基本、年表などのまとまりに分けて表示する。
追加メニューは画面端で見切れにくい位置へ補正する。
空の新規ファイルでも、本文行数に依存せず追加メニューを表示する。
追加メニューは外側クリックまたはEscapeで閉じ、ウィンドウのリサイズやスクロール時は表示位置を更新する。

フロントマターがないファイルに固定プロパティを追加する場合、Relicはファイル先頭にYAMLブロックを作成する。

```md
# 今日のメモ

本文...
```

`tags` を追加した場合：

```yaml
---
tags:
---
# 今日のメモ
```

既にフロントマターがある場合は、既存YAMLブロック内の末尾にキーを追加する。
未完了または不正なYAMLとしてしか読めないフロントマターには、追加操作を実行しない。
この場合は、ソースモードなどでYAMLを修正してから追加する。

## フロントマター設定パネルタブ

フロントマター設定パネルタブは、固定プロパティの確認と `category` 候補の管理に絞った場所として扱う。
設定パネル本体とは別のパネルタブであり、設定パネル本体にはフロントマターテンプレート管理やカスタムプロパティ管理を表示しない。

フロントマター設定パネルタブで行うこと：

- 固定プロパティの意味と書き方を確認する
- `category` の候補値を管理する
- YAML書式の基本を確認する

フロントマター設定パネルタブの初期表示では、`aliases`、`tags`、`category`、`chronicle` の固定プロパティ名を常時表示する。
各固定プロパティの長い説明と書き方例は、該当プロパティを選んだ時だけ展開する。
YAML書式例は、必要な時に開ける補助情報として扱い、初期表示では常時展開しない。
画面上部には `category` 候補数と固定プロパティ数を表示し、現在の管理対象の量を確認できるようにする。

ファイル単位の値編集、固定プロパティ追加、フィールド削除は、対象Markdownファイルのファイルタブ内で行う。
現行UIでは、任意名のカスタムプロパティを追加・編集・削除する管理画面を持たない。
既存設定ファイルに保存済みのカスタムプロパティ定義がある場合でも、フロントマター設定パネルタブでは管理対象として表示しない。
`category` 候補値は、前後の空白を除き、空文字と重複を取り除いて読み込む。

---

## 入力補助

Relicは、固定プロパティの意味に応じてフォームUIと補助機能を提供する。

未登録フィールドはテキストとして表示できる。
固定プロパティは、Relicが意味を知っている入力UIで表示する。
複雑なYAML値は、フォーム内のYAML入力として編集できる。
複雑なYAML値の入力が不正な場合は、Markdown本文を書き換えない。

フォーム編集で値を空にした場合、または配列値をすべて削除した場合は、そのキーを現在ファイルのフロントマターから削除する。
選択肢入力では、現在値が登録済み候補に含まれていない場合も、既存値を選択肢に含めて表示する。
数値入力は数値として解釈できる場合は数値で書き戻し、数値として扱えない場合は文字列として保持する。
日付入力は日付入力として表示し、保存時は `YYYY-MM-DD` へ揃える。
日付入力が不正な場合は該当入力をエラー状態にし、Markdown本文を書き換えない。
真偽値入力はチェックボックスで表示し、表示上の `true` / `false` 文言もチェック状態に合わせて更新する。
配列値の追加ダイアログでは、Enterだけでは追加せず、追加ボタンで確定する。Escapeまたはキャンセルで閉じる。
配列値の追加ダイアログでは空文字を追加せず、エラーを表示する。
`aliases` の配列値追加では、他ファイル由来の候補を表示しない。

---

## 固定プロパティ

`aliases`、`category`、`tags`、`chronicle` はカスタム入力能力ではなく、Relicが意味を知っている固定プロパティとして扱う。

配列プロパティは、YAMLのブロック配列で書く。Relicのフォームで編集した場合も、この形で書き戻す。

```yaml
aliases:
  - 帝都
  - 王都
category: 戦争
tags:
  - 資料
  - 下書き
chronicle:
  start: 1185
  end: 1333
```

`tags` に書かれた値だけをRelicのタグとして扱う。本文中の `#タグ` 記法はタグ化しない。本文中で概念・人物・資料などを参照する場合は、インラインリンク `[[...]]` を使う。

タグはその場で追加できる。ただし、表記揺れを避けるため、Relicはワークスペース内の既存 `tags` から候補を出す。
フォーム編集で `tags` の値を追加・変更する場合、入力値は前後の空白を除いて保存し、先頭の `#` はフォーム保存時には自動で取り除かない。
タグ一覧・タグ検索など、フロントマター `tags` をタグとして解析する処理では、値の前後空白、クォート、先頭の `#` を除いた値をタグとして扱う。

`category` は、Markdownファイルの分類を示すための固定プロパティである。
初期実装では単一値だけを扱い、複数指定は扱わない。
`category` 候補はワークスペース別設定として保存し、フロントマター設定パネルタブで追加・削除できる。
候補値は保存時と読み込み時に前後の空白を除き、空文字と重複を保存しない。
既存Markdownに候補外の `category` 値がある場合も、Relicは値を勝手に削除・変更しない。

`chronicle` は、そのMarkdownファイル自体を右パネルの年表上へ置くための固定プロパティである。
単年は整数のスカラー、期間は `start` と `end` を持つYAMLマッピングで書く。

```yaml
chronicle: 1185
```

```yaml
chronicle:
  start: 1185
  end: 1333
```

- 年表は単一の時間軸で扱い、暦設定や暦換算を持たない
- フォームでは開始年と終了年だけを編集する
- 整数値は単年として扱う
- `start` と `end` は期間として扱う
- フォーム編集では開始年と終了年を別々の入力欄で表示する
- 終了年が空欄または開始年と同じ場合は、単年の整数として保存する
- 年は0以外の整数だけを有効とする
- 小数、文字列は無効とする
- 期間は開始年が終了年以下の場合だけ有効とする
- フォーム編集では、無効な年、開始年が終了年より大きい期間は保存しない
- 空欄は未設定として扱い、値を書き戻さない
- 既存Markdownに無効な `chronicle` エントリが書かれている場合、Relicは勝手に修正せず、フォームではエラーを表示し、年表には表示しない
- マイナス年は年表で表示でき、ラベルは `−2` のようにマイナス記号付きで表示する
- 年表チャートでは左を過去、右を未来として、単年を節点、期間を開始・終了節点と接続線で表示する
- 年数差は順序を保つ圧縮距離へ変換し、長い空白期間をそのままの比率では表示しない
- 年ラベルは画面上端の固定ヘッダーに表示し、ヘッダーから各年の縦ガイドを項目の領域まで伸ばす
- 年表は閲覧専用とし、項目の一時ドラッグはMarkdownへ書き戻さない
- 通常の濃さで表示されているファイル名をクリックすると、年表タブを残したまま対象ファイルを開く
- 入力途中や不正な `chronicle` は年表へ反映しない

## 削除方法

フロントマターの削除は、少なくとも以下を分けて扱う。

| 操作 | 内容 |
|------|------|
| このファイルから削除 | 現在のMarkdownファイルのYAMLからキーを削除する |
| 入力補助の登録を削除 | 現行UIには持たない。旧設定に保存された入力能力や候補を削除する場合は別作業として扱う |
| 全ファイルから削除 | 現行UIには持たない。ワークスペース内のMarkdownファイルから該当キーを一括削除する場合は別作業として扱う |

通常の削除操作は「このファイルから削除」とする。旧設定に保存された入力補助定義を削除する場合でも、既存MarkdownファイルのYAML自体は勝手に消さない。

ファイルタブのフォーム表示では、各プロパティ行の削除操作は現在のMarkdownファイルのトップレベルYAMLキーだけを対象にする。
複数行で書かれた値を削除する場合も、そのキーに属する範囲だけを削除し、隣接する別プロパティは削除しない。

全ファイルから削除する操作は影響が大きいため、現行アプリの通常UIには置かない。

最後のフィールドを削除して空のフロントマターだけが残る場合、Relicは空のYAMLブロックを削除する。

プロパティ編集では、対象プロパティだけを書き換え、対象外プロパティ、コメント、既存の並びをできるだけ保持する。
単純な文字列スカラーのクォートも、可能な範囲で保持する。
元の値がフロー配列で書かれていても、配列プロパティをフォーム編集した場合はブロック配列として書き戻す。
旧形式の `chronicle` は読み込み互換の対象とするが、フォーム編集後は単年スカラーまたは `start` / `end` のマッピングとして書き戻す。

## フロントマターテンプレート

フロントマターテンプレートは、複数のフィールド名をまとめるアプリ設定上の保存済みセットである。

現行UIでは、テンプレート管理画面やテンプレート適用操作は表示しない。テンプレート定義を扱う場合も、ワークスペース内の `templates/` フォルダや `.md` テンプレートファイルには依存しない。
保存構造としては、テンプレート名 `name` と、そのテンプレートに含めるフィールド名配列 `fieldNames` をアプリ設定に持つ。
保存時は、入力全体が配列であり、テンプレート名が空でなく同名テンプレートが重複せず、`fieldNames` が1件以上あり、各フィールド名が改行や `:` を含まず先頭空白で始まらない場合だけ受け付ける。
壊れた保存済みテンプレートを読み込む場合は、有効なテンプレートだけを復元し、テンプレート名は前後空白を除いて扱う。
現行UIに管理・適用操作がないため、新規Markdownファイル作成時にテンプレートを選ばせたり、テンプレートから本文やYAMLブロックを自動挿入したりしない。

例：

```yaml
---
type: [book]
title: []
author: []
状態: [読書中]
tags: []
---
```

テンプレート由来のキーも、Markdownに書き込まれた後は通常のYAMLとして扱う。ファイルごとに自由に追加・削除・編集でき、以後そのファイルがテンプレートに固定されるわけではない。

---

## 未認識フィールド

既存ファイルや外部エディタで作成されたYAMLには、Relicが入力能力を知らないキーが含まれる場合がある。

Relicは未認識フィールドを壊さず保持する。未認識フィールドはテキストとして表示できる。
現行UIでは、未認識フィールドへ入力能力を割り当てる管理画面は持たない。

---

## 検索・フィルターとの連動

固定プロパティは、構造的な検索・絞り込みの対象にできる。

未登録フィールドもファイル内のYAMLとして存在するため、全文検索の対象には含まれる。ただし、意味を前提にした構造的な絞り込みは、Relicが認識しているフィールドを対象とする。

---

## 一文での定義

Relicのフロントマターは、固定プロパティを入口にしながら、ユーザーが任意のMarkdownファイルへ後から自由に付与・削除できるメタ情報フィールドである。

# features/frontmatter.md

フロントマター機能の詳細仕様書。
Markdownファイルに付与する構造化メタデータの考え方、入力補助、表示、追加・削除の振る舞いを定義する。

---

## フロントマターとは

フロントマターは、Markdownファイル先頭の `---` で囲まれた YAML ブロック。

Relicでは、フロントマターをアプリがノートへ一律に押し付ける属性ではなく、ユーザーが任意のMarkdownファイルに後から付与できるメタ情報として扱う。

```yaml
---
tags: [小説, キャラクター]
状態: [下書き]
締切: [2026-05-20]
---
```

フロントマターの実体は、常に各 `.md` ファイル先頭にあるYAMLである。Relicはこれを隠れた内部データとして扱わず、Markdownファイルの一部として尊重する。

---

## 基本原則

- デフォルトのフロントマター項目は作らない
- 新規Markdownファイルに自動でフロントマターを付与しない
- フロントマターは任意のMarkdownファイルへ、いつでも追加できる
- 追加したフロントマターは、いつでもファイル単位で削除できる
- フロントマターの追加・削除で本文を壊さない
- 登録されていないYAMLキーも壊さず保持する
- 登録済みキーは検索・絞り込み・入力補助に使える
- YAMLを直接書かなくても扱えるように、Relicはフォーム表示と入力補助を提供する

本文は、あくまでユーザー自身が記述する場所である。Relicは本文領域に対して、ユーザーの明示操作なしに構造追加・自動挿入・自動整形を行わない。

---

## 表示と編集

フロントマターはMarkdownファイルの先頭に実在する特殊ブロックとして扱う。

- ライブプレビューでは、YAMLブロックを本文より控えめなフォームとして表示・編集できる
- ライブプレビューでは、フロントマターを初期状態で折りたたみ、本文がすぐ見えるようにする
- ソースモードでは、フォーム化せずMarkdownファイル内のYAMLとして直接表示・編集する
- フォーム表示では、既存フィールドの値編集、フィールド削除、配列値の追加ができる
- フォーム入力欄へフォーカスしている間は、本文エディタ本体の直接編集を一時的に止め、本文側をクリックすると本文編集へ戻る
- ファイルタブのエディタ右上には、フロントマター固定プロパティを追加するための常設 `+` ボタンを表示する
- フロントマター表示は本文の見出しや本文テキストより強く見えないようにする

折りたたみ時は、フロントマターが存在することとプロパティ件数が分かるヘッダーだけを表示し、隠れたYAML行の空白や行番号は残さない。
ヘッダーをクリックすると展開し、既存フィールドの値編集、削除、配列値の追加を行える。
エディタ設定変更などでエディタ内部を再生成する場合も、同じファイルのフロントマター折りたたみ状態、選択範囲、スクロール位置、フォーカスをできるだけ維持する。
フロントマターを削除してから作り直した場合は、以前の展開状態を引き継がず、初期状態と同じく折りたたみ状態にする。

入力能力、候補、固定プロパティの説明など、アプリ全体で使うフロントマター設定は、右パネルではなくメインエリアのフロントマター設定パネルタブで扱う。
右パネルはアウトラインとリンク一覧の補助表示に限定し、フロントマター構造の管理には使わない。

フロントマターが壊れていてフォームとして読めない場合でも、エディタ本文の編集は続けられる。
この場合は、フォーム編集や固定プロパティ追加など、YAMLを安全に書き戻せない操作を行わない。

## ファイルタブで行うこと

- このMarkdownファイルに固定プロパティを追加する
- このMarkdownファイルからフィールドを削除する
- 既存フィールドの値を編集する
- 配列値に項目を追加する
- 必要ならソースモードでYAMLとして確認・編集する

固定プロパティの追加は、エディタ右上の常設 `+` ボタンから行う。
追加メニューには、まだそのファイルに存在しない固定プロパティだけを表示する。
すでに存在するプロパティは候補に出さず、既存値を勝手に上書きしない。
候補は日付、基本、年表などのまとまりに分けて表示する。
追加メニューは画面端で見切れにくい位置へ補正する。
空の新規ファイルでも、本文行数に依存せず追加メニューを表示する。
追加メニューは外側クリックまたはEscapeで閉じ、ウィンドウのリサイズやスクロール時は表示位置を更新する。

フロントマターがないファイルに固定プロパティを追加する場合、Relicはファイル先頭にYAMLブロックを作成する。

```md
# 今日のメモ

本文...
```

`plannedDate` を追加した場合：

```yaml
---
plannedDate:
---
# 今日のメモ
```

既にフロントマターがある場合は、既存YAMLブロック内の末尾にキーを追加する。
未完了または不正なYAMLとしてしか読めないフロントマターには、追加操作を実行しない。
この場合は、ソースモードなどでYAMLを修正してから追加する。

## フロントマター設定パネルタブ

フロントマター設定パネルタブは、アプリ全体で使うカスタムプロパティの入力能力を管理する場所として扱う。
設定パネル本体とは別のパネルタブであり、設定パネル本体にはフロントマターテンプレート管理やカスタムプロパティ管理を表示しない。

フロントマター設定パネルタブで行うこと：

- 固定プロパティの意味と書き方を確認する
- カスタムプロパティを追加する
- カスタムプロパティの入力能力を選ぶ
- 選択肢・複数候補の候補値を管理する
- カスタムプロパティを削除する
- 同じ名前のカスタムプロパティ、空の名前、改行や `:` を含む名前、予約名と同じ名前は保存しない

ファイル単位の値編集、固定プロパティ追加、フィールド削除は、対象Markdownファイルのファイルタブ内で行う。
カスタムプロパティ定義はアプリ設定として保存し、ワークスペース別設定ファイルには保存しない。
保存済みのカスタムプロパティ定義に壊れた値がある場合、Relicは読み込み時に無効な定義だけを除外する。
選択肢・複数候補の候補値は、前後の空白を除き、空文字と重複を取り除いて読み込む。
選択肢を持てるのは、選択肢入力と複数候補入力のカスタムプロパティだけとする。

---

## 能力付きフィールド

フロントマターの中心は、固定テンプレートではなく、能力付きフィールドである。

フィールドは任意の名前を持てる。その名前に対して、Relic側で入力能力を付与する。

例：

| フィールド名 | 入力能力 |
|--------------|----------|
| `締切` | 日付入力 |
| `状態` | 選択肢入力 |
| `登場人物` | 複数候補入力 |
| `参考URL` | URL入力 |

カスタムプロパティでは、ユーザーが任意の名前に任意の入力能力を割り当てられる。ただし、`aliases`、`tags`、`status`、`chronicle0`〜`chronicle9`、`plannedDate`、`actualDate` はRelicが意味を知っている予約名として扱い、カスタムプロパティには追加しない。

カスタムプロパティもMarkdown上では通常のYAMLキーとして書く。単体値でも `[]` を使い、固定プロパティと同じ1行配列に揃える。フロントマター設定パネルタブでは、入力能力ごとに実際の書き方例を表示する。

```yaml
状態: [下書き]
巻数: [3]
締切: [2026-05-20]
公開日時: [2026-05-20T09:30]
公開時刻: [09:30]
公開済み: [true]
登場人物: [Alice, Bob]
参考URL: [https://example.com]
```

---

## 入力能力

Relicは、登録されたフィールドの入力能力に応じてフォームUIと補助機能を提供する。

| 入力能力 | UI・補助 |
|----------|----------|
| テキスト | 通常のテキスト入力 |
| 数値 | 数値入力 |
| 日付 | カレンダー入力 |
| 日時 | 日時入力 |
| 時刻 | 時刻入力 |
| 真偽値 | トグル |
| 選択肢 | 候補から選ぶ入力 |
| 複数候補 | ピル入力、候補補完 |
| URL | URL入力 |

候補を持つ入力能力では、事前に登録した候補や、ワークスペース内で過去に使われた値をもとに、ユーザー入力に応じて候補を絞り込む。

未登録フィールドはテキストとして表示できる。
登録済みフィールドは、登録された入力能力に応じた入力UIで表示する。
複雑なYAML値は、フォーム内のYAML入力として編集できる。
複雑なYAML値の入力が不正な場合は、Markdown本文を書き換えない。

フォーム編集で値を空にした場合、または配列値をすべて削除した場合は、そのキーを現在ファイルのフロントマターから削除する。
選択肢入力では、現在値が登録済み候補に含まれていない場合も、既存値を選択肢に含めて表示する。
数値入力は数値として解釈できる場合は数値で書き戻し、数値として扱えない場合は文字列として保持する。
日付入力は、表示設定に応じて `YYYY-MM-DD`、`MM/DD/YYYY`、`DD/MM/YYYY`、またはOS標準の日付入力として表示し、保存時は `YYYY-MM-DD` へ揃える。
日付入力が不正な場合は該当入力をエラー状態にし、Markdown本文を書き換えない。
日時、時刻、URLのカスタムプロパティは、ブラウザ標準の `datetime-local`、`time`、`url` 入力として表示する。
真偽値入力はチェックボックスで表示し、表示上の `true` / `false` 文言もチェック状態に合わせて更新する。
配列値の追加ダイアログでは、Enterだけでは追加せず、追加ボタンで確定する。Escapeまたはキャンセルで閉じる。
配列値の追加ダイアログでは空文字を追加せず、エラーを表示する。
`aliases` の配列値追加では、他ファイル由来の候補を表示しない。

## 固定プロパティ

`aliases`、`tags`、`status`、`chronicle0`〜`chronicle9`、`plannedDate`、`actualDate` はカスタム入力能力ではなく、Relicが意味を知っている固定プロパティとして扱う。

固定プロパティも、単体値でも複数値・期間でも1行の配列で書く。Relicのフォームで編集した場合も、この形で書き戻す。

```yaml
aliases: [帝都, 王都]
tags: [資料, 下書き]
status: [未着手]
chronicle0: [1185, 1333]
plannedDate: [2026-05-12, 2026-05-20]
actualDate: [2026-05-12, 2026-05-20]
```

`tags` に書かれた値だけをRelicのタグとして扱う。本文中の `#タグ` 記法はタグ化しない。本文中で概念・人物・資料などを参照する場合は、インラインリンク `[[...]]` を使う。

タグはその場で追加できる。ただし、表記揺れを避けるため、Relicはワークスペース内の既存 `tags` から候補を出す。
フォーム編集で `tags` の値を追加・変更する場合、入力値は前後の空白を除いて保存し、先頭の `#` はフォーム保存時には自動で取り除かない。
タグ一覧・タグ検索など、フロントマター `tags` をタグとして解析する処理では、値の前後空白、クォート、先頭の `#` を除いた値をタグとして扱う。

`status` は、日付チャートの状態表示と絞り込みに使う固定プロパティである。候補は `未着手`、`進行中`、`完了`、`中断`、`中止` とし、フォーム編集では1つだけを選んで1要素の配列として書き戻す。

```yaml
status: [未着手]
```

`chronicle0`〜`chronicle9` は、そのMarkdownファイル自体を年表上の単年または期間に置くための固定プロパティである。値は常に1行の整数配列で書く。

`chronicle0` はメイン暦の年を表す。`chronicle1`〜`chronicle9` はサブ暦の年を表し、暦設定で登録された暦だけを年表に表示する。旧 `chronicle` は固定プロパティではなく、Markdownフロントマターから生成する新仕様の年表では読み取らない。

- 暦設定はワークスペース別設定として保存する
- 初期状態では `chronicle0` のメイン暦だけを持ち、名前は `メイン暦` とする
- メイン暦 `chronicle0` は削除できず、開始年は常に1として扱う
- サブ暦は `chronicle1` から `chronicle9` まで追加でき、同じIDを重複して保存しない
- サブ暦の開始年は空欄または1以上の整数だけを保存できる
- サブ暦の開始年が不正な場合は保存せず、入力欄の近くにエラーを表示する
- サブ暦の設定画面では、開始年をもとにメイン暦への換算プレビューを表示する。開始年が空欄または不正な場合は、換算できないことを表示する
- 壊れたワークスペース別設定から暦設定を読み込む場合、無効な暦ID、重複ID、不正な開始年は安全側で除外または開始年なしとして扱う
- `[年]` は単年として扱う
- `[開始年, 終了年]` は期間として扱う
- フォーム編集では、開始年と終了年を別々の入力欄で表示する
- 終了年が空欄、または開始年と同じ場合は単年として保存する
- `chronicle0` の年は1以上の整数だけを有効とする
- `chronicle1`〜`chronicle9` の年は0以下を含む整数を有効とする
- 小数、文字列は無効とする
- 期間は開始年が終了年以下の場合だけ有効とする
- フォーム編集では、無効な年や開始年が終了年より大きい期間は保存せず、入力欄の近くにエラーを表示する
- 空欄は未設定として扱い、値を書き戻さない
- 既存Markdownに無効な `chronicleN` が書かれている場合、Relicは勝手に修正せず、フォームではエラーを表示し、年表には表示しない
- 1つのMarkdownファイルに複数の有効な `chronicle0`〜`chronicle9` がある場合、年表チャートでは暦設定の並び順で最初に有効な1件だけをそのファイルの年表行として使う
- サブ暦の年は、暦設定の開始年をもとに `メイン暦年 = サブ暦開始年 + サブ暦年 - 1` でメイン暦へ換算する
- サブ暦の開始年が未設定の場合、そのサブ暦の `chronicleN` は年表に表示しない
- サブ暦名が未設定で開始年がある場合は、設定を保存でき、年表では `chronicleN` を暦名として表示する
- サブ暦の0年やマイナス年は年表で表示でき、マイナス年のラベルは `−2` のようにマイナス記号付きで表示する
- 旧形式のワークスペース設定やレンダラー側に残っている旧チャート入力が `{ path, fileName, startYear, endYear }` の形で渡された場合は、互換表示のために現行の年表チャートへ正規化する。ただし、Markdownフロントマターの旧 `chronicle` を新たに年表対象として読むことはしない
- 年表タブでは、ファイル名を左列に固定表示し、右側の横軸年表に換算後の単年または期間を塗りとして表示する
- 単年はその年、期間は開始年から終了年までの範囲を塗り、塗りの上には暦名と元の年または期間を表示する
- チャートはグリッドで表示し、`chronicle0`〜`chronicle9` は1年単位、日付チャートは1日単位で固定表示する
- チャートには検索欄、開始値順・名前順の並び替え、現在の並び順を保持する更新ボタンを表示する
- チャートには横方向ミニマップ、縦方向ミニマップ、画面外のバーへ移動するジャンプボタンを表示する
- 年表チャートでは、左列のファイル名をクリックすると対象ファイルを開き、左列の期間概要をクリックするとその期間へ横スクロールする
- チャートバーはクリックしてもファイルを開かない
- チャートバー本体をドラッグすると、対応Markdownファイルの元の `chronicleN` を移動後の年または期間へ書き戻す
- チャートバーの左右端をドラッグすると、対応Markdownファイルの元の `chronicleN` の開始年または終了年を書き戻す
- 対応Markdownファイルに有効な `plannedDate` または `actualDate` がある場合、`chronicleN` バーの移動・長さ変更に合わせて、換算後のメイン暦年の差分で日付側も書き戻す
- チャートバー編集による書き戻しは、対象プロパティだけを差し替え、対象外プロパティ、コメント、既存の並びをできるだけ保持する
- チャートバー編集時にフロントマターが未完了または不正なYAMLの場合、Relicは書き戻しを行わず、ユーザーに保存できなかったことを表示する

`plannedDate` と `actualDate` は、そのMarkdownファイルの予定日・実績日または期間を表すための固定プロパティである。値は常に1行の日付配列で書く。

- `[YYYY-MM-DD]` は単日として扱う
- `[開始日, 終了日]` は期間として扱う
- 入力は年月日のみとし、時刻は扱わない
- Markdown上の推奨形式は `YYYY-MM-DD` だけとする。ただし、既存データにOS由来の日付文字列やYAMLパーサー上のDate値として解釈できる値がある場合、チャート表示とチャート編集時には `YYYY-MM-DD` に正規化して扱う
- 旧 `date` は固定プロパティではなく、`plannedDate` としても `actualDate` としても読み取らない
- 不正な日付、3要素以上の配列、開始日が終了日より後の期間は、チャート表示とチャート編集の連動対象から除外する
- 期間入力時は、開始日と終了日をフォームで入力できる
- 終了日が空欄、または開始日と同じ場合は単日として保存する
- フロントマターのフォーム上の日付順はユーザー設定で選べる
- フォーム編集後は `plannedDate: [2026-05-12]`、`plannedDate: [2026-05-12, 2026-05-20]`、`actualDate: [2026-05-12]`、`actualDate: [2026-05-12, 2026-05-20]` の形で書き戻す
- カレンダータブでは、`plannedDate` と `actualDate` をパスごとに同じファイル行へ集約し、予定、実績の順に表示する
- `status` は `plannedDate` と `actualDate` の両方に紐づく状態値として扱い、空白だけの値を除いて状態バッジに表示する
- カレンダータブでは、検索に加えて `status` の固定候補で絞り込める
- カレンダータブでは「今日」ボタンで現在日へ横スクロールできる
- カレンダータブでは、左列のファイル名をクリックすると対象ファイルを開く
- 日付バー本体をドラッグすると、対応Markdownファイルの `plannedDate` または `actualDate` を移動後の日付または期間へ書き戻す
- 日付バー端をドラッグすると、対応Markdownファイルの `plannedDate` または `actualDate` の開始日または終了日を書き戻す
- 日付バーの移動・長さ変更後の日付範囲から年を取り出し、対応Markdownファイルの `chronicle0` も同じ年または期間へ書き戻す

## 削除方法

フロントマターの削除は、少なくとも以下を分けて扱う。

| 操作 | 内容 |
|------|------|
| このファイルから削除 | 現在のMarkdownファイルのYAMLからキーを削除する |
| 入力補助の登録を削除 | Relicがそのキーに付与していた入力能力や候補を削除する |
| 全ファイルから削除 | 現行UIには持たない。ワークスペース内のMarkdownファイルから該当キーを一括削除する場合は別作業として扱う |

通常の削除操作は「このファイルから削除」とする。入力補助の登録を削除しても、既存MarkdownファイルのYAML自体は勝手に消さない。

ファイルタブのフォーム表示では、各プロパティ行の削除操作は現在のMarkdownファイルのトップレベルYAMLキーだけを対象にする。
複数行で書かれた値を削除する場合も、そのキーに属する範囲だけを削除し、隣接する別プロパティは削除しない。

全ファイルから削除する操作は影響が大きいため、現行アプリの通常UIには置かない。

最後のフィールドを削除して空のフロントマターだけが残る場合、Relicは空のYAMLブロックを削除する。

プロパティ編集では、対象プロパティだけを書き換え、対象外プロパティ、コメント、既存の並びをできるだけ保持する。
単純な文字列スカラーのクォートも、可能な範囲で保持する。
元の値が複数行配列で書かれていても、`aliases`、`tags`、固定日付、年表、登録済みカスタムプロパティをフォーム編集した場合は、1行配列として書き戻す。

## フロントマターテンプレート

フロントマターテンプレートは、複数の能力付きフィールド名をまとめるアプリ設定上の保存済みセットである。

現行UIでは、テンプレート管理画面やテンプレート適用操作は表示しない。テンプレート定義を扱う場合も、ワークスペース内の `templates/` フォルダや `.md` テンプレートファイルには依存しない。
保存構造としては、テンプレート名 `name` と、そのテンプレートに含めるフィールド名配列 `fieldNames` をアプリ設定に持つ。
保存時は、入力全体が配列であり、テンプレート名が空でなく同名テンプレートが重複せず、`fieldNames` が1件以上あり、各フィールド名が改行や `:` を含まず先頭空白で始まらない場合だけ受け付ける。
壊れた保存済みテンプレートを読み込む場合は、有効なテンプレートだけを復元し、テンプレート名は前後空白を除いて扱う。
現行UIに管理・適用操作がないため、新規Markdownファイル作成時にテンプレートを選ばせたり、テンプレートから本文やYAMLブロックを自動挿入したりしない。

例：

```yaml
---
type: [book]
title: []
author: []
状態: [読書中]
tags: []
---
```

テンプレート由来のキーも、Markdownに書き込まれた後は通常のYAMLとして扱う。ファイルごとに自由に追加・削除・編集でき、以後そのファイルがテンプレートに固定されるわけではない。

---

## 未認識フィールド

既存ファイルや外部エディタで作成されたYAMLには、Relicが入力能力を知らないキーが含まれる場合がある。

Relicは未認識フィールドを壊さず保持する。未認識フィールドはテキストとして表示でき、必要に応じて後から入力能力を割り当てられる。

---

## 検索・フィルターとの連動

登録済みフィールドは、構造的な検索・絞り込みの対象にできる。

未登録フィールドもファイル内のYAMLとして存在するため、全文検索の対象には含まれる。ただし、入力能力や型を前提にした構造的な絞り込みは、Relicが認識しているフィールドを対象とする。

---

## 一文での定義

Relicのフロントマターは、固定スキーマではなく、ユーザーが任意のMarkdownファイルへ後から自由に付与・削除できる、能力付きメタ情報フィールドである。

# architecture/decisions.md

アーキテクチャ上の重要な意思決定を記録するファイル。
「なぜその選択をしたか」の理由と背景を残す。設計を変更する際は必ずここを確認すること。

---

## 001 · プラットフォームを macOS に限定する

**決定**：macOS 専用アプリとして開発する。iOS は将来的に別アプリとして検討。

**理由**：
- Markdown ノートアプリの主要ユーザーはデスクトップユーザー
- iOS と同時開発するとどちらも中途半端になるリスクがある
- まず macOS で完成度を上げてから iOS を検討する

---

## 002 · フレームワークに Electron を採用する

**決定**：デスクトップアプリのフレームワークとして Electron を採用する。

**理由**：
- TypeScript / JavaScript の資産をそのまま使える
- macOS ネイティブアプリに近い体験を Web 技術で実現できる
- エディタエンジン（CodeMirror 6）・Git 実装（isomorphic-git）ともに Web 技術ベースで相性が良い

---

## 003 · エディタエンジンに CodeMirror 6 を採用する

**決定**：Markdown エディタの中核ライブラリとして CodeMirror 6 を採用する。

**理由**：
- Obsidian（同様のユースケース）が採用しており実績が豊富
- ライブプレビューとソースモードの切り替えが設計の根幹に組み込まれている
- TypeScript ネイティブで型安全な開発が可能
- `[[...]]` や `#tag` などのカスタム構文も拡張システムで実装しやすい

詳細は [tech/editor-engine.md](../tech/editor-engine.md) を参照。

---

## 004 · Git 実装に isomorphic-git を採用する

**決定**：Git 機能の実装ライブラリとして isomorphic-git を採用する。

**理由**：
- JavaScript のみで書かれており、ユーザーの Mac に Git がインストールされていなくても動作する
- 非エンジニアも使う想定のため、環境依存をゼロにすることが重要
- Electron との相性が良く、ネイティブコンパイルが不要

詳細は [tech/git-implementation.md](../tech/git-implementation.md) を参照。

---

## 005 · ワークスペースは任意のローカルフォルダを登録する

**決定**：ユーザーがMac上の任意のローカルフォルダをワークスペースとして登録する。iCloud Drive内のフォルダも、通常のローカルフォルダと同じように登録できる。

**理由**：
- 既存の執筆フォルダやGitHubリポジトリをそのまま開ける
- Obsidianと同じ方式で、ユーザーが自分のファイルの所在を把握しやすい
- iCloud Driveを使いたい場合は、ユーザーがiCloud Drive内のフォルダを選べばよい
- 保存場所をアプリ側で固定しないことで、ローカル保存・iCloud同期・GitHub連携のどれにも対応しやすい

---

## 006 · Markdown 専用とし画像を対象外にする

**決定**：テキスト（Markdown）のみを対象とし、画像ファイルは管理対象外にする。

**理由**：
- 画像を含めるとストレージ・同期・表示の複雑さが大幅に増す
- テキスト特化にすることでシンプルで高速なアプリを実現する
- 画像が必要な用途は別ツールで対応できる

---

## 007 · プラグイン機能を持たない

**決定**：ユーザーが機能を追加・拡張する仕組み（プラグイン）は実装しない。

**理由**：
- プラグインシステムの設計・維持コストが高い
- 機能をアプリ本体に内蔵することで品質と一貫性を保つ
- 必要な機能はアプリ自体に追加していく方針

---

## 010 · UIフレームワークに React を採用する

**決定**：UIフレームワークとして React を採用する。

**理由**：
- AI駆動開発（Claude Code / Codex）との相性が最良。学習データが最も豊富
- Obsidianと同じフレームワークで、同様のユースケースでの実績がある
- TypeScriptとの相性が良く、大規模なUIでも管理しやすい

---

## 009 · メインエリアは常にエディタを表示し遷移しない

**決定**：メインエリア（右側の大きいエリア）は常にエディタを表示する。設定・検索・Gitなどの操作によってメインエリアが別のビューに切り替わることを禁止する。

**理由**：
- エディタへのアクセスを常に確保することがこのアプリの根幹
- 設定やGitなどはサイドバー内のビュー切り替えで完結させることで、作業の流れを途切れさせない
- ユーザーが「今どこにいるか」を見失わないシンプルなナビゲーション構造を保つ

---

## 008 · 配布を GitHub Releases で行う（当面）

**決定**：当面は GitHub Releases で配布する。Mac App Store・コード署名はリリース品質になった段階で検討。

**理由**：
- 開発初期は配布の手軽さを優先する
- Mac App Store の審査・サンドボックス制約は開発速度を下げる
- ある程度完成してから正式な配布方法に移行する

# architecture/decisions.md

アーキテクチャ上の重要な意思決定を記録するファイル。
「なぜその選択をしたか」の理由と背景を残す。設計を変更する際は必ずここを確認すること。

---

## 001 · プラットフォームを macOS に限定する

**決定**：macOS 専用アプリとして開発する。iOS は、macOS版を自分で継続利用し、外出先・モバイルで使いたい場面が明確になったら別アプリとして検討する。

**理由**：
- Markdown ノートアプリの主要ユーザーはデスクトップユーザー
- iOS と同時開発するとどちらも中途半端になるリスクがある
- まず macOS で自分が継続利用できる完成度まで育て、その後にモバイルで必要な体験を判断する

---

## 002 · フレームワークに Electron を採用する

**決定**：デスクトップアプリのフレームワークとして Electron を採用する。

**理由**：
- TypeScript / JavaScript の資産をそのまま使える
- macOS ネイティブアプリに近い体験を Web 技術で実現できる
- エディタエンジン（CodeMirror 6）・Git 実装（isomorphic-git）ともに Web 技術ベースで相性が良い

---

## 003 · エディタエンジンに CodeMirror 6 を採用する

**決定**：Markdown エディタの中核ライブラリとして CodeMirror 6 を採用する。

**理由**：
- Obsidian（同様のユースケース）が採用しており実績が豊富
- ライブプレビューとソースモードの切り替えが設計の根幹に組み込まれている
- TypeScript ネイティブで型安全な開発が可能
- `[[...]]` や `#tag` などのカスタム構文も拡張システムで実装しやすい

詳細は [tech/editor-engine.md](../tech/editor-engine.md) を参照。

---

## 004 · Git 実装に isomorphic-git を採用する

**決定**：Git 機能の実装ライブラリとして isomorphic-git を採用する。

**理由**：
- JavaScript のみで書かれており、ユーザーの Mac に Git がインストールされていなくても動作する
- 非エンジニアも使う想定のため、環境依存をゼロにすることが重要
- Electron との相性が良く、ネイティブコンパイルが不要

詳細は [tech/git-implementation.md](../tech/git-implementation.md) を参照。

---

## 005 · ワークスペースは任意のローカルフォルダを登録する

**決定**：ユーザーがMac上の任意のローカルフォルダをワークスペースとして登録する。iCloud Drive / OneDrive / Dropbox などのクラウド同期フォルダも、通常のローカルフォルダと同じように登録できる。

**理由**：
- 既存の執筆フォルダやGitHubリポジトリをそのまま開ける
- Obsidianと同じ方式で、ユーザーが自分のファイルの所在を把握しやすい
- クラウド同期を使いたい場合は、ユーザーがiCloud Drive / OneDrive / Dropbox などの同期フォルダを選べばよい
- 保存場所をアプリ側で固定しないことで、ローカル保存・クラウド同期フォルダ・GitHub連携のどれにも対応しやすい
- Relicは各クラウドサービスのAPI連携を持たず、OS上の通常のフォルダとして読み書きする

---

## 006 · Markdown 内の添付画像だけをサポートする

**決定**：Markdown本文に添付された画像のみサポートする。ライブプレビューでは画像として表示するが、画像ファイル単体の整理・タグ付け・検索・アルバム管理は行わない。

**理由**：
- 小説の挿絵や参考画像など、テキストを補助する画像が必要になる場面はある
- 完全非対応にすると、Markdownエディタとしての実用性が不自然に狭くなる
- 一方で画像管理アプリに広げると、Relicの中心であるMarkdownファイル管理がぼやける
- 画像は本文を補助する添付物として扱い、主役は `.md` ファイルのままにする

---

## 007 · プラグイン機能を持たない

**決定**：ユーザーが機能を追加・拡張する仕組み（プラグイン）は実装しない。将来も原則として持たない。

**理由**：
- プラグインシステムの設計・維持コストが高い
- 機能をアプリ本体に内蔵することで品質と一貫性を保つ
- 必要な機能はアプリ自体に追加していく方針
- サポート範囲を広げすぎず、Relicとして統一された体験を保つ

---

## 008 · まず自分用ビルドとして開発する

**決定**：最初の配布対象は自分用ビルドとする。第三者に配る段階でコード署名と自動アップデートを検討し、一般ユーザー向けに継続提供する価値が見えたらMac App Store配布も検討する。

**理由**：
- まず自分が使うために完成させることを優先する
- 開発初期から第三者配布の要件を背負うと、署名・更新導線・審査対応で開発が重くなる
- 自分で継続利用して価値と不満点が見えた後に、外部配布や収益化を判断する

---

## 009 · メインエリアは常にエディタを表示し遷移しない

**決定**：メインエリア（右側の大きいエリア）は常にエディタを表示する。設定・検索・Gitなどの操作によってメインエリアが別のビューに切り替わることを禁止する。

**理由**：
- エディタへのアクセスを常に確保することがこのアプリの根幹
- 設定やGitなどはサイドバー内のビュー切り替えで完結させることで、作業の流れを途切れさせない
- ユーザーが「今どこにいるか」を見失わないシンプルなナビゲーション構造を保つ

---

## 010 · UIフレームワークに React を採用する

**決定**：UIフレームワークとして React を採用する。

**理由**：
- AI駆動開発（Claude Code / Codex）との相性が最良。学習データが最も豊富
- Obsidianと同じフレームワークで、同様のユースケースでの実績がある
- TypeScriptとの相性が良く、大規模なUIでも管理しやすい

---

## 011 · パッケージマネージャに pnpm を採用する

**決定**：パッケージマネージャとして pnpm を採用する。

**理由**：
- インストールが高速でディスク使用量が少ない（ライブラリをシステム全体で共有する仕組みのため）
- Electron × React 構成との相性が良く、現在のエコシステムで広く使われている
- npm と比べても運用上のデメリットがほぼない

---

## 012 · ビルド・配布ツールに Electron Forge を採用する

**決定**：Electronアプリのビルド・配布ツールとして Electron Forge を採用する。

**理由**：
- Electron 公式が推奨するツールであり、現在のスタンダード
- プロジェクト作成・ビルド・配布を一括で扱えるため設定が簡潔
- pnpm との組み合わせも公式にサポートされている

---

## 013 · テストフレームワークに Vitest を採用する

**決定**：テストフレームワークとして Vitest を採用する。

**理由**：
- 設定が簡潔で、TypeScript プロジェクトとの相性が非常に良い
- Jest に比べて実行が高速
- Vite ベースの現代的なツールチェーンと親和性が高い
- 将来的に E2E テストが必要になった場合は Playwright を追加すればよい

---

## 014 · GitHub連携は手動確認とローカル履歴を重視する

**決定**：GitHub認証はブラウザOAuthを採用し、Personal Access Token貼り付け方式はサポートしない。GitHubに接続していない場合でもローカルGitリポジトリとして履歴管理できる。自動プル・自動プッシュはデフォルトOFFとし、手動同期時は必ず差分確認画面を表示する。

**理由**：
- 非エンジニアも使う想定のため、GitHub接続は一般的なアプリに近いログイン体験にする
- ローカルファイル中心のアプリとして、GitHubアカウントがなくても履歴管理できることを重視する
- 意図しない変更が勝手にGitHubへ送られる不安を避ける
- 同期やブランチ切り替えなど取り消しづらい操作では、ユーザーが内容を確認してから実行できるようにする

---

## 015 · 料金モデルとAI機能は現時点では設計対象外にする

**決定**：料金モデルは現時点では設計対象外にする。まず自分用に作って使い、良ければ収益化を検討する。AI機能も現時点では入れず、自分で使う中で必要性が見えたら再検討する。

**理由**：
- 無料固定でも有料前提でもなく、まず自分用の道具として成立させる段階である
- 料金モデルを早く入れると、機能トグルや仕様判断が収益化前提に引っ張られる
- AI機能は便利だが、API料金・プライバシー・プロダクトの芯への影響が大きい
- まずはMarkdownエディタとしての基本体験を固める

---

## 016 · React状態管理にZustandを限定採用する

**決定**：React状態管理にはZustandを採用する。ただし、タブ・ペイン・サイドバー・右パネル・アクティブワークスペース・UI設定反映などの横断的UI状態に限定する。フォーム入力・メニュー開閉・モーダル内の一時状態はReactローカルstateで扱い、ファイル内容・Git状態・検索インデックス・LinkIndex / TagIndex・設定の永続化はZustandに入れない。

**理由**：
- Relicはタブ・分割表示・サイドバー・右パネル・ワークスペース復元など、初期段階から横断的UI状態の複雑さが見込まれる
- React標準だけで始めるより、予測できる複雑さに先に置き場所を用意できる
- 一方で、永続データやファイル内容までstoreに入れると責務が曖昧になるため、用途を限定する

---

## 017 · 設定と認証情報の保存場所を分ける

**決定**：アプリ設定とワークスペース設定はElectronの `userData` 配下にJSONで保存し、`app/src/main/settings/` の自前設定サービスで読み書きする。GitHub OAuthで取得した認証情報はmacOS Keychainに保存し、設定JSONには保存しない。

**理由**：
- 設定構造はRelic固有であり、自前サービスのほうが仕様との対応関係を保ちやすい
- ユーザーのMarkdownワークスペース内にRelic専用設定ファイルを置かず、ノートフォルダをきれいに保てる
- GitHub認証情報は通常の設定値より機密性が高いため、macOS標準の安全な保管場所を使う

---

## 018 · Relic用の必須フォルダをワークスペース準備時に作る

**決定**：添付画像フォルダは `attachments/`、テンプレートフォルダは `templates/` とし、ワークスペース作成時または既存フォルダ登録時に自動作成する。ユーザーにフォルダ名や作成有無は選ばせない。

**理由**：
- Relicは自由度そのものを売りにするアプリではなく、必要な構造をアプリ側で整える方針に合っている
- 画像貼り付け時に初めてフォルダ作成とファイル保存が同時に起きると、失敗時に中途半端な状態になりやすい
- 必須フォルダが最初から存在するほうが、実装とユーザー体験の両方で挙動が安定する

## 計画・決定

- 年表バーの幅は、文字を読むために広げず、単年・期間の実際の年数幅を優先する。
- 年表内の数値は、バーからはみ出しても読めることを優先する。
- フロントマタープロパティは、固定・カスタムを問わず、ユーザーへ見せる書き方を `プロパティ名: [値]` の1行配列に揃える。
- フロントマター設定画面のメリハリは、有彩色ではなくグレースケールの背景濃淡・境界線・文字ウェイト・等幅コード枠で付ける。
- 固定日付フィールド名は、単日・期間のどちらにも使いやすい汎用名として `date` を採用する。
- `date` は時刻を扱わず、年月日だけを扱う。
- Git機能は、コミット・履歴・差分・GitHub同期を中心に残し、ブランチ管理とGitタグ管理は削る。
- GitHub同期は内部的には現在のGit履歴を対象にするが、通常UIではブランチ管理を前面に出さない。
- 年表はユーザーが任意に増やすものではなく、`chronicle` と `date` の2個固定とする。
- `chronicle` / `date` の年表は、該当プロパティを持つファイルから候補を作り、年表タブへドラッグ&ドロップしたものだけを表示対象にする。
- 年表タブ上の削除はファイル削除ではなく、その年表の表示対象から外す操作として扱う。
- ユーザー向けの入口名は「年表」より広い概念として「チャート」/ `Charts` にするが、Markdown互換性のため内部キーや固定プロパティ名 `chronicle` は変更しない。
- チャートサイドバーの `chronicle` / `date` 切り替えは、相互排他的な表示切替ではあるが、意味としてトグルを強調する必要がないため通常ボタンとして扱う。
- チャートモードのファイル候補表示は、ファイルモードの見た目に揃える。ただしファイルモード本体には影響させず、チャートモード内だけで使用中ファイルの色付き表示を行う。
- 検証用ワークスペースはRelic開発リポジトリ内に作らず、`/private/tmp` 配下に作成する。
- P17は、ユーザーの違和感・削りたい機能を起点に、フロントマター、Git、チャート、専用フォルダ扱いを整理した段階で完了とする。
- 次フェーズへは自動遷移せず、いったん `P-hold` で次の開発フェーズを判断する。

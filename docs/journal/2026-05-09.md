- ユーザー判断により、次フェーズは `P14: GitHub連携の安全導入` とする。
- P14は、ユーザー操作を前提に、セキュリティ最優先でGitHub OAuth / clone / remote接続 / push / pull / conflict解決を段階導入する。
- P14の最初の作業として、リポジトリへ秘密情報を上げないためのignore強化とpre-commit / pre-push hookを導入する。
- GitHub公式ドキュメントを基準に、OAuth Appより細かい権限・短命トークンを扱えるGitHub Appを第一候補として検討する。
- デスクトップアプリの認証では、client secretをリポジトリやアプリ設定に置かないため、Device Flowを第一候補として検討する。
- GitHub公式のsecret scanning / push protectionには検出範囲上の限界があるため、GitHub側の設定確認とローカルhookの二重防衛にする。
- Relic本体は特定のGitHubアカウント・リポジトリ・Client IDに依存させず、開発中のGitHub App / OAuth設定は検証環境として切り分ける。
- 自分用ビルドであっても、GitHub連携の設計は将来ほかのユーザーが各自のGitHub設定で使える普遍的な構造にする。
- 既存のGitHub認証コードは `RELIC_GITHUB_CLIENT_SECRET` 前提のWeb OAuth flowだったため、client secretを使わないDevice Flowへ変更する。
- Device Flowではaccess token / device codeをRendererへ渡さず、Main process内で処理し、access tokenはKeychainに保存する。
- OAuth scopeの既定値は空にし、`repo` のような広い権限は明示設定した場合だけ要求する。
- GitHub Appの最小権限案は `Contents: Read and write` のみにし、Issues / Pull requests / Actions / Administration / Secrets / Webhooks / Workflows は要求しない。
- 検証用GitHub App作成案を整理し、Relic本体へ渡す値は `RELIC_GITHUB_CLIENT_ID` のみにする。
- 期限付きトークンは `tokenExpiresAt` を保存し、期限切れならKeychainから削除して未接続扱いにする。
- Git remote接続は、既存originを別URLで上書きしないようにする。
- Git UIの見直しとして、通常UIから作者名・メール入力を削除し、コミット作成とメモ付きタグ作成に必要なauthor / taggerはGitHub接続情報からMain側で内部決定する方針にした。
- UI文言ではコミットに「ローカル」などの冗長な修飾を付けず、`コミット` と表記する方針にした。
- 開発中の確認では `app/out/` 配下のパッケージ版をテスト対象にせず、テスト用ワークスペースをRelic開発リポジトリ内に作らないルールを `AI.md` に明記した。

## 構想・アイデア

- GitHub機能の本格導入前に、OAuth / clone / push / pull / conflict解決を実アプリ操作で通し確認する。
- GitHub認証情報は設定JSONやワークスペース内ファイルに保存せず、OS標準の安全な保管先を使う。
- GitHub OAuthのclient secret、access token、Keychain export、`.env` はGit管理対象にしない。
- ローカルhookではGitHub token、client secret、private key、HTTP認証ヘッダー、DB接続文字列、npm tokenなど、GitHub公式のsecret scanning対象に含まれる既知カテゴリを明示的に止める。
- 次回はP15として、GitHub機能の追加実装には進まず、UI/UXの現状診断から始める。
- 次回はP16として、ユーザーが具体的に挙げたUI/UXの違和感から小さく修正する。

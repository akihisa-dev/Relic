# AGENTS.md

このファイルはすべてのAIエージェントが参照する**唯一のルールファイル**です。
Claude Code・Copilot・Codex・Gemini・その他すべてのAIはセッション開始時にこのファイルを読んでください。

> **ルールはここにだけ書く。別のAI向け入口文書を作らない。**

---

## セッション開始時の手順

AIはセッション開始時に必ず以下の順序で確認してください：

1. この `AGENTS.md` で、自分に課せられた行動ルールと禁止事項を確認する
2. `docs/project/overview.md` を読み、Relicのプロジェクト概要を把握する
3. `docs/development/phases.md` で、現在フェーズと `checklist` に書かれた現在フェーズ正本だけを確認する
4. 現在フェーズ正本を読み、その文書の「読み込みルール」に従う
5. 正本を読んでも判断できないことだけ、ユーザーに確認する

> 各段階では、その段階の文書が扱う範囲だけを認識すること。下位文書で扱うべきプロジェクト固有の内容を、上位文書から推測しないこと。
> `docs/development/phases.md` には、フェーズ別の注意、作業方針、読み込みルール、進捗、完了条件を書き足さないこと。これらは現在フェーズ正本に書くこと。
> 文書群の分類と現行正本は `docs/INDEX.md` を参照すること。

### 最初の応答

セッション開始直後の応答では、現在フェーズを理解した状態で話すこと。

- 現在フェーズ名を明示する
- 現在フェーズの正本文書を読んだ前提で、次に進める具体的な作業範囲を提示する
- 汎用的な「今日は何を進めましょう？」だけで終わらない
- ユーザーに確認する場合も、現在フェーズを前提にした具体的な確認にする

例:

> 現在は `P1` 暦設定の実装フェーズです。正本は `docs/development/phases/P1.md` なので、対象範囲であるチャートビューとフロントマターに限り、暦設定の詳細仕様を1問ずつ確認してから小さな単位で実装します。

## 作業区切り時の文書更新確認

AIは実装・検証・修正・調査の区切りごとに、完了報告の前に必ず以下を確認してください。

1. `docs/development/phases.md` で現在フェーズを確認する
2. 現在フェーズの正本文書を確認する
3. 今回の作業内容に対して、更新すべき文書があるか判断する
4. 更新が必要な場合は、ユーザーが明示した編集範囲、または現在フェーズの正本文書に限って更新する
5. 関連ファイルの更新が必要だが、ユーザーの明示範囲・現在フェーズの正本文書を越える場合は、理由・対象ファイル・変更内容を説明してユーザーに確認する
6. `docs/development/phases.md` は、下記「セッション終了時の手順」と「編集範囲と許可のルール」に従い、勝手に更新しない
7. 完了報告には「更新した文書」または「更新不要と判断した文書」を含める

現在フェーズの正本文書には、そのフェーズ固有のルール、制約、作業記録を書く。
作業記録は `YYYY-MM-DD HH:mm JST` 形式の日時付きで、実施したことだけを簡潔に書く。
方向性、確認、残りを項目として分けて書かない。
必要な確認結果や未確認事項がある場合も、実施内容の一部として短くまとめる。
現在フェーズの正本文書には、次回作業時に判断できる最小限の実施内容だけを残す。

## 正本更新時のコミットルール

現在フェーズ正本に作業記録を書く場合は、その更新を行ったタイミングでコミットする。
AIが実装・修正・検証・調査・文書整理などの作業を行った場合は、完了報告前に必要な文書更新を確認し、更新が必要な文書を反映したうえでコミットする。

- 正本を更新した時点で、ワークツリーに残っているすべての差分をまとめてステージングしてコミットする
- 作業内容に対して更新すべき文書がある場合は、文書更新を同じ作業単位に含め、未反映のまま完了報告しない
- フェーズ正本に作業記録を書く場合は、`app/package.json` のPATCHバージョンを1つ上げ、その変更も同じコミットに含める
- AIルールの追加・修正に伴う現在フェーズ正本更新、バージョン更新、コミットは、そのAIルール追加・修正作業に含める
- MAJORとMINORは、ユーザーが明示的に指示した場合だけ変更する
- 「自分が直接編集したファイルだけ」を選別してコミットしてはならない
- コミットには、その時点で残っているコード、テスト、関連文書、フェーズ正本の更新をまとめて含める
- 正本更新だけを未コミットのまま残し、次の作業へ進まない
- コミット前に、必要な確認を終える
- pushはユーザーが明示的に指示した場合だけ実行する
- pushを行わない場合は、完了報告に未pushであることを明記する
- AIは、ユーザー指示なしに既存コミットを分割し直したり、履歴を書き換えたりしない
- コミットメッセージは日本語で、更新の目的と内容が分かるように書く
- ユーザーが明示的に「コミットしない」「まだコミットしない」と指示した場合は、その指示を優先し、完了報告に未コミット理由を明記する

## GitHub Issue対応時の完了処理

GitHub Issueに対応した場合は、対応内容、確認したテスト・検証、関連コミットをIssueへコメントする。
コメント後、対応が完了しているIssueは `completed` として閉じる。
Issueを閉じない理由がある場合は、その理由をIssueコメントまたは完了報告に明記する。

## セッション終了時の手順

AIはセッション終了時に必ず以下を行ってください：

1. ユーザーが `終了`、`ここまで`、`区切り` など開発の区切り・終了を明示した場合は、終了返答より先に現在フェーズ正本の作業記録更新が必要か判断する
2. 更新が必要な場合は、終了返答より先に現在フェーズ正本へ作業記録を書く
3. それ以外のタイミングでは、`docs/development/phases.md` はユーザーから明示的に指示された場合のみ更新する
4. 現在フェーズ正本の作業記録には、その作業単位で実施したことだけを日時付きで簡潔に記録する

### フェーズ完了時の扱い

- あるフェーズが完了しても、次の開発フェーズへ自動的に進めてはならない
- ユーザーが次フェーズへの移行を明示した場合だけ、`docs/development/phases.md` の `current` を次フェーズに変更する
- 次フェーズが未決定の場合は、完了内容・残り・次候補を整理し、ユーザーが次フェーズを明示するまで `current` を変更しない

## 仕様書の作成方針

仕様書（`docs/features/`）を作成・更新する際は、以下の手順で進めてください：

1. 利用可能な入力依頼ツール、または通常の質問で、ユーザーに1問ずつヒアリングする
2. ユーザーの回答をもとに仕様を決定する
3. ユーザーの承認を得てからファイルに書き込む

ユーザーは非エンジニアのため、質問は具体的な例を交えてわかりやすく聞いてください。

---

## ユーザー要望の扱い

ユーザーが「〜したい」「〜をやりたい」「こういう方向にしたい」と言っただけで、AIは仕様書・設計書・フェーズ・チェックリスト・デザイン方針を勝手に作成してはならない。

この場合は、まずユーザーにヒアリングする。

- 何が嫌か、何を変えたいかを確認する
- 変更してよい範囲を確認する
- 文書化するか、実装するか、調査だけにするかを確認する
- AIが提案する場合も、提案として出し、ユーザーが採用したものだけを正本化・実装する

チェックリスト作成やフェーズ作成自体も作業であり、ユーザーの確認負担を増やす。
そのため、ユーザーが明示的に求めた場合、またはヒアリングの結果として合意した場合だけ作成する。

### 指示解釈の確認

ユーザーから作業指示を受けた場合、AIは実装・編集・削除・整理などの作業に入る前に、指示をどう解釈したかをユーザーへ明示する。
指示解釈は長い段落で書かず、対象・内容・除外範囲・進め方がすぐ分かる箇条書きで提示する。
ユーザーがその解釈で正しいと判断してから作業に入る。

### 親エージェントとサブエージェントの役割

親エージェントはマネージャーとして、計画、作業指示、品質チェック、統合、コミットを担当する。
実装作業は、可能な範囲で `GPT5.5Codex-spark`、または環境で利用可能なCodex Spark系サブエージェントに任せる。

親エージェントがユーザーとの対話窓口を一貫して担い、サブエージェントは対外的な説明、質問、報告を直接行わない。
サブエージェントは実装・調査などの実作業を担当し、必要な確認事項や進捗は親エージェントへ返して親エージェントがユーザーへ共有する。

親エージェントがサブエージェントへ渡す作業は、対象、目的、編集範囲、除外範囲、完了条件、確認方法が明確な小さな単位にする。
サブエージェントは、渡された編集範囲と現在フェーズの範囲を越えて作業してはならない。
範囲外の変更や追加確認が必要だと判断した場合は、勝手に進めず、親エージェントへ戻す。

親エージェントは、サブエージェントの作業結果をそのまま採用せず、差分、動作、文書更新の要否、ユーザー指示との一致を確認してから統合する。
サブエージェントが実施した作業の最終責任は、親エージェントが持つ。

### ユーザー判断を求めるときの選択肢提示

AIがヒアリング、提案、方針確認、技術選定、作業範囲確認などでユーザーに判断を求める場合は、必ず2個以上の選択肢を提示する。

- 各選択肢には番号を振る
- どれを推奨するかを明示する
- 推奨理由を非エンジニアにも分かる言葉で添える
- ユーザーは番号を明示して指示する前提で待つ
- 選択肢を出さずに「どうしますか？」だけで判断を求めてはならない

ただし、`/goal` モード中に選択肢を提示した場合は、ユーザーが明示的に別の番号を選ばない限り、AIが「推奨」と示した選択肢をユーザーが選んだものとして扱う。
この例外は `/goal` モード中だけに適用し、通常の作業ではユーザーの明示判断を待つ。

## 技術選定・意思決定のルール

- **決してユーザーへの説明なく独自判断によって選定しない**
- 技術的な選択肢が生じた場合は、各選択肢を非エンジニアにもわかる言葉で説明し、ユーザーに決定を委ねること
- AIが推薦を示すことはOK。ただし最終決定は必ずユーザーが行う
- 「技術的なことはわからない」と言われた場合も、わかりやすく言い換えて説明し直すこと。勝手に決めてはならない

## 診断ツール利用ルール

React Doctorなどの診断ツールは、Relicの品質確認として定期的に利用してよい。
ただし、診断ツールの点数や指摘を盲信してはならない。

- React Doctorは、大きめの実装後、リリース前、依存更新後、または月1回程度の定期点検として実行する
- 指摘は「実害があるもの」「Relicの性質と合わないもの」「大規模な設計変更として別作業にすべきもの」に分けて判断する
- 実害がある指摘は、現在フェーズとユーザーの指示範囲に収まる場合に修正する
- Electron、CodeMirror、ローカルファイル操作、順序が重要な処理など、Relicの性質と合わない指摘は、理由を説明したうえで設定除外または保留にしてよい
- 設定除外を追加・変更する場合は、除外理由をユーザーに説明し、必要に応じて現在フェーズ正本にも記録する
- 点数を上げるためだけに、保存、ファイル操作、エディタ本体、状態管理などの大規模な設計変更を勝手に行ってはならない

## 未決定事項の管理ルール

- 仕様書・設計書・UI文書に「未決定事項」セクションを書いてはならない
- 未決定の問いはすべて `docs/development/questions.md` に一元管理する
- 各問いには `関連:` フィールドで由来となる文書を明記する
- `questions.md` は未決定事項が発生したとき、または `phases.md` が読むよう指定したときだけ読む

## 編集範囲と許可のルール

- ユーザーが明示した対象ファイル・対象範囲以外を編集してはならない
- 関連ファイルの更新が必要だと判断した場合も、勝手に編集せず、理由・対象ファイル・変更内容を説明してからユーザーの許可を得る
- 変更の取り消し・巻き戻しも、ユーザーが明示的に指示するまで実行してはならない
- 「整合性のため」「慣例上必要」「セッション終了時の手順」などを理由に、ユーザーの直近指示を越えて作業してはならない
- UI変更では、ユーザーが指定した画面・場所・操作対象だけを変更する。別の場所に同じ機能の入口を追加する、表示要素・ボタン・説明・装飾を足す、操作方法を増やすなどの範囲拡張は、どれほど有用に見えても事前に確認する
- ユーザーが「Aの話をしている」と対象を訂正した場合、A以外へ加えた変更は撤回し、以後その会話ではA以外を触らない。提案がある場合も実装せず、提案として分けて伝える
- ただし、ユーザーが開発の区切り・終了を明示した場合は、必要に応じて現在フェーズ正本へ作業記録を書く
- 上記以外では、`docs/development/phases.md` は、ユーザーが明示的に指示した場合のみ更新してよい
- 開発位置とフェーズ順は `docs/development/phases.md` を正とする
- 実装時も、ユーザーが明示した対象・現在のフェーズ範囲を越える変更は行わない
- ユーザーから明示的に指示がない限り、ブランチは切らずにメインブランチで作業する

### Relic固有の検証禁止事項

- `app/out/` 配下のパッケージ版アプリは、ユーザーが明示的に配布ビルド確認を求めた場合以外、開発中のテスト・確認対象にしてはならない
- 実アプリ検証は、開発版であることを確認できる起動経路だけを対象にする
- テスト用ワークスペース・検証用リポジトリ・一時データをRelic開発リポジトリ内に作ってはならない
- 実アプリ検証でユーザーのワークスペースや外部サービスに触る必要がある場合は、事前に対象・操作内容・戻し方を説明し、ユーザーの明示許可を得る
- ユーザーが「触るな」「話しかけるな」と指示した場合は、必要な修正・検証が完了するまで途中報告を挟まない

# AGENTS.md

この文書は、RelicリポジトリでAIエージェントが常に守る判断規則を定義する。
製品仕様、設計、開発手順の詳細はここへ重複させず、`docs/INDEX.md` から必要な正本だけを確認する。

## 優先順位

- ユーザーが求める結果と明示範囲を最優先にする
- 品質、正確性、ユーザーデータ保護を満たしたうえで、読み込み量、ツール呼び出し、所要時間を抑える
- 現行文書と実装は現在の状態を確認する根拠として扱う。ユーザーが変更を指示した場合は、変更後の仕様と実装を一致させる
- 確認済みの事実、推論、不明点を区別し、誤った前提には根拠を示して修正を提案する

## 対話

- 丁寧な日本語で、結論から説明する
- 専門用語は必要な場合だけ使い、結果や影響を一般の人にも分かる言葉で補足する
- 複数段階のツール作業では開始前に対象と最初の確認を短く伝え、途中報告は主要段階または方針変更時だけ行う
- ユーザーが「途中報告は不要」と指定した場合は報告だけを省略する。「触らないで」「変更しないで」は書き込み停止として扱う

## 情報源と文書の確認

- 最初から全文書を読まず、対象コード、`rg`、`docs/INDEX.md` の該当する作業別ルートから始める
- 製品方針の判断では `docs/project/overview.md`、機能では `docs/features/`、UIでは `docs/design/`、設計では `docs/engineering/` の関連文書だけを読む
- 実装、検証、依存関係、バージョン、リリースの判断では `docs/development.md` の関連節だけを読む
- 上位文書から下位文書の詳細を推測しない。正本と関連コードでも判断できない重要事項だけをユーザーへ確認する
- 外部情報、ファイル、コード、Issue、Pull Request、文書に基づく重要な判断では、最終報告に主要な参照対象を明示する
- 推測で検索範囲を広げ続けず、必要な根拠がそろった時点で作業へ進む

### GitHub URL

- ユーザーがGitHub URLを提示した場合は、回答前にGitHub連携ツールでそのURLの対象を確認し、一次情報源として扱う
- Web検索、リポジトリ名検索、推測で代替しない。連携ツールで確認できない場合は、確認できなかった事実と理由を明示する

## 指示と権限

- 質問、説明、調査、相談、レビュー、診断、計画では、必要な資料を確認して結果を報告する。編集や外部変更は行わない
- 変更、実装、修正、更新、削除、整理、コミットの明示指示では、指定範囲内の調査、ローカル編集、必要な文書更新、通常の技術判断、非破壊的な検証、既定のコミットまで進める
- ユーザーが「計画だけ」「編集しない」「コミットしない」などと限定した場合は、その限定を優先する
- 希望や方向性だけが示された場合は、会話と正本から期待結果を特定する。結果を特定できない場合だけ、最小限の質問をする
- 小さく可逆な実装判断は、既存仕様、設計、依存関係、一般的な実装規約に基づいてAIが行い、結果を報告する

### 作業を止めて確認する条件

次のいずれかに該当し、ユーザーが具体的に許可していない場合だけ作業を止める。

- 正本と関連コードを確認しても、期待する結果を特定できない
- 成立する複数案で、ユーザーに見える結果、互換性、アーキテクチャ、セキュリティ、運用負担が大きく変わる
- 明示された対象範囲を実質的に越える必要がある
- 指示が確認済みの仕様、プロジェクト方針、セキュリティ条件と矛盾する
- データ消失、Git履歴の書き換え、秘密情報、権限変更、外部送信、公開、購入、費用発生、容易に戻せない操作が必要になる

確認時は、確認済みの事実、結果の違い、影響範囲、推奨案と理由だけを示す。
ユーザーが対象と高影響操作を具体的に指示済みの場合は、同じ内容を再確認しない。作業中に新しい影響が判明した場合だけ確認する。

## 作業範囲と変更保護

- ユーザーがファイルや画面を限定した場合は、その範囲外を編集しない
- 目的や機能が指定された場合は、達成に直接必要なコード、テスト、正本文書、生成物、バージョン更新を作業範囲に含めてよい
- 「このファイルだけ」など明示的な限定と必須の関連更新が両立しない場合は、範囲を勝手に広げず理由を説明する
- 無関係な差分やユーザーの変更を編集、削除、コミットしない
- ユーザーの変更やGit履歴を、明示指示なしに巻き戻さない。今回の作業でAIが加えた未コミット変更は、検証結果に応じて修正または取り除いてよい
- ユーザーの明示指示がない限り、新しいブランチを作らない
- 提案だけの内容は実装せず、実施した変更と分けて報告する

## Relic固有の不変条件

- Markdownファイルをユーザーデータの正本として扱い、Relic専用形式への無断変換や不可視メタデータの埋め込みを行わない
- ファイル操作はワークスペース境界と既存の安全条件を守り、実ユーザーのワークスペースをテストに使わない
- Relicの仕様、設計、UI文言、コード、コメント、テスト名、コミット、Issue、Pull Request、作業報告へ、比較・模倣対象として他製品名を残さない
- 外部情報の出典として固有名詞が必要な場合は参照元としてだけ明示し、Relicの仕様や実装の表現へ持ち込まない
- 秘密情報、個人情報、認証情報、内部URL、ローカル環境固有値を、コード、文書、ログ、テスト、コミットへ含めない

## 検証と完了条件

- 変更内容のリスクと影響範囲に応じ、対象テスト、型チェック、ビルド、文書整合、差分確認から必要なものを実行する
- 実アプリの操作を伴うテストは、その作業中に起動した開発版だけを対象とする。既存のRelicウインドウ、配布版、ビルド済みアプリは認識・起動・操作せず、通常の実テストのためにパッケージや配布ビルドを生成しない
- 変更した振る舞いと失敗時の影響が大きい処理を優先し、単に確認回数を増やすための重複検証は行わない
- ツールの成功表示だけで完了とせず、差分と重要な生成結果を確認する
- 検証を実行できない場合は、理由と代替確認を報告する
- 完了報告には、変更結果、更新した正本、実施した検証、未実施理由、主要な参照対象を簡潔に含める

- pushと公開は、ユーザーが明示的に指示した場合だけ行う

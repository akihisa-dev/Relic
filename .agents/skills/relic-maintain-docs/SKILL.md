# Relic Documentation Maintenance

## 更新範囲を決める

1. 文書監査、説明、レビューだけの依頼では編集しない。作成、更新、整理が明示されている場合だけ変更する。
2. `git status --short` と `docs/INDEX.md` を最初に確認し、無関係な文書差分を保護する。
3. ユーザー依頼、現行コード、package scripts、workflowから、文書へ反映する確認済み事実を特定する。
4. コミット履歴や古い実装だけを現在の仕様や運用の根拠にしない。
5. 正本と実装が食い違う場合は、今回の変更目的と確認済み事実から更新対象を決める。

## 正本を選ぶ

1. AI行動規則は `AGENTS.md`、目的と対象ユーザーは `docs/project/overview.md` へ置く。
2. ユーザーから見える振る舞いは `docs/features/`、画面とデザインは `docs/design/DESIGN.md` へ置く。
3. プロセス、データ、技術判断は `docs/engineering/`、開発運用は `docs/development.md` へ置く。
4. 用語、表示名、実装識別子、参照先の対応は `docs/project/terms.md` へ置く。
5. 対外説明と利用・開発案内はREADME、参加手順はCONTRIBUTING、セキュリティ方針はSECURITY、行動規範はCODE_OF_CONDUCT、ライセンス本文はLICENSE、公開前確認はRelease Checklistへ置く。
6. 同じ判断を複数文書へ複製せず、他文書からは正本へリンクする。
7. 新しい文書を作る前に、既存正本への追記で役割を満たせないことを確認する。

## 内容を更新する

1. プロジェクト文書を日本語で書き、READMEの機能、コマンド、対応OS、配布説明を変える場合は英語部と日本語部を同期する。
2. 現在の仕様、設計、運用判断だけを残し、作業履歴や一時的な調査記録を蓄積しない。
3. 未確定事項を確定仕様として書かず、仕様・設計・UI文書に未決定事項セクションを作らない。
4. 専門用語は必要な範囲に絞り、一般の利用者にも結果と影響が分かる説明を添える。
5. コマンド、version、パス、成果物名は `app/package.json`、実装、workflowの現行値と照合する。
6. 標準Markdownの相対リンクを使い、リポジトリ外を指すローカル絶対リンクを作らない。
7. 文書名とフォルダ名を役割が分かる一般的な名前にする。
8. 文書の追加、移動、削除では `docs/INDEX.md` の作業別導線と正本文書カタログを更新する。
9. 新しい正本文書をカタログへ1回だけ掲載し、削除済み文書や重複entryを残さない。
10. `pnpm docs:tree` の出力を文書へ保存せず、必要な時だけ標準出力で確認する。
11. README画像が現行UIを説明できなくなっても、ユーザーが実アプリのスクリーンショット取得を明示的に指示した場合だけ `$relic-change-ui` の実画面手順で更新する。
12. 他製品名、秘密情報、内部URL、ローカル絶対パスを仕様や作業例へ持ち込まない。

## 検証する

1. 更新した説明を対象コード、仕様、package scripts、workflowと行単位で照合する。
2. `rg` で旧用語、旧パス、旧コマンド、移動前リンクの残存を必要な範囲で確認する。
3. `app/` で `pnpm docs:index:check` を実行し、全tracked Markdownの相対リンクと見出しanchor、Git追跡、正本文書カタログの漏れ・重複を確認する。
4. 新規文書が未追跡の間は索引検査を完全には通せないため、その理由を報告する。stage後の再検査はコミットが依頼されている場合だけ `$relic-commit` に委ねる。
5. リポジトリルートで `git diff --check` と全差分を確認する。
6. 文書だけの変更では製品テストを機械的に実行せず、文書整合と参照切れを重点確認する。
7. コード変更を伴う場合は、対応する実装スキルが求めるテスト結果も完了条件へ含める。
8. 文書に検証済みでない動作、OS、成果物、将来計画を書かない。

## 完了する

更新した正本、同期した対外説明、索引変更、参照した実装、検証結果、未確認事項を報告する。コミットする場合は `$relic-commit` に従う。

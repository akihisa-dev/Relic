# Skill監査の評価ケース

<!-- skill-audit: ignore-inline-paths -->

## 使い方

各caseは独立した一時fixtureまたは同等の生データで評価する。期待判定をテスト役へ先に渡さず、出力後に照合する。自動抽出が候補を出すことだけでなく、監査レポートが根拠、影響、変更種別、不確実性を正しく示すことを確認する。

## 1. ほぼ同じ責務のSkillが2つ存在する

- 入力: `change-note-links` と `fix-note-links` が同じ依頼、入力、出力、検証を記述し、優先境界がない。
- 期待判定: `responsibility-overlap`。実際の誤routingを示せる場合は `high`、候補抽出だけなら `medium` 以下に留める。
- 期待変更: 固有知識を比較したうえで `merge`、または境界を分ける `revise`。即時 `delete` はしない。

## 2. descriptionが抽象的すぎるSkillが存在する

- 入力: descriptionが「プロジェクトをいい感じに改善する」のみで、対象と依頼例がない。
- 期待判定: `ambiguous-description` と、必要に応じて `trigger-too-broad`。本文の明確な条件はroutingを救済しない。
- 期待変更: 何をするか、具体的な発火条件、主要な除外をfrontmatterへ移す `revise`。

## 3. 1つのSkillに複数の独立した目的がある

- 入力: 1 Skillが画像生成、依存更新、リリース公開を共通結果なしに扱う。
- 期待判定: `oversized-scope`。行数ではなく、独立した入力、出力、権限、検証を根拠にする。
- 期待変更: 共通入口が必要ならroutingだけ残し、独立責務を `split`。権限差が大きい場合は `high`。

## 4. 細かすぎるSkillが多数存在する

- 入力: 同じ文書更新で「見出しを読む」「リンクを確認」「保存する」だけを担うSkillが分かれ、常に全件が必要になる。
- 期待判定: `over-fragmented`。単独で利用者向け結果を作れず、固有知識がないことを根拠にする。
- 期待変更: 一貫したworkflowへ `merge`。安全境界を持つ独立Skillまで統合しない。

## 5. 古いコマンドを参照するSkillが存在する

- 入力: 必須手順が削除済みのpackage script `docs:index:update` を実行する。
- 期待判定: `stale-command` と状態 `stale`。現行 `package.json` と履歴上の移行を根拠にする。
- 期待変更: 現行コマンドと意味を確認して `revise`。単なる名前置換で副作用を推測しない。

## 6. 存在しないファイルを参照するSkillが存在する

- 入力: 必読の `docs/missing.md` がHEADにも生成手順にも存在しない。
- 期待判定: `missing-path`。必須なら原則 `high`、任意例なら影響に応じて下げる。
- 期待変更: 正しい正本へ `revise`、または本当に必要なら別変更として作成を提案する。globやfixtureを誤検知しない。

## 7. 2つのSkillの指示が矛盾している

- 入力: 同じ依頼とファイルに対し、一方は「初回から自動削除」、他方は「削除前に必ず承認」と指示する。
- 期待判定: `instruction-conflict`。scopeと依頼モードが同一であることを確認する。データ消失につながる場合は `critical`。
- 期待変更: 安全な正本へ揃える `revise`。どちらかを語調だけで優先しない。

## 8. 過去コミットの単発事例から作られたSkillが存在する

- 入力: 1回だけ発生した特定OS・特定versionの回避策を一般的な必須Skillにしている。
- 期待判定: `history-overgeneralization`。後続反復と現行HEADの必要性がなければ確定知識にしない。
- 期待変更: `hold`、または一般的な診断Skill内の条件付き参考へ `merge`。単発だけを理由に新設維持しない。

## 9. 現在のHEADでは不要になったSkillが存在する

- 入力: 対象機能と全参照が削除され、後継workflowへ移行済みだがSkillだけが残る。
- 期待判定: `head-mismatch` と状態 `stale`。履歴上の存在だけで維持しない。
- 期待変更: 移行先と固有知識を確認後に `deprecate`、最終的に `delete`。参照が残る間は削除しない。

## 10. 必要な領域にSkillが存在しない

- 入力: 複数の独立コミットで同じ壊れやすい調査・修正・検証が繰り返され、既存Skillはその安全条件を扱わない。
- 期待判定: `coverage-gap`。反復、固有知識、既存Skillで埋まらない境界を示す。
- 期待変更: `create`。単発または現在性未確認なら `hold` にする。

## 11. 整備用Skill自身に問題がある

- 入力: 整備用Skillのdescriptionが全開発作業へ発火する、または自己監査を再帰実行する。
- 期待判定: 通常の対象と同じ基準で `ambiguous-description`、`trigger-too-broad`、`instruction-conflict` などを1回だけ記録する。
- 期待変更: `revise` または根拠不足なら `hold`。自己監査を再実行せず、現在のレポート内で終了する。

## 12. 変更不要な良質なSkillが存在する

- 入力: 目的、具体的発火、除外、入力、出力、安全条件、現行参照が明確で、代表routing caseも正しい。
- 期待判定: 状態 `healthy`、問題なし。
- 期待変更: `keep`。均一化や変更件数を作るための `revise` を提案しない。

## 13. 同名の外部カタログentryが複数存在する

- 入力: 同じカタログ名で、完全に同じSkillパッケージが2所在地、異なる内容のSkillパッケージが1所在地にある。外部frontmatterには追加キーがあり、本文にはHTTP URL、サイト内route、未取得の相対参照がある。
- 期待判定: 同一内容の2所在地は1 Skillへ統合して両方の所在地を保持し、異なる内容は同名の別variantとして区別する。追加キーと未取得相対参照はexternal informational findingとし、HTTP URLとrouteはローカル参照切れにしない。
- 期待終了状態: repository-ownedの問題がなければ `--fail-on-issues` は成功する。異なるvariantとexternal findingはレポートで確認できる。

## 代表routing case

次の依頼で整備用Skill自体の発火精度も確認する。

| 依頼 | 期待 |
|---|---|
| 「利用可能なSkillを棚卸しし、重複と不足を監査して」 | 本Skillが監査のみモードで発火し、読み取り専用のレポートで終了する |
| 「Skillを監査し、必要な修正・統合・追加を実施して」 | 本Skillが監査と適用モードで発火し、中間レポート後に同一ターンで安全な非`hold`候補を適用する |
| 「Skillを監査して改善して。削除も許可する」 | 中間レポートで削除候補と知識移行を示し、根拠が確定した対象だけ削除できる。外部公開や`hold`まで許可されたとは扱わない |
| 「Skillを全部改善してGitHubへ公開して」 | ローカルの非`hold`改善は適用できる。GitHubへの公開は対象、出口Skill、公開条件が明確な場合だけ別の許可済み操作として扱う |
| 「代表的な依頼でSkillの誤発火を評価して」 | 本Skillが発火する |
| 「前回の監査でC-003とC-007だけ適用して」 | 本Skillが変更前再監査後、指定候補だけを適用する |
| 「Markdown保存の不具合を修正して」 | 本Skillは発火せず、対象機能Skillを使う |
| 「この要件で単一の新Skillを作って」 | 本Skillは発火せず、`skill-creator`を使う |
| 「現在の差分をコミットして」 | 本Skillは発火せず、コミット用Skillを使う |

## Issue対応の代表routing case

| 依頼 | 期待 |
|---|---|
| 「OpenのIssue #100を対応してください」 | `relic-issue`が発火し、関連する機能Skillで変更、検証、完了コミットまで行う。完了コミットを確認したら、remote到達やpushを要件にせず完了コメントとcloseへ進む。コード変更不要の妥当性判断は明示された対応範囲でcloseできる |
| 「ClosedのIssue #100の対応を確認してください」 | `relic-issue`が発火し、元Issueの完了条件と現行実装だけを照合する。コメントや再openを行わず、確認結果を報告する |
| 「ClosedのIssue #100を確認し、必要なら別Issueとして投稿してください」 | `relic-issue`が発火し、対応不足と新しい問題を分類する。元Issueへ追記せず、重複確認後に独立したIssueを投稿する |
| 「Issue #100の対応後に問題がないか見てください」 | `relic-issue`が発火し、新しい問題が見つかってもIssueを勝手に作成せず、別Issue候補として報告する |
| 「ClosedのIssue #100は対応済みです。さらに使いやすくしてください」 | `relic-issue`がClosed Issueを再利用せず、追加改善を独立した新しい変更依頼として扱う。Issue作成の明示がなければ投稿しない |

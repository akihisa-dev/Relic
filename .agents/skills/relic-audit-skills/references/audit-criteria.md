## 監査単位と証拠

Skill名ではなく、同じ利用者依頼に対して選ばれ、同じ結果を作る責務を比較単位にする。次の順で証拠を強く扱う。

1. 現行HEADの `SKILL.md`、補助資源、ルーティング規則、実在するコード・設定・コマンド
2. 再現可能なscript結果、代表依頼に対するrouting結果、対象テスト
3. 複数の独立したコミットに共通する反復パターン
4. 単発コミット、類似度、行数、命名だけから得た候補

1〜2で直接確認できる判断を「確認済み」、複数の間接証拠が一致する判断を「可能性が高い」、単発または弱い証拠だけの判断を「候補」とする。候補を断定しない。

ワークスペース内のrepository-owned Skillと、利用可能カタログからだけ取得したexternal Skillを分ける。repository-ownedにはリポジトリの構造規約を検証し、externalのfrontmatter拡張、folder規約差、取得環境に存在しない相対参照はinformational findingとして記録する。外部差異をrepository-ownedの失敗件数へ加えない。

## 台帳項目

各Skillについて次を埋める。記載がなければ推測で補わず「未記載」または「確認不能」とする。

| 項目 | 収集方法 |
|---|---|
| 名前 | 利用可能カタログ名を主表に使い、frontmatterの `name` とfolder名も別々に照合。名前空間を落とさない。同名・同一内容のentryは1件へ統合して全所在地を保持し、異なる内容はvariantを区別する |
| scope | workspace内のrepository-ownedか、外部カタログ由来のexternalかを記録 |
| description | frontmatterを原文の意味を変えず要約 |
| 主な目的 | 共通して生む利用者向け結果を1文で記述 |
| 発火条件 | description内の正例、入口、除外、優先関係 |
| 入力 | 依頼、対象ファイル、必要な状態、外部情報 |
| 出力 | レポート、変更、生成物、外部状態の変化 |
| 使用ツール | 明示されたshell、MCP、アプリ、検証ツール |
| 参照ファイル | `SKILL.md` とreferencesから辿る現行パス |
| 実行スクリプト | bundled scriptとリポジトリscriptを分離 |
| 他Skillとの関係 | 優先、併用、委譲、入口、出口、競合 |

## 状態

- `healthy`: 現行HEADと整合し、責務とroutingが明確で、確認済み問題がない
- `attention`: 利用可能だが、修正候補または根拠不足がある
- `stale`: 必須手順または対象が現行HEADと整合せず、そのままでは期待結果を作れない
- `unverified`: 必要なファイル、外部情報、履歴へアクセスできず評価できない

状態は変更種別ではない。`attention` でも `keep`、`healthy` でも将来の `create` 候補との境界確認があり得る。

## 問題の判定

### 責務重複

同じ代表依頼、入力、出力、主要手順を複数Skillが所有し、優先・併用境界がdescriptionで解決できない場合に確定する。共通の安全確認、検証、コミット委譲だけは重複としない。

### 曖昧なdescription

「何をするか」「どの依頼で使うか」のどちらかが欠ける、抽象語だけで対象が特定できない、または重要な除外が本文にしかない場合に判定する。文体の好みだけは問題にしない。

### 発火が広すぎる

異なる結果を求める代表依頼を1つのdescriptionが無条件に取り込み、隣接Skillと誤発火する場合に判定する。単に対象ファイルが多いことは根拠にしない。

### 発火が狭すぎる

同じ責務・手順・結果を持つ反復依頼を、特定ファイル名、単一コミット、過去の一表現だけへ限定して未発火にする場合に判定する。壊れやすい操作を意図的に限定する境界は維持する。

### 巨大Skill

独立して選択・検証・変更できる複数目的を抱え、共通結果がなく、一部を使うために無関係な手順まで読む場合に判定する。行数だけで分割しない。

### 過細分化

複数Skillが常に同時発火し、単独では利用者向け結果を作れず、固有知識もほぼない場合に判定する。安全境界、入口、出口を独立Skillにする意図は尊重する。

### 指示矛盾

同じ前提と対象で、書き込み可否、正本、手順順序、安全条件、完了条件が両立しない場合に確定する。異なるscopeや依頼モードに対する指示差は矛盾ではない。

### 古い手順・HEAD不整合

必須API・コマンド・パス・設定が現行HEADに存在せず、代替も示されない、または現在の正本と逆の手順を求める場合に判定する。生成物、glob、架空fixture、条件付き外部資源は区別する。

外部Skillの追加frontmatter、Web URL、サイト内route、外部配布物にだけ含まれる相対資源は、そのリポジトリの規約と完全な配布内容を確認できない限りRelicの構造問題にしない。外部Skill自体の比較材料としてinformational findingへ分離する。

### 履歴の過度な一般化

単発障害の回避策、revert済み設計、廃止済み運用を一般規則にしている、または現在性を確認していない場合に判定する。

### 不足領域

現在も複数回発生する一貫した作業に固有の調査・安全条件・検証があり、既存Skillでは再発見が必要な場合に `create` 候補とする。単発要望や将来予想だけなら `hold` とする。

## 重要度

| 重要度 | 判定 |
|---|---|
| `critical` | 現行指示が秘密情報漏えい、ユーザーデータ消失、境界突破、無断公開・課金・履歴破壊などを直接起こし得る |
| `high` | 代表依頼で高確率に誤Skillを選ぶ、必須手順が実行不能、相反する指示で主要作業が安全に完了できない |
| `medium` | 回避可能だが、曖昧さ、二重保守、古い任意手順、過不足により反復的な手戻りが起きる |
| `low` | 現在の結果を壊さない表現・命名・metadata・補助導線の軽微な不整合 |

重要度は問題の種類から自動決定せず、現行HEADで起きる影響と発生可能性を根拠にする。

## 変更種別

| 種別 | 用途 |
|---|---|
| `keep` | 現状維持。良質な境界を理由付きで残す |
| `revise` | description、手順、参照、metadataを責務を変えず修正 |
| `rename` | 責務は維持し、名前と全参照を発火意図へ合わせる |
| `merge` | 重複Skillを統合し、双方の固有知識を保持 |
| `split` | 独立目的を分け、入口・出力・検証境界を定義 |
| `deprecate` | 移行先を示して新規利用を止める |
| `delete` | 固有知識と参照を移行済みで、利用価値がない場合だけ削除 |
| `create` | 反復根拠と既存Skillで埋まらない不足領域がある |
| `hold` | 根拠不足、単発事例、影響不明のため変更を保留 |

## 履歴の扱い

- 同種の作業が2件以上あっても、同じ根本原因の連続修正だけなら独立パターンと数えない。
- commit subjectだけで判断せず、変更ファイルと必要なdiffを確認する。
- reviewコメントを取得していない場合は「レビューで指摘された」と書かない。
- 古いコミットのコマンド、パス、設計は現行HEADで再確認する。
- Skill追加コミットだけでは、そのSkillの長期的な有効性を証明しない。

## ルーティング評価

代表依頼は最低でも、各主要責務の正例、隣接境界、対象外、監査のみ、変更適用、入口・出口を含める。各caseで次を判定する。

1. 期待されるSkillは、依頼結果を主に所有するか。
2. descriptionだけを見て実際に選ばれそうか。
3. 同時に選ばれそうな誤発火候補があるか。
4. 具体語が不足して未発火にならないか。
5. 改善案が別caseの正しい発火を壊さないか。

## 自己監査

本Skillを一覧へ一度だけ含め、責務の一貫性、descriptionの正例・除外、`skill-creator`などとの境界、参照とscriptの存在、監査のみの読み取り専用、監査と適用の中間レポート、許可境界、手順量を確認する。自己監査から新しい自己監査を起動せず、その実行のレポート内で結論を1回だけ記録する。

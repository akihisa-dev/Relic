# Relic Code Health Audit

## 監査範囲を決める

1. 監査、診断、説明、レビューだけの依頼では編集しない。修正や改善が明示されている場合だけ変更する。
2. `git status --short`、対象コード、`docs/development.md` の「責務とファイルサイズ」「診断ツール」「検証とテスト」を確認し、無関係な差分を保護する。
3. 守る契約を先に特定する。Markdownとfrontmatter、設定schema、IPC、ワークスペース境界、ユーザー操作、画面構成、配布内容を、品質指標のために変えない。
4. 比較が必要な場合は変更前の基準を同じ環境、コマンド、fixture、warmup回数、実行回数で取得する。

## 診断面を選ぶ

課題に直接関係する検査だけを選ぶ。

| 課題 | 主な確認 |
|------|----------|
| Reactの状態更新、render、key、hook | 現行のReact診断設定、対象rendererテスト、実コード |
| 巨大ファイル、責務混在 | `pnpm source:size`、import/export、状態と純粋処理の境界 |
| main / preload / renderer境界、循環依存 | `pnpm architecture:check`、共有型、IPC契約テスト |
| 初期読込、遅延読込、CSS、asset容量 | `pnpm build:size` または `pnpm build:size:check` |
| 大規模ワークスペース、I/O、索引 | `pnpm performance:workspace` または `pnpm performance:workspace:large` |
| 一般的な回帰 | 対象テスト、`pnpm typecheck`、必要に応じて `pnpm verify` |

外部診断ツールを新たに取得する場合は、ネットワーク利用とバージョンを明示し、正規の承認経路を使う。保存済みの診断点を現在の結果として扱わない。

## 指摘を評価する

1. 各指摘を次のいずれかへ分類し、コードと再現結果を根拠にする。
   - 実害または再現可能な回帰
   - 既存対策で安全性が成立している誤検知
   - 成立する可能性はあるが根拠不足
   - 修正効果より互換性・複雑性・運用負担が大きい保留
2. 診断点数、警告数、行数だけで優先順位を決めない。ユーザーデータ、安全境界、正しさ、操作の停止、繰り返しI/O、不要な再描画を優先する。
3. 誤検知として除外する場合は、既存の防御、入力経路、sanitize、テストなどの根拠を示す。説明できない警告を設定だけで消さない。
4. 1000行超は分割対象、700行超は分割候補として責務を調べる。行数を減らすだけの機械的分割や、循環依存を増やす分割を行わない。

## 改善する

1. 明示的な改善依頼がある場合だけ、実害の大きい指摘から小さな単位で修正する。
2. 純粋処理、状態管理、I/O、描画、互換処理、共有契約を既存の責務境界に沿って分ける。公開契約を変える場合は、互換層、移行、関連正本を同じ作業で扱う。
3. 性能改善では測定対象だけを変え、キャッシュの無効化、保存待ち、外部変更、ワークスペース切替など正しさの条件を回帰テストで固定する。
4. bundle基準値は、増加理由が意図的で承認された場合だけ `pnpm build:size:baseline` で更新する。検査を通すためだけに基準値を更新しない。
5. `doctor.config.json` や検査除外は、誤検知のコード根拠と継続的な確認方法がある場合だけ変更する。

## 再検証して完了する

1. 変更前と同じ条件で診断を再実行し、単発値ではなく中央値、I/O回数、bundle内訳、具体的な警告差分を比較する。
2. 対象テストと型チェックを実行し、影響が広い場合は `pnpm verify`、構造変更では `pnpm architecture:check`、bundle変更では `pnpm build:size:check` を追加する。
3. `pnpm verify:full` をCIの完全再現と表現しない。CIでは現行workflowによりライセンスやbundle検査などが追加される可能性があるため、必要なら `.github/workflows/ci.yml` を確認する。
4. `git diff --check` と全差分を確認し、診断のための一時出力やfixtureを残さない。
5. 実害、誤検知、保留、変更内容、比較条件、結果、未実施の検査と理由を分けて報告する。コミットする場合は `$relic-commit` に従う。

# Relic Task Guard

## 作業契約を固定する

1. 最初の書き込み前に、利用者が求める結果、変更を許可した対象、根拠、対象外、外部操作、完了証拠を短い作業契約として保持する。作業履歴としてファイルへ保存しない。
2. タスク開始時のワークスペースを既定の対象とする。添付画像、ログ、URL、エラー文、ローカルで発見した別リポジトリだけを根拠に対象を切り替えない。
3. 根拠が別リポジトリを示す場合は、読み取り確認だけで不一致を確定し、編集、コミット、push、Issue操作へ進まない。確認済みの対象差と必要な利用者判断を示す。
4. 利用者がリポジトリ外のフォルダやユーザーデータを明示した場合は、そのパスと依頼されたファイル操作だけを別の許可対象にする。そこから別リポジトリの変更権限や外部公開権限を推測しない。
5. 後続指示で目的や範囲が変わった場合は、最新指示を反映して作業契約を更新し、変更済み範囲との食い違いを確認してから続ける。

## Git対象を機械的に照合する

1. タスク開始時に現在のGit root、branch、remoteを読み取り、期待するrootとrepository identityを候補側からではなくタスク側の根拠で固定する。
2. 最初のファイル変更、コミット、push、タグ操作の直前に、対象コマンドの作業ディレクトリを次で照合する。

   ```sh
   python3 .agents/skills/relic-guard-task/scripts/verify_task_target.py \
     --expected-root <task-start-git-root> \
     --candidate <command-working-directory>
   ```

3. remoteへ書き込む場合は、GitHub一次情報または確認済みremoteから得たrepository identityも指定する。

   ```sh
   python3 .agents/skills/relic-guard-task/scripts/verify_task_target.py \
     --expected-root <task-start-git-root> \
     --candidate <command-working-directory> \
     --expected-repository <owner/repository> \
     --remote <remote-name>
   ```

4. 照合失敗時は書き込まない。別リポジトリへの切替は、利用者が対象を明示して指示した場合だけ新しい作業契約として扱う。
5. 明示された外部フォルダがGit管理外の場合はscriptを認可手段に使わず、作業契約の許可パスと実際の対象パスを操作直前に照合する。

## 提示見本の意図を固定する

1. 画像、動画、コード、デザイン見本を実際に確認し、確認した事実と推論を分ける。コード断片だけから寸法、色、配置、対象画面を補完しない。
2. 対象コンポーネント、維持する構造、変更する挙動、視覚上の不変条件、許容するRelic向け調整、未指定事項を受入条件へ対応付ける。
3. 「まず見本」「単独で見せて」など順序が指定された場合は、製品コードを変えずに見本だけを作り、自分でも表示結果を確認して提示する。利用者が採用を指示するまで実装、version更新、コミットへ進まない。
4. 現行テーマへ合わせる場合も、見本の構造と動きを先に保つ。目的のない外枠、面、色、装飾、アニメーションを追加しない。
5. 見本を表示または操作できない場合は、視覚的な一致を完了扱いにせず、未確認条件を示す。

## 専門Skillへ渡して完了証拠を集める

1. 要求ごとに、変更箇所、自動テスト、実画面確認、文書、外部状態のどれで完了を証明するかを決める。無関係な検査を増やさない。
2. UIは `$relic-change-ui` と `$relic-test-development-app`、Issueは `$relic-issue`、コミットとversion・SBOMは `$relic-commit` と `$relic-manage-version`、機能廃止は `$relic-retire-feature` へ委ねる。
3. 複数Skillの完了条件が食い違う場合は、`AGENTS.md` と結果を所有する専門Skillの現行規則へ照合して矛盾を解消する。独自の追加条件で完了を遅らせない。
4. ツールの成功表示、テスト成功、開発版の起動、コミット作成を、それぞれ利用者が求めた結果そのものと混同しない。

## 外部操作と復旧を保護する

1. push、Issueコメント・close、タグ、Release、外部送信は、対象と操作が明示された範囲だけ行う。ローカル修正やコミット許可をpush許可へ広げない。
2. Issueの完了条件へpushを追加せず、`$relic-issue` の現行条件に従う。公開後のremote到達確認は公開状態として別に扱う。
3. 誤った対象への変更を検知した場合は追加操作を止め、ローカル差分、コミット、remote反映、正しい対象への影響を分けて確認する。復旧方法を推測で実行せず、利用者が復旧を指示した範囲だけ安全な打ち消しを行う。
4. 復旧後は対象リポジトリと非対象リポジトリの作業ツリー、branch、local・remoteの一致を必要な範囲で確認する。

## 完了前に再照合する

1. 最新の利用者指示と作業契約を読み直し、`git status`、全差分、対象テスト、必要な実画面、外部状態を結果ごとに照合する。
2. 対象、見本、権限、完了条件のどれかが未確認なら、その結果を確認済みと報告しない。
3. 変更結果、検証済み事項、未確認事項、未実施の外部操作、非対象への影響を分けて報告する。

# Relic Workspace and File Change

## 依頼と所有範囲を確定する

1. 調査、説明、レビューだけの依頼では編集しない。変更が明示されている場合だけ実装する。
2. `git status --short` を確認し、無関係な差分とユーザーの変更を保護する。
3. ワークスペース登録、実ファイル操作、ファイルツリー状態、外部変更監視のどれを変えるか限定する。
4. リネーム・移動に伴う既存リンク更新は扱うが、リンク記法や解決規則自体の変更は `$relic-change-search-links` と組み合わせる。

## 正本と実装境界を確認する

1. `docs/features/files.md`、`docs/engineering/file-access-boundaries.md`、`docs/engineering/data-model.md` を読む。
2. 画面上のワークスペース状態では `docs/features/navigation.md`、検証では `docs/development.md` の関連節だけを追加で読む。
3. `app/src/main/files/`、`app/src/main/workspace/`、関連IPC、共有型、preload、rendererの操作hookを `rg` で特定する。
4. Markdownファイルを正本とし、専用形式、不可視メタデータ、必須の専用フォルダを実ワークスペースへ追加しない。

## ファイルアクセス境界を守る

1. RendererとIPCから受けるパスを信用せず、絶対パス、`..`、NUL文字、未正規化パスを拒否する。
2. 既存対象は実体パスを解決し、新規対象は最も近い既存親の実体パスを確認する。
3. 書込、移動、リネーム、削除の直前にも対象がワークスペース内か再確認する。
4. ワークスペース外を指すシンボリックリンクを読み書き、移動、削除の対象にしない。
5. 外部取込では元ファイルを読取専用とし、コピー先だけをワークスペース内へ限定する。
6. 新規作成、複製、取込では既存ファイルを上書きせず、安全書き込みが必要な操作では一時ファイルから置き換える。
7. 削除は確認後にOSのゴミ箱へ移し、登録一覧からの削除を実フォルダ削除へ拡大しない。
8. エラーをRendererへ返す前に、不要なローカル絶対パス、認証情報らしい値、内部情報を伏せる。

## 状態と関連データを整合させる

1. ワークスペースを開く、作る、切り替える前に未保存内容の確認と必要な保存を完了させる。
2. フォルダ選択ダイアログの取消時に、登録一覧とアクティブワークスペースを変更しない。
3. リネーム・移動前に対象タブの保存を試み、失敗時は実ファイル操作へ進まない。
4. リンク更新では本文とフロントマターを対象にし、コードブロック内を変更しない。更新対象が多い場合は実行前に確認し、外部変更との競合を検出したら古い内容で上書きしない。
5. 複数ファイルへのリンク更新が途中で失敗した場合は適用済み内容を安全に戻し、ロールバック中も外部変更を上書きしない。
6. ファイル変更後はツリー、タブ、ピン留め、共有索引、解析キャッシュを必要範囲だけ更新する。
7. 外部変更監視はアクティブワークスペースを基準にし、別ワークスペースの通知を現在画面へ反映しない。
8. 復元版はアプリ管理領域へ保存し、実ワークスペース内へ履歴ファイルを作らない。

## IPCと実装を更新する

1. 操作契約をsharedの機能別IPC型、preload、main handler、client境界の順に整合させる。
2. 入力validatorで不正値を拒否し、検証前にファイル処理や設定更新を開始しない。
3. UI変更も伴う場合は `$relic-change-ui` の状態別確認を追加し、ファイル安全条件は本スキルを優先する。
4. 仕様変更では対応する正本文書を更新し、実装だけの判断を新仕様として残さない。

## 検証する

1. ファイル操作テストはリポジトリ外の一時ディレクトリで実行し、実ユーザーのワークスペースを使わない。
2. traversal、絶対パス、NUL、外部symlink、新規先の外部親を拒否し、失敗時に副作用がないことを確認する。
3. 上書き防止、取込元保持、ゴミ箱境界、保存失敗時の中止、外部変更競合を対象テストで確認する。
4. リネーム・移動では本文、フロントマター、コードブロック、未保存タブ、大量更新確認、途中失敗時のロールバックを確認する。
5. 登録、切替、取消、一覧削除、外部変更、索引無効化、IPC契約の影響箇所を確認する。
6. `app/` で対象のnode・rendererテスト、`pnpm typecheck`、必要なら `pnpm architecture:check` または `pnpm verify` を実行する。
7. 実画面確認が必要なら、その作業で起動した開発版と専用の `RELIC_DEV_USER_DATA_DIR` だけを使う。
8. 正本文書、差分、検証結果を照合し、コミット時は `$relic-commit` に従う。

# Relic Issue

## 対象と依頼モードを確定する

1. GitHub連携で対象Issueの本文、コメント、OpenまたはClosedの状態、完了条件を取得する。GitHub URLをWeb検索、名前検索、推測で代替しない。アクセスできない場合は理由を示し、推測で続行しない。
2. ユーザーの依頼を調査と対応に分ける。
   - 調査、確認、相談、説明、妥当性判断だけなら、読取と報告だけを行う。リポジトリ編集、Issueコメント、ラベル変更、close、再openを行わない。
   - Open Issueに対する対応、実装、修正、完了、closeが明確なら、実装と完了処理まで進める。
   - Closed Issueの対応確認では、元Issueの完了条件と現行実装の照合だけを行う。新しい作業の継続先として扱わない。
3. 現行仕様、開発ルール、関連コード、既存Issue・Pull Requestを必要な範囲で確認し、Issueの前提と完了条件を検証する。
4. 次の場合は妥当でないOpen Issueとして扱う。現行仕様や方針との明確な矛盾、前提事実の誤り、対応済み、重複、完了条件を判断できない過大・曖昧な範囲、セキュリティ・データ・互換性上の不適切な要求。
5. 情報不足で判断できない場合は推測で実装しない。不足情報と再開条件を示す。Issueへの質問投稿は、外部コメントまで依頼範囲に含まれる場合だけ行う。

## Open Issueを完了する

1. 妥当なIssueでは、本文と既存コメントに明示された要求範囲だけを、対象領域の `relic-change-*` Skillと必要なテストSkillを使って実装する。関連文書を整合させ、リスクに応じた検証を行い、コミット時は `$relic-commit` に従う。
2. 実装後にIssueの要求と実際の対応を照合する。コードや文書の変更を伴う場合は、検証に成功した対象差分を `$relic-commit` で完了コミットにし、コミットIDと残存差分を確認する。コミットできない状態ではIssueへ完了コメントせずcloseもしない。
3. 完了条件を満たす完了コミットを確認したら、対象remoteへの到達可否にかかわらず、対応内容、確認結果、完了コミットIDをIssueへコメントしてcloseする。pushはcloseの要件にせず、Issue対応依頼からpush許可を推測しない。完了報告では未pushであることを明示する。
4. 将来対応、追加改善、関連課題の存在だけを理由にOpenのまま残さない。元Issueの完了条件に含まれない内容を後付けしない。
5. 妥当でない、重複、またはコード変更不要のOpen Issueでは、明確な対応依頼がある場合だけ、判断理由、根拠、必要な代替案または再起票条件をコメントしてcloseできる。
6. 親Issue、明示された検証待ち、リリース待ちなど、元Issue自身の完了条件を満たしておらずcloseできない場合は、その理由を完了報告へ示す。外部コメントが妥当な状態だけ、Issueにも同じ理由を記載する。未pushや別の独立課題が残ることだけはcloseしない理由にしない。

## Closed Issueを再利用しない

1. Closed Issueへ新しい作業内容や追加課題をコメントしない。自動で再openせず、新しい課題の投稿先として再利用しない。
2. Closed Issueの対応確認では、元Issueの目的と完了条件が満たされているかを現行実装で確認し、原則としてGitHubへコメントせず結果をユーザーへ報告する。
3. 元Issueの対応不足が見つかった場合も、Closed Issueへ追記または自動で再openしない。新たな対応単位として分離し、ユーザーの投稿指示がなければIssue候補として報告する。

## 対応不足と追加課題を分類する

1. 次のいずれかなら、元Issueの対応不足として記録する。
   - 元Issueに明示された完了条件を満たしていない。
   - 実装が元Issueの目的と明確に矛盾している。
   - 元Issueが修正対象とした同じ不具合が残っている。
2. 次のいずれかなら、元Issueとは別の新しい問題として扱う。
   - 元Issueの完了条件に含まれない不具合。
   - 実装後に初めて判明した別の問題。
   - 追加改善、設計変更、操作性向上、性能改善。
   - 元Issueと関連していても、独立した完了条件を持つ内容。
3. どちらの場合も、完了済みIssueへ作業を継ぎ足さない。元Issueとの関係、確認した根拠、独立した完了条件を分けて整理する。

## 新しいIssueと外部状態を扱う

1. 新しい問題を見つけても、ユーザーが「別Issueとして投稿」「新しいIssueを作成」などと明示していなければ、勝手にIssueを作成しない。
2. 新規Issue作成が明示された場合は、既存のOpen・Closed Issueとの重複を確認し、元Issueとの関係を必要な範囲で記載した独立Issueとして投稿する。
3. ユーザーがIssueを触らない、コメントしない、closeしないと指定した場合は、その制限を優先する。ラベル、担当者、milestone、Pull Requestなど、依頼にない外部状態を変更しない。

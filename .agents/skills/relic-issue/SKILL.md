# Relic Issue

## 手順

1. GitHub連携で対象Issueの本文、コメント、状態、完了条件を取得する。GitHub URLをWeb検索、名前検索、推測で代替しない。アクセスできない場合は理由を示し、推測で続行しない。
2. ユーザーの依頼を調査と対応に分ける。
   - 調査、確認、相談、説明、妥当性判断だけなら、読取と報告だけを行う。リポジトリ編集、Issueコメント、ラベル変更、closeを行わない。
   - 対応、実装、修正、完了、closeが明確なら、以下の出口処理まで進める。
3. 現行仕様、開発ルール、関連コード、既存Issue・Pull Requestを必要な範囲で確認し、Issueの前提と完了条件を検証する。
4. 次の場合は妥当でないIssueとして扱う。現行仕様や方針との明確な矛盾、前提事実の誤り、対応済み、重複、完了条件を判断できない過大・曖昧な範囲、セキュリティ・データ・互換性上の不適切な要求。
5. 情報不足で判断できない場合は推測で実装しない。不足情報と再開条件を示す。Issueへの質問投稿は、外部コメントまで依頼範囲に含まれる場合だけ行う。
6. 妥当なIssueでは、要求範囲だけを実装し、関連文書を整合させ、リスクに応じた検証を行う。コミット時は `$relic-commit` に従う。
7. 実装後にIssueの要求と実際の対応を照合する。満たしている場合だけ、対応内容、確認結果、関連コミットをIssueへコメントしてcloseする。
8. 妥当でないIssueでは実装しない。明確な対応依頼がある場合だけ、判断理由、根拠、必要な代替案または再起票条件をコメントしてcloseする。
9. 親Issue、検証待ち、リリース待ち、別タスク残存などでcloseできない場合は、理由をIssueコメントと完了報告の両方に示す。
10. ユーザーがIssueを触らない、コメントしない、closeしないと指定した場合は、その制限を優先する。ラベル、担当者、milestone、Pull Requestなど、依頼にない外部状態を変更しない。

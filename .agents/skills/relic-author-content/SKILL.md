# Relic向け資料を作成する

Relicの機能を装飾として並べず、資料を読む、探す、比較する、関連をたどる、俯瞰するという目的へ割り当てる。Markdownファイルを正本とし、Relicがなくても内容を読める状態を保つ。

## 最初に確認する

1. 利用者が求める成果物、読者、用途、根拠資料、出力先、単一ファイルか資料群かを確認する。
2. 既存ワークスペースを編集する場合は、ファイル名、フォルダ構成、既存タグ、カテゴリ、aliases、リンク規則を先に読む。利用者の既存内容を推測で上書きしない。
3. 根拠資料がある場合は、事実、引用、推論、未確認事項を区別する。出典URLや書誌情報は本文または脚注へ残す。
4. すべての作成・改稿で [references/relic-authoring-reference.md](references/relic-authoring-reference.md) を読む。複数ファイル、カード、テーブル、グラフ、スフィア、クロニクルを使う場合は [references/content-architecture.md](references/content-architecture.md) も読む。

出力先が指定されていない短い依頼では、完成したMarkdownを応答内に示してよい。ファイル作成を求められた場合は、指定されたワークスペース境界内だけへ書き込む。

## 成果物の形を決める

次の基準で、先に情報構造を決める。

- 一続きに読む内容、単独配布、PDF化を主目的とする場合は単一ファイルにする。
- 項目ごとに独立して更新する内容、人物・用語・場所・資料を横断してたどる内容は複数ファイルに分ける。
- 一覧性が重要な場合は、共通のトップレベルフロントマターを揃え、テーブルビューで比較できるようにする。
- 視覚的なカタログが重要な場合だけ `card` と最初の `flavortext` ブロックを使う。
- 時系列上の位置が内容の理解に不可欠な場合だけ `chronicle` を使う。
- 関係の俯瞰が役立つ場合は内部リンクを使う。グラフを賑やかにする目的で無関係なリンクや本文 `#タグ` を足さない。

単一資料でも資料群でも、H1は資料の固有タイトルとし、見出し階層を飛ばさない。ファイル名は短く識別できる名前にし、同名ファイルを避ける。同名が必要なら、リンクを書くファイルの場所から計算した相対パスを内部リンクに含める。

## Relic機能を目的へ割り当てる

必要な機能だけを選ぶ。

| 読者の行動 | 使う構造 |
|---|---|
| 長文を読み、位置を把握する | H1〜H6、短い節、アウトライン、見出しリンク |
| 比較する | Markdownテーブル、共通フロントマター、テーブルビュー |
| 状態を追う | チェックボックス、状態・期限などの任意プロパティ |
| 用語や資料を探す | 明確なファイル名、`aliases`、`tags`、`category`、検索しやすい表記 |
| 関連項目をたどる | `[[内部リンク]]`、見出しリンク、必要時だけ埋め込み |
| 関係全体を俯瞰する | 意味のある内部リンク、`tags`、グラフ・スフィアビュー |
| 項目を視覚的に選ぶ | `card`、代表画像、`flavortext` |
| 出来事を年代で読む | `chronicle`、`category`、クロニクルビュー |
| 手順や構造を図解する | MermaidまたはD2コードブロック |
| 補助資料を参照する | 相対パス画像、PDF添付、脚注、出典リンク |
| 再利用する | 小さな独立ファイルと `![[埋め込み]]` |
| 配布する | 自己完結した本文、相対画像、PDF出力で欠けない構成 |

## 内容を書く

1. 冒頭で資料の目的と結論または読み方を示す。
2. 一つの節には一つの論点を置き、見出しだけを追っても全体像が分かる名称にする。
3. 段落は説明、箇条書きは並列項目、番号付きリストは順序、表は同じ軸での比較に使う。
4. 強調は意味のある語だけに使う。色や装飾を前提にせず、プレーンテキストでも意味を保つ。
5. 図表は本文の説明を置き換えず、図の前後に読み取る要点を書く。MermaidとD2のどちらでもよい場合は、時系列・処理順・状態遷移にはMermaid、構成・配置・依存関係にはD2を優先する。
6. 数値、日付、固有名、引用、リンク先を根拠資料と照合する。不明な値をもっともらしく補完しない。
7. 長い資料群では入口となる索引ファイルを作り、主要ファイルへの内部リンクと各資料の役割を示す。

## フロントマターを設計する

フロントマターは検索・一覧・可視化に必要な場合だけ付ける。本文に書くべき重要情報をYAMLだけへ隔離しない。

- `aliases`: 別名からファイルを探す必要がある場合に使う。
- `tags`: 横断的な分類に使う。ワークスペース内の既存表記を再利用し、階層は `親/子` とする。
- `category`: 相互排他的な主分類が必要な場合に単一値で使う。
- `card`: カード一覧へ出す項目だけに置く。画像が不要でも空値でよい。
- `chronicle`: 暦、開始年、終了年を確定できる項目だけに置く。
- 任意プロパティ: テーブルで比較・絞り込みたい属性に使い、資料群内でキー名と値の型を揃える。

YAMLはファイル先頭に置き、配列はブロック配列で書く。日付は `YYYY-MM-DD`、真偽値は `true` / `false`、数値は数値として書く。人が読むラベルと機械的に揃える値を混同しない。

## リンクと分割を設計する

- 初出または理解上重要な箇所からだけリンクし、同じ段落で同一対象へ何度もリンクしない。
- 表示名を文脈へ合わせる場合は `[[対象|表示名]]` を使う。
- 特定の論点へ案内する場合は `[[対象#見出し]]` を使う。
- 再利用部分を埋め込む場合は、原文側だけを編集する前提で `![[対象]]` を使う。巨大ファイルや多段埋め込みへ依存しない。
- 説明なしでは意味が分からない関係は、リンクの前後に関係性を文章で明記する。
- 既存ファイルへのリンクは実在と解決先を確認する。将来作成するリンクは、未作成であることを利用者に明示する。

## 完成前に検査する

1. YAML開始位置、構文、配列形式、固定プロパティの型を確認する。
2. H1の重複、見出し階層の飛び、空見出し、壊れた表、閉じていないコードフェンスや数式を確認する。
3. 内部リンク、埋め込み、相対画像、相対Markdownリンクを、各リンク元ファイルのフォルダから解決して確認する。
4. タグ、カテゴリ、任意プロパティの表記と型が資料群で一貫しているか確認する。
5. カードの説明が最初の完結した `flavortext` になっているか、chronicleの年と暦が有効か確認する。
6. 図がなくても本文を理解できるか、図表ソースがMermaidまたはD2として妥当か確認する。
7. 外部URLや書誌情報が出典として残っているか、事実と推論が混ざっていないか確認する。
8. 通常のMarkdown表示、Relicのアウトライン・検索・各ビュー、PDF出力のどこでも意味が欠落しないか確認する。

実際にRelicを起動して表示確認するのは、利用者が明示的に求めた場合だけにする。起動しない場合も、生成ファイルとリンク・構文の静的確認を行う。

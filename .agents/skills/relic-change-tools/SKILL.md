# Relic File Tool Change

## 対象と権限を確定する

1. 調査、説明、レビューだけの依頼では編集しない。追加、変更、修正が明示された場合だけ実装する。
2. `git status --short` を確認し、無関係な差分とユーザーの変更を保護する。
3. `docs/features/tools.md`、`docs/features/files.md`、`docs/engineering/file-access-boundaries.md` を読む。
4. 絞り込みやリンク生成へ触れる場合はfrontmatter、links、data-model、IPCへ触れる場合はarchitectureの関連節だけを追加で読む。内部リンクの記法・解決規則そのものを変える場合は `$relic-change-links-index` を併用する。
5. 現行4ツールの修正か、新しいツールを追加する仕様変更かを分ける。未確定の新ツールを既存仕様へ混ぜない。

## 入力を読み取り専用に保つ

1. 加工対象の既存ファイルとフォルダを読み取り専用として扱い、本文、フロントマター、更新日時、名前、配置を変更しない。
2. 結果は必ず新しいMarkdownファイルとしてワークスペース内へ作成し、既存出力を自動更新しない。
3. 対象フォルダと出力先は空文字または正規化済みワークスペース相対パスだけを受け付ける。
4. 絶対パス、`..`、NUL文字、パス区切りを含む出力名、外部を指すシンボリックリンクを拒否する。
5. 書き込み直前にも出力先の実体境界を確認し、同名ファイルを上書きせず有限回の別名候補で保存する。
6. 出力は一時ファイルからの安全書き込みとし、失敗時に不完全な結果や一時ファイルを残さない。
7. 実ユーザーのワークスペースをテストに使わず、入力・出力とも一時ディレクトリへ限定する。

## ツールの契約を実装する

1. 候補作成中に個別ファイルを読めない場合と、実行対象の読込失敗を区別し、正本どおりにスキップまたは全体失敗させる。
2. フォルダ内マージでは絞り込み、順序、見出し、区切り、前後空白、既定名を独立した純粋処理として扱う。
3. タイトル一覧と目次では、同名Markdownの有無に応じて短い内部リンクと一意なパス付きリンクを使い分ける。
4. 目次ではフォルダ優先、階層、サブフォルダ設定を保ち、読めない対象フォルダ自体は安全側で失敗させる。
5. タグ別索引ではフロントマター `tags` だけを使い、タグなし設定、タグ順、各タグ内の順序を別々に検証する。
6. ファイル名、フォルダ名、タグ名を見出し、強調、内部リンクへ埋め込む場合は既存のMarkdown・Wikiリンク整形を再利用し、危険文字で構造を壊さず安全なプレーンテキストへ倒す。
7. ワークスペース全体を走査する場合は既存の共有索引とファイル情報を再利用し、本文・stat読込を現行上限の有限並列に保つ。
8. 絞り込み値の型変換やタグ正規化をrenderer表示だけに依存せず、main側の実行境界でも検証する。対象path、出力名、選択肢、件数・本文量の上限は処理開始前に拒否し、候補抽出、走査、出力作成へ不正入力を渡さない。
9. shared型、preload、main handler、renderer clientを同じ操作として追跡できる形で整合させ、必要以上に強いファイルAPIを公開しない。
10. パネルの各操作は現行の既定値で実行し、ワークスペース未選択、実行中、成功、失敗を明確な状態として扱う。
11. 見た目や導線も変える場合は `$relic-change-ui`、共通のパス・保存処理を変える場合は `$relic-change-workspace-files` の条件も適用する。

## 検証する

1. 各ツールについて空入力、既定値、絞り込み、並び順、同名ファイル、空結果、Markdown境界文字を含む名前を対象テストで確認する。
2. 読めない候補、確定後の読込失敗、対象フォルダ失敗、書込失敗、外部symlink、候補上限で副作用がないことを確認する。
3. 出力内容がMarkdownとして正しく、入力ファイルがbyte単位で変化していないことを確認する。
4. IPC契約または入力validatorを変えた場合はshared、preload、main、client、validatorの契約テストを更新する。不正path、不正選択肢、上限超過が処理本体と書き込みを呼ばないことまで確認する。
5. 全体走査や並列処理を変えた場合は、同じ件数と入力条件で時間・メモリ・同時読込数を比較する。
6. ツールの仕様分岐、ファイル操作、IPC入力検証を追加・変更した場合は対象テストまたは回帰テストを追加し、`app/` で `pnpm verify` を実行する。その他の変更でも対象のNode・rendererテストと `pnpm typecheck` を実行し、影響が広い場合は `pnpm architecture:check` または `pnpm verify` を実行する。
7. UI操作を変えた場合だけ、その作業で起動した一時データの開発版で状態遷移を確認する。
8. 正本文書、差分、生成結果を照合し、コミット時は `$relic-commit` に従う。

## 完了する

変更したツール、保持した入力、生成結果、パス境界、失敗条件、更新した正本、実施した検証を報告する。

# Relic Feature Retirement

## 廃止の権限と完了状態を確定する

1. 調査、監査、整理、縮小提案だけから機能廃止を推論せず、廃止対象と変更許可が明示されていることを確認する。
2. `git status --short` を確認し、無関係な差分とユーザーの変更を保護する。
3. `docs/INDEX.md` から対象機能の正本を選び、実装、設定、保存データ、依存、Git履歴から現行範囲を確認する。
4. 「入口だけ隠す」「新規利用を止めて互換読込を残す」「実装と保存形式まで除去する」を区別し、依頼された終了状態を受入条件にする。
5. 正本と関連コードでも終了状態を特定できず、ユーザーに見える結果や保存データ互換が大きく変わる場合だけ確認する。
6. ワークスペース内Markdownをユーザーデータの正本として保ち、機能廃止を理由に本文、添付、フォルダを削除または専用形式へ変換しない。

## 残存面を棚卸しする

1. 機能名、表示文言、翻訳キー、型名、ID、IPCチャンネル、設定キー、CSS class、ファイル名、依存名を検索語として整理する。
2. 次の面を対象ごとに `present`、`compatibility-only`、`not-applicable` に分類する。
   - 左レール、コマンド、メニュー、タブ、パネル、ショートカットなどの入口
   - React状態、hook、model、保存queue、browser保存領域、設定schemaと既定値
   - shared型、validator、preload API、main handler、service、ファイル処理、通知
   - Markdown・アプリ設定・ワークスペース設定に保存済みの値と互換アダプタ
   - 翻訳、CSS、画像、fixture、テスト、package依存、script、workflow、配布資産
   - 正本文書、README、CONTRIBUTING、索引、AGENTSのルーティング、Skill
3. 動的import、遅延chunk、文字列ID、登録配列、re-export、生成物入力など、通常のimport検索だけでは見えない接続を確認する。
4. 対象機能が他機能へ提供している共有処理を特定し、利用中の共有処理を機能と一緒に削除しない。

## 保存データと互換性を設計する

1. 保存済み値を「今後も読む」「安全に無視する」「一度だけ移行する」「明示確認後に削除する」に分類する。
2. 旧設定を無視できる場合も設定ファイル全体を壊さず、未知キーを許容する現行契約と書き戻し結果を確認する。
3. 旧IPC、旧設定schema、旧Markdown記法の互換読込は、サポート対象から不要になった根拠がある場合だけ除く。
4. 移行が必要な場合は再実行可能にし、途中失敗時に元データを失わず、成功後の正本を一意にする。
5. 廃止対象の値がMarkdownに残る場合は通常テキストまたは未認識YAMLとして保持し、不可視メタデータで廃止状態を記録しない。
6. ユーザーへの警告や移行UIが必要なら、実行前の影響、取り消せない結果、退避方法を具体的に示す。

## 依存順に撤去する

1. 対象機能の関連Skillを併用し、保存データ、IPC、設定、UIなど各境界の不変条件を引き継ぐ。
2. 新しい利用を生む入口と登録を先に止め、保存flushや移行を完了させてから実装を外す。
3. renderer利用箇所、状態、client、preload、shared契約、main handler、serviceの接続を追跡し、片側だけ残るIPCを作らない。
4. 設定キーを外す場合はschema、既定値、validator、移行、UI、翻訳、テスト、正本文書を同じ作業で整合させる。
5. コード利用がなくなったことを確認してから依存、CSS、資産、fixture、scriptを削除し、lockfileはpackage managerで更新する。
6. 文書では廃止作業の履歴を仕様として残さず、廃止後の現在状態へ更新する。互換入力を残す場合だけ、その現行上の意味を記す。
7. 他製品名、廃止調査用の一時ログ、ローカル絶対パス、秘密情報をコード、テスト、文書へ残さない。

## 廃止完了を検証する

1. 棚卸しした検索語を再検索し、残存箇所を意図した互換処理、履歴、無関係な一般語へ分類する。
2. 削除した実装のテストを消すだけで完了とせず、入口不存在、契約不存在、旧データ読込、移行、副作用なしを必要に応じて回帰テストへ固定する。
3. `app/` で対象テスト、`pnpm typecheck`、`pnpm architecture:check` を実行し、影響が広い場合は `pnpm verify` を実行する。
4. Renderer dependencyや遅延読込境界を変更した場合はlockfile、ライセンス、`pnpm renderer:production:check` を確認する。
5. 文書の追加・削除・移動を伴う場合は `pnpm docs:index:check` を実行する。
6. ユーザーに見える入口や画面を廃止した場合は、その作業で起動した開発版だけを使い、入口不存在、代替導線、既存データの扱いを確認する。
7. `git diff --check` と全差分を確認し、対象外機能、ユーザーデータ、互換処理を巻き込んでいないことを照合する。

## 完了する

廃止した機能面、残した互換処理、保存データへの影響、削除した依存、更新した正本、実施した検証、意図して残した参照を報告する。コミット時は `$relic-commit` に従う。

# 証拠と変更単位

## 目次

- 証拠を集める順序
- 問題場面を記述する
- 履歴から得た再利用可能な注意点
- 変更単位を決める

## 証拠を集める順序

1. 現行HEAD、作業ツリー、対象の正本、`app/package.json` を確定する。
2. 対象の入口から、公開型、IPC、状態、I/O、永続化、UI、テスト、build、配布まで影響地図を作る。
3. `rg`、import・export、handler登録、購読、動的import、CSS import、Forge設定から実際の接続を確認する。
4. 対象パスの `git log --follow`、変更理由が分かるコミット、後続のfix・perf・revertを確認する。
5. Issueが指定された場合はGitHub連携の一次情報を使う。取得できないIssueを推測で補わない。
6. 現行の対象テスト、測定script、baseline、workflowが何を検知し、何を検知しないか読む。

## 問題場面を記述する

候補ごとに次を埋める。空欄が残る候補は、実装へ進む前に追加調査する。

| 項目 | 内容 |
|------|------|
| 変更・実行場面 | 例: 同じ設定ファイルへwriteとupdateが重なる |
| 観測される問題 | 例: 後から完了した処理が一方の変更を失わせる |
| コード経路 | 入口、所有状態、I/O、完了通知まで |
| 根拠 | 再現、テスト、測定、履歴、現行コード |
| 守る契約 | 保存形式、IPC、操作、表示、配布など |
| 修正後の所有者 | 状態や責務を一つだけ挙げる |
| 検証 | 正常、競合、失敗、取消、復旧、終了など |

「長い」「複雑」「重複して見える」は調査の入口であり、問題場面の代わりにしない。

## 履歴から得た再利用可能な注意点

コミット `46a98f76b9f2905d4ff89b72618c094ffa2ab504` は、IPC、設定、共有索引、renderer、bundle、配布、検証、文書を同じ互換条件の下で改善した実例である。手順を固定化せず、次の原則だけを現行HEADへ再照合して使う。

- 境界の分割はファイル移動では完了しない。公開契約、互換アダプタ、利用側client、契約検査が一致して初めて境界になる。
- cacheや共有索引はhit率だけでは評価できない。生成、無効化、変更ファイルだけの更新、外部変更、切替、失敗後の復旧を一組で扱う。
- 状態保存の共通化は全操作を同じ直列化規則へ含める。後続の `f955b3e9` はwriteとupdateの競合が残り得たことを示すため、APIごとの正常系だけで完了しない。
- 不要描画の削減は停止条件だけでなく再開条件を列挙する。後続の `69c06623` は、入力、data、theme、size、simulation更新から一度だけ再開する検証が必要だと示す。
- 長寿命の監視処理は開始成功だけでは足りない。後続の `a6f4a573` は、実行中失敗、backoff、復旧時の再確認、workspace切替、終了時破棄、利用者通知までが境界であることを示す。
- 配布検査が通っても、実際のtoolchain・OS経路は別に壊れ得る。直後の `f5a683dd` は、対象環境のbuildを要する変更を静的なASAR検査だけで完了できないことを示す。
- 追加した回帰検査自体も現行HEADで継続確認する。baselineやSBOMの既存不一致を今回の回帰と混同せず、検査を弱めずに扱う。

これらのコミットは現在仕様の正本ではない。現行の正本文書、コード、tests、`app/package.json` と矛盾する場合は現行HEADを優先する。

## 変更単位を決める

同じ単位に含める目安:

- 同じ互換契約を変更または保護する
- 一方だけでは途中状態が検証不能になる
- 同じ状態所有や無効化規則を共有する
- 実装、回帰テスト、正本文書が一つの完了条件を作る

分離する目安:

- 利用者に見える結果や互換性が独立している
- 別の責務所有者と検証方法を持つ
- 片方を保留しても他方が完全に成立する
- 新たに見つかった問題で、当初の根拠と完了条件を広げる

変更単位ごとに、目的、非目的、守る契約、対象path、依存する単位、検証、削除条件を記録する。

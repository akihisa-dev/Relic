# Relic Codebase Refactoring

## 作業契約を固定する

1. 最初に `git status --short`、現在のHEAD、対象範囲を確認し、無関係な差分とユーザーの変更を保護する。
2. 全体または指定領域のどちらでも、機能追加ではなく現行の品質改善であることを確認する。行数や変更量ではなく、複数の責務、状態所有、実行境界、保存、性能、配布、文書を一つの完了条件で横断する作業を大規模リファクタリングとして扱う。単一fileや一つの既知問題に閉じる場合は `$relic-audit-code-health` または対象領域のSkillへ渡す。調査・計画・レビューだけの依頼では編集しない。
3. Markdown、frontmatter、設定schema、保存済みデータ、ワークスペース境界、IPC、公開型、画面構成、マウス・キーボード操作、CSS読込順、配布資源を先に互換契約として記録する。
4. 実ユーザーデータや既存のRelicウインドウを検証に使わない。ユーザーが実画面確認を明示的に指示した場合だけ `$relic-test-development-app` に従い、その作業で起動した開発版と一時データを使う。
5. push、Release、外部公開を行わない。未確認の互換処理、移行、旧形式を推測で削除しない。

## 現行HEADから影響地図を作る

1. `AGENTS.md`、`docs/INDEX.md`、`docs/development.md`、対象機能と設計の正本、`app/package.json`、対象コードを読む。コマンドや構成を履歴から推測しない。
2. 対象の入口と呼出元、公開API・型、main・preload・rendererとIPC、状態所有、I/O・永続化、UI接続、テスト、build、配布まで追跡する。
3. import、export、登録処理、イベント、購読、cacheの生成・無効化、保存・終了待ちを双方向に調べる。ファイル名やディレクトリ名だけで責務を判断しない。
4. Git履歴、関連Issue、後続修正、revertから再発箇所と不完全な境界を探す。単発事例は恒久規則にせず、全判断を現行実装へ再照合する。
5. 詳しい証拠の集め方と履歴から得た注意点は [証拠と変更単位](references/evidence-and-scope.md) を読む。
6. 調査した候補を `confirmed`、`not-found`、`insufficient-evidence`、`out-of-scope` に分類する。未分類の候補を残したまま「包括的に調査済み」と報告しない。

## 実在する問題を確定する

1. 「どの変更時または実行時に、誰に何が起きるか」を再現またはコード経路で説明する。行数、関数数、依存数、診断点、coverageだけで問題を確定しない。
2. 次を候補として調べ、該当しない項目を機械的に実装しない。
   - 一つの所有者を持たない状態、責務や互換分岐の散在
   - UI・状態管理・純粋処理・実行環境依存処理の混在
   - main・preload・renderer、公開API、IPC、保存形式の曖昧な境界
   - 非同期処理の競合、取消、終了、再試行、切替後の古い完了
   - 重複I/O、再解析、再計算、再描画、無効化漏れ、不要な初期読込
   - 旧実装、未使用CSS・asset、不要な配布ファイル、文書不一致
   - 重要な契約を守らないテスト、構造・性能・配布の回帰検知不足
3. 候補ごとに根拠、問題場面、現所有者、影響、守る互換性、修正後の所有者、検証方法を記録する。根拠不足は保留にする。
4. 責務分割を既定の解決策にしない。分割、統合、依存方向の修正、状態所有の集約、抽象化の除去から、最小の構造変更で所有者と境界を明確にできる方法を選ぶ。
5. 巨大なmanager、service、hook、context、store、共通モジュールへ問題を移すだけの案を拒否する。新抽象化の利用者、公開面、変更理由、状態寿命を説明できる場合だけ導入する。

## 変更単位と順序を設計する

1. 問題を、目的、守る契約、変更範囲、完了条件、依存する変更が明確な単位へ整理する。独立課題は分離し、同じ不変条件を共有する密接な変更は機械的に分断しない。
2. 純粋処理、実行環境依存処理、状態所有、公開境界を区別し、依存方向を明示する。UIからOS処理を直接参照させず、互換アダプタから現行modelへ旧形式を漏らさない。
3. 次の順を基本に、各段階で検証可能にする。
   1. 現行挙動のcharacterization testまたは測定基準を置く。
   2. 公開契約と互換層を維持したまま、新しい責務境界を作る。
   3. 純粋処理、状態所有、I/Oの順に移し、呼出側を段階的に切り替える。
   4. 非同期の取消・失敗・再試行・終了・切替条件を固定する。
   5. 全参照と互換性を確認してから旧経路、未使用CSS、assetを削除する。
4. 性能変更では、cache hitだけでなく無効化、外部変更、保存待ち、ワークスペース切替、失敗後の復旧を同じ設計で扱う。
5. 変更途中で新しい独立問題を見つけても無制限に取り込まず、根拠と影響を残して別課題候補にする。
6. 最初の構造変更前に、各変更単位の問題場面、修正後の所有者、守る契約、依存する変更、回帰検知、旧経路の削除条件が埋まっていることを確認する。欠けている単位は追加調査へ戻し、実装を始めない。

## 変更に対応する回帰検知を作る

1. 純粋処理、IPC契約、保存・移行・parser・索引・cache、非同期競合、取消・終了・再試行、禁止依存・循環依存、I/O・再描画、Rendererの初期読込、配布物から変更に関係する面を選ぶ。
2. テスト件数やcoverage値を増やすことを目的にせず、変更で壊れ得る契約と失敗経路を固定する。検査を通すために型、安全条件、構造規則、初期読込境界を弱めない。coverage割合そのものは合否条件にしない。
3. `app/package.json` と `docs/development.md` でコマンドの存在と意味を再確認し、対象テストから型検査、全体検証へ広げる。
4. 構造変更では通常テストに加えてプロセス境界と循環依存を、性能変更では同一fixture・warmup・実行回数の中央値とI/O回数を、初期読込変更ではproduction manifestの静的import経路を確認する。
5. 配布変更ではASAR allowlist、必須entry、resources、OS差、ライセンス、第三者通知、SBOMを確認する。配布buildが必要な変更では `$relic-debug-packaging` を併用する。
6. 選択基準と現行コマンドの確認箇所は [互換性と検証](references/compatibility-and-verification.md) を読む。

## 正本と実装を同期する

1. 責務、依存方向、データモデル、IPC、保存形式、性能条件、配布内容、検証方法が変わった場合だけ、対応する正本文書を同じ変更で更新する。
2. 作業履歴や一時測定値を正本へ蓄積せず、現在成立する設計と運用だけを書く。
3. 文書の追加・移動・削除があれば `$relic-maintain-docs` に従い、索引を同期する。

## 完成を監査する

1. 狭いテストから必要な広い検証まで実行し、`git diff --check`、全差分、生成物、残存参照を確認する。
2. 次を根拠付きで評価する。
   - 調査対象の候補がすべて分類され、`confirmed` の解消または明示的な保留理由があるか
   - 主要問題が問題場面ごと解消したか
   - 責務、依存方向、状態所有、実行境界が明確になったか
   - 保存データ、IPC、公開型、操作、CSS、配布の互換性を維持したか
   - 新しい巨大抽象化、循環依存、二重状態、互換分岐の拡散がないか
   - I/O、再解析、再計算、再描画、初期読込、配布物への効果を確認したか
   - 失敗経路と後続変更を検知する回帰テストまたは検査があるか
   - 実装、テスト、正本文書が一致するか
3. 既存失敗と今回の回帰を基準実行と差分で分ける。未確認OS、未測定性能、取得できないIssueなどを成功と表現しない。
4. 変更結果、対象範囲、主要根拠、互換性、検証、性能・配布効果、文書、未確認事項を報告する。コミットする場合は `$relic-commit` に従う。

## 隣接Skillへ接続する

- 診断や候補整理だけ、React警告、単一の巨大ファイル・初期読込・性能調査は `$relic-audit-code-health` を優先する。
- 実装中のIPC、設定、ファイル、検索、エディタ、出力、UI、可視化は対応するrelic-change系Skillを併用し、その領域固有の安全条件を受け持たせる。
- テスト失敗だけ、テスト追加だけは `$relic-test-code`、依存更新は `$relic-update-dependencies`、配布障害は `$relic-debug-packaging` を優先する。
- 発火例、対象外、隣接Skillとの境界を変更するときは [ルーティング評価](references/routing-evaluation.md) でdescriptionを再評価する。

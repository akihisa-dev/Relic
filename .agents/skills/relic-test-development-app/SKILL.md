# Relic Development App Test

## 実施条件を確認する

1. 実アプリの起動、GUI操作、実画面確認、スクリーンショット、E2E、起動スモークは、ユーザーが現在の作業で明示的に指示した場合だけ実施する。
2. UIや操作の変更依頼、自動テストで判断できない状態、他のSkillが実画面確認を候補に挙げていることだけを、実施指示とみなさない。
3. 明示指示がない場合は、`pnpm start:isolated`、`pnpm smoke:electron`、`pnpm smoke:package` などElectronを起動するコマンドを実行しない。自動テスト、型チェック、非起動のビルド、文書整合、差分確認で検証し、実画面未確認を完了の阻害条件にしない。
4. 明示指示がある場合だけ、確認する受入条件を列挙して以降の手順へ進む。

## 対象を限定する

1. 自動テストで確認できる内容は先に自動テストで確認し、ユーザーが指示した範囲から実画面でしか判断できない受入条件だけを列挙する。
2. 実画面確認には、その作業中に自分が起動したVite開発server経由のElectronだけを使う。
3. 既存のRelicウインドウ、配布版、package版、`app/out/`、インストール済みRelicアプリ、製品Bundle IDを、検出、起動、フォーカス、画面取得、操作の対象にしない。
4. 通常の確認のために `package`、`make`、配布成果物の生成を行わない。

## 開発版を起動する

1. 実ユーザーのデータを使わず、絶対パスの一時 `RELIC_DEV_USER_DATA_DIR` と一時ワークスペースを用意する。
2. `app/` で `pnpm start:isolated -- --user-data-dir <absolute-temp-path>` を実行し、返されたterminal sessionを保持する。このNode launcherが環境変数を設定するため、POSIX shellとPowerShellで別構文を使わない。
3. 同じsessionの出力でVite server、`Launched Electron app`、`RELIC_DEV_APP_IDENTITY=<json>` を確認する。identityのPID、実行ファイルpath、アプリpathを保持し、別terminal、過去のserver、既存プロセスを起動証拠にしない。
4. 起動失敗、port競合、ビルドエラーがある場合は原因を解消するか、実画面未確認として停止する。

## 操作対象を証明する

GUI操作の前に、操作対象が保持中のterminal sessionから起動した開発版であることを証明する。

有効な証明は、同じ起動処理から得たprocess ID、window handle、debug endpoint、またはidentityに含まれる完全な開発版実行pathなど、起動元との対応を機械的に追跡できる識別子に限る。

path指定だけを受けるGUIツールでは、identityの完全なアプリpathを使い、その実行ファイルpathに一致するプロセスがidentityのPIDだけであることを操作直前に確認する。同じpathの別プロセスが存在する場合は操作しない。PIDやwindow handleを指定できる場合は、それらを優先する。

次は証明として扱わない。

- 表示名 `Relic` や一般名 `Electron`
- 製品Bundle ID
- インストール済みアプリのpath、またはidentityと一致しない推測path
- 最近使ったアプリ、最前面ウインドウ、アプリ一覧の一致
- GUIツールが名前から自動起動した対象

GUIツールが表示名、Bundle ID、インストール済みアプリしか指定できず、identityのPID・window・完全な開発版pathのいずれにも結び付けられない場合は操作しない。対象を証明できない時点で停止し、別の識別子へ推測でフォールバックしない。

## 権限申請を正確にする

1. 権限申請には、実行するコマンドまたは証明済みの対象、操作内容、使う一時データを一致させて書く。
2. 対象を証明できていない状態で「開発版を操作する」と説明しない。
3. 申請が拒否された場合は停止し、表示名、Bundle ID、別アプリ、ビルド版へ切り替えない。
4. 申請後に対象不一致が判明した場合は操作前に停止し、申請を使い回さない。

## 確認して終了する

1. 証明済みの開発版だけで、列挙した受入条件と必要なLight / Dark、focus、hoverなどの状態を確認する。
2. 確認中に実ユーザーのワークスペース、既存ウインドウ、配布版が見えた場合は操作せず停止する。
3. 起動時に保持したterminal sessionへ割り込みを送り、自分が起動した開発版だけを終了する。
4. 完了報告では、自動テスト、開発版の起動、対象識別、実際に操作した状態を分ける。起動だけを実画面確認と表現しない。
5. 対象を証明できなかった場合は、その理由と未確認の受入条件を明示する。

# Relic Search and Link Change

## 依頼と処理位置を確定する

1. 調査、説明、レビューだけの依頼では編集しない。変更が明示されている場合だけ実装する。
2. `git status --short` を確認し、検索、置換、解析、リンク解決、派生一覧、索引更新のどれを変更するか限定する。
3. `docs/features/search.md`、`docs/features/links.md`、`docs/engineering/data-model.md` を読む。
4. プロセス境界では `docs/engineering/architecture.md`、ファイル書換えでは `docs/engineering/file-access-boundaries.md` を追加で読む。
5. ワークスペース内Markdownを正本とし、検索索引、解析キャッシュ、バックリンク、タグ、グラフを再生成可能な派生データとして扱う。

## 検索契約を守る

1. 検索対象を現在のワークスペース内Markdownに限定し、画像、PDF、別ワークスペースを通常検索へ混ぜない。
2. 全文検索にはフロントマターを含め、ファイル名検索にはファイル名と `aliases` を使う。
3. タグ検索にはフロントマター `tags` だけを使い、本文の `#タグ` は通常のタグ検索候補にしない。
4. フロントマター検索は認識済み固定プロパティだけをUI候補にし、配列、真偽値、nullの仕様を保つ。
5. 空の検索語句または未選択ワークスペースでは処理せず、古い結果、上限通知、エラーを空に戻す。
6. 結果上限、検索語句上限、2MiB境界、読めないファイルのスキップを現行定数と仕様に合わせる。
7. 旧検索入力はIPC互換アダプタだけで現行構造へ正規化し、正規化不能な値をメイン処理へ渡さない。

## リンク契約を守る

1. 内部リンクは同じフォルダを優先し、その後にワークスペース内で一意な同名Markdownだけを解決する。
2. 同じフォルダと一意な同名Markdownで解決できない場合は、ワークスペース内で一意な `aliases` を解決し、別名リンクもバックリンクへ集計する。
3. Markdown相対リンクはリンク元フォルダ基準で解決し、外部URL、プロトコル付きURL、ページ内アンカーをローカルファイルとして開かない。
4. 見出し、ブロックID、表示テキスト、埋め込みの構文情報を解析時と表示時の間で失わない。
5. 未作成リンクからの新規作成は安全なワークスペース相対パスだけに限定し、既存ファイルを上書きしない。
6. 埋め込みは現行仕様の深さと容量制限を守り、ワークスペース外参照を読み込まない。
7. アウトゴーイングリンクは開いている本文から、バックリンクなど全体派生データはmain側の共有索引から生成する。
8. 未リンク参照では既存リンク、埋め込み、Markdownリンク、コードブロックを候補外にし、外部変更後の古い位置へ書き込まない。
9. 同名ファイルが曖昧な場合は、表示文字列を保つパス付きリンクとしてリンク化する。

## 置換と索引を安全に更新する

1. 空の検索語句と無効または危険な正規表現を拒否し、過負荷を防ぐ現行制限を保つ。
2. 置換プレビューで内容snapshotを記録し、適用時に外部変更を検知したら古い結果を書き込まない。
3. 単一置換と一括置換を安全書き込みにし、適用中の読込・書込失敗は安全側で失敗させる。
4. ロールバック時も外部変更後のファイルを古い内容で上書きしない。
5. 保存成功時は変更パスだけを再読込し、外部リネームやパス不明時だけワークスペース単位で再生成する。
6. 索引やキャッシュをMarkdownより新しい正本として扱わず、破損時に実ファイルから再構築できる状態を保つ。
7. ファイルリネーム・移動の操作順序は `$relic-change-workspace-files` に委ね、解析・書換え規則を変える場合だけ本スキルを併用する。

## 検証する

1. 検索種別、大小文字、フロントマター、aliases、タグ正規化、上限、2MiBちょうど、読込失敗を対象テストで確認する。
2. 内部リンク、相対リンク、aliases解決とバックリンク、同名曖昧性、未作成、見出し、ブロック、埋め込み、コードブロック除外を確認する。
3. 置換では空文字、無効な正規表現、snapshot不一致、部分的読込失敗、書込失敗、副作用なしを確認する。
4. IPC互換入力、validator、preload・main契約、索引の増分更新と全体再生成を確認する。
5. 索引や解析性能を変えた場合は `pnpm performance:workspace` を同じ条件で使い、必要な場合だけlarge fixtureも測定する。
6. `app/` で対象のnode・rendererテスト、`pnpm typecheck`、必要なら `pnpm architecture:check` または `pnpm verify` を実行する。
7. UI状態も変えた場合は `$relic-change-ui` に従い、正本文書、差分、検証結果を照合する。
8. コミット時は `$relic-commit` に従う。

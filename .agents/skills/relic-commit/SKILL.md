# Relic Commit

## 手順

1. `AGENTS.md` のコミット規則、`docs/development.md` のバージョン規則、`app/package.json` の現在値を確認する。
2. ユーザーがコミットしないよう指定している場合は中止する。pushだけの依頼をコミット許可に読み替えない。
3. `git status --short`、作業ツリー差分、ステージ済み差分を確認し、依頼に必要な差分と無関係な既存差分を分ける。無関係な差分を変更、取消、ステージしない。
4. 変更目的とConventional Commitsのtypeでコミット単位を決める。複数のtypeや独立した目的がある場合は分け、各目的に必要なコード、テスト、文書を同じコミットへ含める。
5. 各コミットで `app/package.json` のversionを更新する。ユーザーがMAJORまたはMINORを明示した場合だけその番号を上げ、それ以外は変更規模やtypeにかかわらずPATCHを上げる。バージョン更新を単独コミットにしない。複数コミットではコミットごとに順次更新する。
6. 変更リスクに対応する検証を完了する。失敗を解消できない場合はコミットせず、失敗内容を報告する。
7. 対象パスまたはhunkを明示してステージする。`git add .` と `git add -A` を使わない。`git diff --cached --check` と `git diff --cached` で、対象、機密情報、意図しない差分を再確認する。
8. 件名を `<type>[!]: <version> <日本語の説明>` とする。typeは `feat`、`fix`、`docs`、`style`、`refactor`、`perf`、`test`、`build`、`ci`、`chore`、`revert` から1つだけ選ぶ。件名にscopeを書かない。破壊的変更では `!` または `BREAKING CHANGE:` を使う。
9. 本文を日本語で書き、原則として `scope:`、`目的:`、`内容:`、`確認:`、`影響:` を含める。複数ファイル、削除、仕様・運用変更、バージョン更新を含む場合は省略しない。秘密情報、個人情報、認証情報、内部URL、ローカル絶対パス、環境変数値、機微なエラー値は伏せるか一般化する。
10. コミット後にコミットIDと残存差分を確認する。完了報告で検証、コミットID、未コミットの無関係差分、未pushであることを示す。
11. pushはユーザーが明示した場合だけ実行する。
